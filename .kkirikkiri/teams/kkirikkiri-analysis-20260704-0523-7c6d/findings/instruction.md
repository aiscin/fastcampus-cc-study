## 갭 분석

### P0 — 이걸 안 고치면 빌더가 라우팅 실패 (표준 03)

**닥터 4종 + /new-workspace + /audit의 description/라우팅 계약이 0.**
- line 31·104-109는 닥터 이름과 "담당 결정"만 있고, **트리거 발화·후속 키워드·NOT-trigger 경계가 전무.** line 209 "그 자리에서 만든다"로 전량 Clip 2에 위임.
- 표준 03 §1 실측: generic description 서브에이전트 카드는 수십 회 위임 기회에서 **0회 선택**. 닥터는 스킬/서브에이전트 성격이므로 정확히 이 함정.
- 특히 `instruction-tuner`(지시)·`harness-audit`(검증)·`guard-installer`(강제)는 표면이 겹친다. "이 워크스페이스 점검해줘" 한마디에 셋 중 누가 뜨는지 설계도가 답하지 못함. 표준 03 §2.3 NOT-trigger 경계가 없으면 오발동/미발동 불가피.
- **개선 방향**: 설계도에 닥터별 `트리거 발화 2-3개 + 산출물 계약 1구절 + "단, X는 /other" 경계`를 표로 박아 넣는다. 이건 Clip 2 구현이 아니라 설계 결정 그 자체다.

### P1 — 품질 심각

1. **"자연어 트리거 자동 생성"이 마법(line 159).** 입력(요청서 어느 칸에서?)·판정 기준·출력(발화 템플릿) 미정의. 표준 03 §2는 "사용자가 실제로 말할 자연어"를 요구하는데, 무엇을 근거로 합성하는지 없음.
   → 개선: `요청서 "여기서 할 동작" 칸 → 발화 후보 규칙`을 명시하거나, "빌더가 초안 생성 후 사용자 확인"으로 강도 하향.
2. **빌더가 자기 규칙을 도그푸딩 안 함.** 2단 규칙 2순위(커맨드는 CLAUDE.md 트리거 표로 감쌈, line 156)를 output에는 강제하면서, 정작 빌더 자신의 `/new-workspace`·`/audit`용 트리거 표는 설계도에 없음.
   → 개선: 빌더 CLAUDE.md에 자기 커맨드용 트리거 표 1개 추가.
3. **매칭 임계값이 비결정적.** line 85 "자주·길게 반복", line 87 "증거 있을 때만", line 60 "아키타입 판정"(10종 중 헷갈리는 쌍은 line 198-202에 3쌍뿐). 표준 02 §3.2 "적절히/충분히 금지, 수치로"의 정면 위반. 같은 요청서로도 매 실행마다 다른 레이어/아키타입이 나올 여지.
   → 개선: "자주"에 횟수/세션빈도 예시, "증거"의 정의(예: 탐색 대상 N개↑), 10종 disambiguation 쌍 확충.

### P2 — 개선

- **빌더 CLAUDE.md가 표준 02 §2 구조 대비 과소.** line 30은 "두 층 순서" 한 문장뿐 — 표준 02 §6 "너무 짧음→컨텍스트 부족" 쪽. line 13의 정체성 허용/금지가 CLAUDE.md L0로 슬롯되지 않았고, 얇은 CLAUDE.md→`20-blueprints` 카탈로그/닥터 스킬로 가는 **포인터 배선(표준 07 §4 정본+포인터)이 문장으로 없음.**
  → 개선: 빌더 CLAUDE.md 골격을 표준 02 §2 순서(정체성 허용/금지 + 핵심원칙 + 폴더 + 포인터)로 실제로 한 장 써 둔다. Q1이 우려한 "예산 폭발"은 오히려 아님 — 이 설계는 반대로 **과소 provisioning** 쪽이다.
