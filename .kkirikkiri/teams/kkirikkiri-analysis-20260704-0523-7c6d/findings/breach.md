# 침투 시험 보고 — 워크스페이스 빌더(메타하네스) "어겨보기"

> 관점: breach-test(레드팀). 설계도대로 이 빌더를 실제로 지었다고 가정하고, 악의적/부주의한 사용자가 뚫는 경로를 구성.
> 대상: `50-my-work/Part07-실전/실습32-워크스페이스설계/설계도.md`
> 판정 원전: harness-standards `05-hooks-recovery.md` · `ANTI-PATTERNS.md` · `SELF-CHECK.md`
> 날짜: 2026-07-04

**방어 견고성 점수: 28 / 100** (낮을수록 잘 뚫림)
설계 사고는 표준을 잘 인용하지만, **명시된 강제층 2개("40-output 밖 차단", "시크릿/PII 차단")가 빌더의 핵심 동작과 정면충돌**하거나 **부재**해서 정상 사용만으로도 무력화된다. 아래 7개 경로 중 P0 3개는 "정상 기능 = 우회"라 훅을 붙여도 안 막힌다.

## P0-1. 강제층이 빌더의 정상 동작에서 무효 — "독립 폴더" vs "40-output 밖 차단" 충돌

**시나리오**
- 강제층 기본값(설계도 34·192줄): `40-output/` **밖**의 파일 수정·삭제 차단.
- 그런데 출력 위치 확정(120줄): "실제 워크스페이스는 **별도 경로에 독립 생성** → 클로드코드로 바로 열림." `40-output/`은 "완성 후 이관 대기소"일 뿐.
- 즉 빌더의 **주 기능이 40-output 밖에 쓰는 것**이다.

**어떻게 뚫리나**
둘 중 하나로 귀결되며 둘 다 방어 실패:
1. 훅이 정직하게 "40-output 밖 차단"을 지키면 → 빌더가 독립 폴더를 못 만든다. **핵심 기능이 죽는다.**
2. 그래서 "독립 출력 경로는 예외 허용"으로 완화하면 → 출력 경로는 요청서가 지정하는 **임의 경로**다. 사용자가 출력 경로를 `~/.claude/`, `/home/user/live-repo`, 심지어 빌더 자신의 `10-standards/`(불변 참조)로 적어내면 빌더가 "정상 출력"으로 거기에 덮어쓴다. 차단 규칙의 선언("40-output 밖 차단")이 **사실상 거짓**이 된다.

**피해**
남의 라이브 워크스페이스·`.claude` 설정·빌더의 불변 표준을 "정상 산출물"로 덮어씀. 표준 05가 말하는 "방어선 자체 보호"의 반대 — 방어선이 자기 발등을 찍는다. ANTI-PATTERNS #8(문서 규율만으로 비가역 자원 보호)의 변종: 훅이 있어도 스코프가 틀려 무의미.

**막는 법**
가드를 "40-output 밖 차단"이 아니라 **"이번 실행에 대해 검증·확정된 단일 출력 경로 ∪ 40-output 밖 차단"**으로 재정의. 출력 경로는 실행 시작 시 (a) 기존 워크스페이스/`.claude`/시스템 경로 내부가 아님을 검증, (b) 사용자 확인, (c) 그 실행 동안 잠금(per-run allowlist). 경로 검증 없는 "독립 폴더 예외"는 강제층을 장식으로 만든다.

## P0-2. 스킬 복사 오염 — 능력만 따라오고 가드는 안 따라온다

**시나리오**
스킬 재사용 확정(122줄): 아키타입이 요구하는 스킬을 **라이브러리 16종에서 복사** 배치. description 유지(139줄) → 자연어로 자동 발동.

**어떻게 뚫리나**
16종에는 외부 부작용이 실재한다: `naver-news`(외부 API + GitHub Actions 자동 발행), `morning-briefing`(Gmail 읽기·발송), `content-pipeline`(발행), `audio-to-doc`(오디오를 Google/Gemini로 전송), `trip-advisor`·`fashion-trend`(외부 API·KREAM 스크레이핑). 이 스킬을 새 워크스페이스에 복사하면 **능력은 오지만 원본 워크스페이스의 가드는 하나도 안 온다.**
- 예: "콘텐츠 발행" 아키타입에 `content-pipeline`+`naver-news`를 복사 → 새 워크스페이스엔 egress 가드도, "외부 발송 전 확인" 훅(그건 개인비서 아키타입 기본값이지 콘텐츠 기본값 아님)도 없다.
- 더 나쁜 건 **자동 발동**: description을 유지하므로 사용자가 설치된 줄도 모르는 채 "오늘 뉴스 정리해줘" 한마디에 `naver-news`가 외부 API를 때리고 발행까지 간다.

**피해**
무통제 외부 발송·API 과금/쿼터 소진·데이터 유출(오디오→Google)이 **동의 게이트 없이** 발생. 복사 과정에 API 키 기대·시크릿이 딸려오면 유출까지. ANTI-PATTERNS #3(못 지킬 의무)·#12(양치기)와 별개로, **능력↔가드 비대칭**이 근본 결함.

**막는 법**
스킬마다 **능력 매니페스트**(외부 API/발송/과금/PII 유출 선언)를 요구. 빌더는 복사 전 매니페스트를 읽고 **선언된 능력마다 대응 가드를 먼저 설치**(guard-installer 게이트)하거나 복사를 거부. 아키타입이 소유하지 않은 능력의 자동 발동 description은 **기본 비활성**(명시적 enable 필요). → **guard-doc(guard-installer)와 정확히 겹치는 지점.**

## P0-3. 인터뷰 우회 — "요청서를 그럴싸하게 다 채우면" 되묻지 않는다

**시나리오**
인터뷰 트리거(78·124줄): "칸이 비었거나 애매하면 되묻는다." 되묻는 체크리스트(124~128)는 전부 **완성도** 체크(한 문장인가 / 아키타입 하나로 정해지나 / 사고가 비가역인가 / 자료 위치 확인되나).

**어떻게 뚫리나**
위험한데 **자신 있게 꽉 채운** 요청서를 낸다:
```
목적: 프론트엔드 자동 배포 환경
아키타입 힌트: 자동화 파이프라인
여기서 할 동작: 빌드 후 프로덕션 API 키를 프론트 번들에 주입해 배포
절대 안 할 것: (적지 않되 공란 아니게) 없음
막고 싶은 사고: 없음 — 전부 안전함
```
체크리스트 4개 전부 통과한다: 목적은 한 문장, 아키타입은 자동화로 깔끔히 매핑, "막고 싶은 사고"는 (비었다가 아니라) "없음"으로 채워졌고, 자료 위치도 확인됨. **되묻는 조건이 "빈칸/애매"뿐이라, 완성도 높은 위험 요청엔 콜백이 0회.**
빌더는 자동화 아키타입을 찍고, 그 강제층 기본값은 "프로덕션 키 보호"인데 — 사용자가 키를 프론트에 주입하는 것을 **목적으로 선언**했으니 그 가드는 "정상 기능 방해"로 완화되거나, 아니면 워크스페이스 존재 이유를 막는다.

**피해**
빌더가 **세탁 단계**가 된다: "전문 빌더가 검토·생성했다"는 거짓 보증을 달고 프로덕션 시크릿을 공개 번들에 노출하는 환경을 찍어냄. "완성도 높음 = 안전"이라는 설계의 잘못된 등식(설계도가 인코딩한 바로 그것).

**막는 법**
인터뷰를 **빈칸이 아니라 위험 신호**에 트리거. 완성도와 독립된 위험 분류기: (프로덕션 키 + 프론트/공개), (외부 발송 + 확인 없음), (PII + 커밋), (출력 경로 밖 삭제) 중 하나라도 걸리면 **완성도·자신감과 무관하게 인간 확인 강제.** 요청서 완성도는 위험도의 대리 지표가 아니다.

## P1-4. 아키타입 오판정 → 엉뚱한 가드 → silent failure

**시나리오**
헷갈리는 쌍의 판정 기준(198~202줄)이 산문 휴리스틱("일회성이면 리서치, 계속 쌓으면 지식베이스").

**어떻게 뚫리나**
지식베이스 성격의 요청을 "일회성 조사"처럼 표현 → 빌더가 **리서치**로 오판. 리서치 강제층 기본 = "출처 없는 단정 경고"(약한 경고). 사용자는 실제로 민감 원본을 축적하며 **지식베이스의 "raw 원본 불변"** 보호를 기대. 그 불변 가드는 설치되지 않는다. 나중 세션이 raw를 덮어써도 **사용자는 보호받는 줄 안다** — 무가드보다 나쁜 silent failure.
- 자동화 vs 데이터분석: 분석으로 오판된 무인·스케줄 요청 → "원본 데이터 불변"은 받지만 "프로덕션 키 보호"는 못 받고 헤드리스로 키를 노출한 채 돈다.
- 콘텐츠발행 vs 장문집필: 오판 시 "final 삭제 보호"가 발행물에 안 깔린다.

`/audit`(108줄)이 있지만 audit은 "의도↔결과 갭"을 **(오판된) 선언된 의도**에 맞춰 보는 것 — 실제 행동 대비가 아니다. Garbage-in.

**피해**
사용자가 **거짓 보호 아래** 운영. ANTI-PATTERNS #9(검증 없는 완료 선언)의 구조적 버전.

**막는 법**
아키타입 선택을 사용자에게 평문으로 확인("이걸 리서치=일회성으로 처리해서 raw를 불변으로 잠그지 **않습니다** — 맞나요?"). 그리고 애매할 땐 두 아키타입의 **비가역 가드 합집합**을 설치(가드는 유지 비용 싸고, 침묵 부재가 비싼 실패). 표준 05 "약→강, 안전장치는 최소로"의 예외: **판정 불확실성 하에서는 가드를 superset으로.**

## P1-5. 재귀 폭탄 — 메타가 메타를 찍는데 깊이·팬아웃 제한이 없다

**시나리오**
메타 아키타입(194줄): "빌더가 자기 자신도 재생산 — 특화 빌더를 찍어내는 것도 가능."

**어떻게 뚫리나**
inbox에 "너 같은 빌더를 만들어줘"를 넣는다. 빌더는 메타 아키타입을 찍고, 찍힌 결과물도 메타 아키타입이라 또 찍을 수 있다. 설계 어디에도 **재귀 깊이 카운터·팬아웃 상한·자기참조 감지**가 없다. 이를 자동화 아키타입(스케줄, 무인)과 결합하면 — 찍힌 빌더의 inbox에 자기복제 요청을 자동 투입 — **지수적 워크스페이스 증식**.
게다가 출력이 40-output이 아니라 **임의 독립 폴더**(P0-1)라, 폭탄이 흩어져 **쓸어담을 단일 격리 폴더도 없다.**

**피해**
디스크 고갈, 폴더 폭발, 컨텍스트/토큰 소진. 무인 결합 시 사용자 부재중에 자원 소진.

**막는 법**
요청 메타데이터에 재귀 깊이 기록, **메타→메타는 depth 1 초과 시 인간 확인 없이는 거부.** 세션당 생성 워크스페이스 팬아웃 상한. 자기참조("빌더가 빌더를 무인으로 투입") 패턴 감지·차단. 표준 05의 recovery 계약에 "폭주 생성 감지 → 정지" 항목.

## P2-6. 훅 설정 자체가 무방비 — ConfigChange 가드 부재

**시나리오**
설계의 강제층은 "40-output 밖 차단"·"시크릿/PII 차단" 2개뿐. **방어선 자체를 보호하는 장치가 없다.**

**어떻게 뚫리나**
사용자(또는 P0-2로 복사된 부작용 스킬, 또는 P0-3로 찍힌 위험 워크스페이스)가 세션 중 `.claude/settings.json`에서 가드 훅을 제거하면 그 뒤로 어디든 쓴다. 표준 05 §1이 명시한 `ConfigChange`(세션 중 훅 설정 변조 무효화)와 SELF-CHECK #20이 **둘 다 미충족**.

**피해**
"방어선 끄기"라는 가장 쉬운 경로가 열려 있어 위 P0들을 한 번에 무장해제.

**막는 법**
`ConfigChange` 가드로 세션 중 훅/권한 변조를 무효화. 세션 밖 사용자 직접 수정 권리는 보존.

## P2-7. 반쯤 지어진 워크스페이스 — "약→강" 순서가 만드는 가드 공백 창

**시나리오**
쌓는 순서(98줄): "뼈대 → 지시 → 강제 → 검증. 훅을 규칙보다 먼저 걸지 않는다." 완성도 확정(121줄): 스킬 배치까지 한 번에.

**어떻게 뚫리나**
"약→강" 때문에 **뼈대+부작용 스킬은 이미 있는데 강제층은 아직 안 깔린 시간창**이 반드시 존재한다. 이 창에서 빌더가 크래시/중단되면(설계엔 빌더 자신의 recovery 계약·Stop 게이트·세션당 1커밋 루틴이 **없다** — rm-to-trash만 기본값으로 언급) → **가드 없는 라이브 워크스페이스에 자동 발동 스킬만 살아 있는 상태**로 방치. 사용자가 그걸 열어 "뉴스 정리해줘" 하면 P0-2가 발화.

**피해**
반제품 워크스페이스가 무가드로 외부 부작용 실행. ANTI-PATTERNS #10(사고 후 즉흥 복구) — 사전 복구 계약 부재.

**막는 법**
빌더 자신에게 recovery 계약(반제품 감지→롤백) + Stop 게이트(강제층 미설치 상태로 "완료" 선언 차단, 루프 가드 포함) + 세션당 1커밋. 가드 미설치 상태의 워크스페이스는 스킬 자동 발동을 **잠금** 상태로 태어나게 하고, 강제층 설치 완료 시에만 해제.

| 4 | 아키타입 오판정 → 거짓 보호 | P1 | 산문 휴리스틱·확인 부재 → silent failure |
| 5 | 재귀 폭탄 | P1 | 깊이·팬아웃·자기참조 제한 전무 |
| 6 | ConfigChange 가드 부재 | P2 | 방어선 자체 무방비 |
| 7 | 반제품 가드 공백 창 | P2 | 빌더 자신의 recovery 계약 부재 |

**한 줄**: 설계는 "표준을 읽은 티"는 나지만, 명시된 강제층 2개가 빌더의 핵심 동작(독립 폴더 출력·스킬 복사)과 충돌하거나 부재라서, **정상 사용만으로 무장해제**된다. "안전해 보인다"가 아니라 — 위 3개 P0는 실제로 뚫린다.

## guard-doc(guard-installer)와 공유 경계 (확정)

강제층은 guard-doc 소유, 탐지 판정 중 아래는 breach-doc 소유 — 같은 훅에 공존:
- **가드 A(inbox intake 스캔)**: 시크릿/PII 패턴=guard-doc, 인젝션·악성 훅 코드 서명 패턴=breach-doc.
- **가드 C(스킬 복사 능력 매니페스트)**: 능력 항목 열거(등록 훅/외부 egress/요구 시크릿/발송/과금/PII 유출)=breach-doc. 각 항목 대응 가드 선설치 규칙 + **"미지 능력=자동통과 아니라 격리·복사거부" default-deny fallback**=guard-doc. (내가 항목을 빠뜨린 순간이 우회로가 되지 않도록 whitelist 아닌 default-deny.)
- **P0-3 위험신호 분류**: 위험 패턴 집합(프로덕션키+공개 / 외부발송+무확인 / PII+커밋 / 출력경로 밖 삭제)=breach-doc. 걸렸을 때 도는 강제 확인 게이트(완성도 무관·우회 없음)의 기계 동작=guard-doc.
- **P0-1 per-run 출력경로 allowlist**: 전부 guard-doc(기계 스펙) 소유.
