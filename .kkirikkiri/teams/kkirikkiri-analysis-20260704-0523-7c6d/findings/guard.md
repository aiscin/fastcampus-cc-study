## 핵심 질문별 적대적 판정

### Q1. 강제 층 2개가 빌더의 비가역 사고 지점을 실제로 커버하나 → 아니오

빌더의 실제 동작 = 폴더 생성 · 파일 쓰기 · 스킬 복사 · git init. 비가역 사고 지점을 훑으면:

| 사고 지점 | 2개 강제 층이 막나 | 근거 |
|---|---|---|
| 완성본을 **기존 워크스페이스에 덮어쓰기** | ✗ | rm-to-trash는 `rm`만 가로챈다(Bash 첫 토큰). Write/Edit/`cp -f`/`mv`로 덮으면 무방비. 휴지통 사본도 안 남음 |
| **독립 폴더**(line 120)에 실제 워크스페이스 생성 | ✗ | 강제 #1은 "40-output 밖 차단"인데 실제 출력은 40-output **밖** 독립 경로 → 방어선 밖 (아래 모순 참조) |
| **자기 복제 무한 재귀** | ✗ | line 194 "빌더가 자기 자신도 재생산" 명시. 깊이 제한·자기 디렉토리 쓰기 금지 없음 |
| **대량 파일 생성** (10 아키타입 × 폴더 골격) | ✗ | 생성량 캡 없음 |
| 스킬 복사 시 훅 코드 동반 | ✗ | 라이브러리→새 WS로 코드 실행 훅 복사, 신뢰 검사 없음 (침투 닥터 겹침) |

**치명적 모순 (P0):** 강제 #1 = "40-output 밖 수정 차단" ↔ 빌드 정의 line 120 = "실제 워크스페이스는 40-output **밖** 별도 독립 경로에 생성(클로드코드로 바로 열림)". 둘 중 하나만 참이다.
강제 #1을 문자 그대로 걸면 빌더의 핵심 기능(독립 워크스페이스 찍기)이 차단되고, 독립 폴더를 허용하면 강제 #1은 "40-output 밖 아무 데나 써도 됨"이 되어 사실상 무력화.

### Q2. "40-output 밖 차단"이 훅으로 어떻게 구현되나가 설계도에 있나 → 없다 (= 지시, 강제 아님)

- 판정 로직(경로 화이트리스트)·이벤트·exit 코드가 **설계도 어디에도 없다.** line 108이 `guard-installer`가 "되돌림 X 판정"을 맡는다고 미룰 뿐 — Clip 2 약속이지 스펙이 아니다.
- 표준 05: 판정 로직 없는 규칙은 **지시(부탁)이지 강제(장치)가 아니다.** 설계도가 스스로 line 90~94에서 표준 05 되돌림 테스트를 정확히 인용해놓고, 정작 그 규칙 #1엔 적용을 안 했다.
- **도구 커버리지 공백:** rm-to-trash는 Bash 전용. 40-output 밖 파일을 **Edit/Write 도구**로 고치면 Bash 훅은 아예 발화 안 함. "밖 차단"이 어느 이벤트(PreToolUse on Write/Edit/Bash 전부?)를 거는지 미정.
- 표준 05 되돌림 테스트를 규칙 #1 자체에 적용하면: "밖 수정"은 git 있으면 되돌림 O(=지시), 근데 빌더는 git 미추적 **새 경로**에 쓰므로 복구 경로 부재 → 판정조차 안 풀렸다.

### Q3. Recovery-First — 차단만 있고 되돌림은? → rm-to-trash 하나뿐, 그것도 위험을 빗나감

- ✅ rm-to-trash.py 이식은 진짜 복구 장치다 (line 141). `rm` → 휴지통, 위험경로 거부, 복합명령 차단까지 견고.
- ✗ **그러나 빌더의 지배적 사고는 `rm`이 아니라 덮어쓰기다.** "output을 잘못 찍었을 때" = 좋은 워크스페이스 위에 손상본 Write/cp. rm-to-trash는 rm이 아니라 무력. 되돌림 경로 **없음**.
- ✗ recovery.md **복구 계약 부재** (표준 05 §3): 실패 루프·재시도 한계·scope 잠금·되돌리기 규칙 표 — 표준 05 자가점검 "사고 **전에** 작성됐나"에 전부 미달.
- ✗ **Stop 훅 게이트 없음:** /audit(검출)은 있으나 "audit 통과 전 '끝났습니다' 차단"하는 Stop 훅이 없다 → audit은 모델이 협조해야만 도는 지시. 표준 05가 명시적으로 요구하는 게이트.
- ✗ ConfigChange 훅 없음 (표준 05 §1 메타 방어) — 빌더가 자기/자식 훅 설정을 고치는 존재인데 방어선 자체 보호 없음.
- ✗ 세션 커밋=복구지점 언급 없음.

### Q4. PII/시크릿 3중 방어 → 1겹, 그것도 엉뚱한 repo에

표준 06은 3중(gitignore 3단 + git init 안전순서 + 커밋 게이트 훅·루틴 이중화)을 Mandatory로 요구. 설계도는 line 35 "시크릿·PII 커밋 차단" **한 줄**뿐.

- ✗ **자식 워크스페이스에 gitignore 안 심음:** 빌더는 워크스페이스를 **찍어내는** 존재다. 표준 06 §2 순서(gitignore→init→check-ignore 실측→add)를 자식마다 뼈대 단계에서 박아야 하는데 규칙 없음.
- ✗ **틀린 altitude:** 강제 #2는 빌더 **자기** repo 커밋만 막는다. 그런데 line 46 재사용 자료(CLAUDE.md·샘플데이터)에 키가 섞이면 → 빌더가 그걸 **자식 워크스페이스로 복사**(line 122) → 자식의 **새 repo** 첫 커밋에 박제(표준 06 §1 비가역). 빌더 커밋 게이트는 이 전파를 못 본다.
- ✗ **inbox intake 무스캔:** 00-inbox 재사용 자료가 반입 시점에 스캔되지 않음. 3중(작성·저장·발송) 중 발송(커밋) 1겹만, 그것도 자기 repo 한정.
- ✗ 게이트 로직·정밀 grep 패턴(`sk-`, `api_key=`)·"우회 없음" 명시 전부 부재.

- G3. "밖 차단" 판정 로직 공백 = 강제 아닌 지시. → 개선: PreToolUse(Write/Edit/Bash 전부) 경로 화이트리스트 + exit 2 스펙을 설계도에 명시.

**P2**

# 강제층 개선 스펙 — 통합 가드 세트 (최종 종합용)

> 2라운드 조율 결과. breach-doc과 수렴한 3방어를 **중복 없이 하나의 세트**로 확정.
> 소유 경계 (breach-doc과 확정): 트리거·차단·복구·**"미지=거부" fallback**의 **기계 스펙 = 강제층(guard) 소유**. **"무엇이 위험한가"(인젝션·악성훅 서명, 능력 항목 열거, 위험패턴 집합 분류) = 침투(breach) 소유**. 시크릿/PII 탐지 판정 = 강제층 소유. "무엇이 위험한가"=breach / "걸리면 어떻게 멈추나"=guard. 한 훅에 두 판정이 얹힌다.

## 강제 #1 재정의 — 확정 문구 (line120 모순 해소)

> **"40-output 밖 절대 차단"을 폐기하고 → `per-run 쓰기 잠금(write-scope lock)`으로 대체한다.**
> 빌드 착수 시 그 run의 **쓰기 허용 경로**를 잠근다:
> (a) 빌더 자신의 `30-drafts/` · `40-output/`,
> (b) **이번 요청이 선언한 독립 출력 경로 1개**(line120의 "별도 독립 경로"),
> (c) 그 외 전부 차단.
> 단 (b)의 선언 경로가 **이미 존재하면** 쓰기 차단 → 타임스탬프 백업 후 `GUARD_OK=1` 의식적 우회.
> 빌더의 `10-standards/` · `20-blueprints/` · `00-inbox/` 원본은 **항상 읽기전용(불변)**.
>
> → 이러면 line120 독립폴더 출력과 **모순 없이**, 정확히 "남의 작업물 덮어쓰기"만 막는다. 40-output은 "이관 대기소"로만 남고, 강제 경계는 "이번 run이 선언하지 않은 경로에 쓰기"로 이동한다.

## 이 스펙이 없앤 중복 3건
- inbox 스캔이 **G-A 하나**로 통합(시크릿·PII·인젝션·악성훅 4탐지가 한 훅). 강제/침투가 각자 훅 만들면 중복될 뻔.
- "경계"가 **G-B 하나**로 정리 — 기존 강제 #1(모순)을 폐기하고 per-run 잠금으로 대체.
- PII가 **G-D(생성 시)+G-E(커밋 시)** 2겹으로 altitude 교정 — 빌더 자기 repo가 아니라 자식 워크스페이스에서 막는다.
