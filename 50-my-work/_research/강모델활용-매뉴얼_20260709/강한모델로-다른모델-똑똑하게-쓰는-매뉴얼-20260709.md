> ★ **완성본(타사 혼합 라우팅 심화 + "모델 자동지정 훅" 실전 지침 포함)은 같은 폴더의 `강모델-활용-핸드북.html` 참조.** 이 문서는 초기 리서치 보고서 판.

## ① 핵심 한 줄 요약

**"강한 모델을 상시로 돌리지 말고, 판단·설계·검증 같은 '비싼 순간'에만 쓰고 나머지 실행은 싼 모델에 맡겨라"** — 이게 X에서 도는 노하우들의 공통 뼈대다. 검증된 방법은 크게 4가지: **① 어드바이저 위임, ② 오케스트레이터-워커, ③ 라우팅·캐스케이드, ④ 자산 생성**. 이 중 개인이 지금 당장 가장 효과 보는 건 **①(어드바이저)** 과 **④(자산 생성)** 다.

| 패턴 | 한 줄 정의 | 언제 | 강 모델의 역할 | 절감·효과(근거 수치) | 난이도 | 신뢰도 |
|---|---|---|---|---|---|---|
| ① 어드바이저 위임 | 싼 모델이 일하다 막히면 강 모델에게 "훈수"만 받음 | 대부분의 에이전트·긴 작업 | 결정적 순간에 계획·교정 | Haiku+Opus조언 점수 19.7→41.2%, Sonnet 단독 대비 85%↓ | 낮음 | A(패턴)/C(수치) |
| ② 오케스트레이터-워커 | 강 모델이 계획 세워 여러 서브에이전트에 병렬 분배 | 넓게 훑는 리서치·병렬 작업 | 총괄 지휘관 | 단일 에이전트 대비 +90.2%, 단 토큰 ~15배 | 중간 | A/B |
| ③ 라우팅·캐스케이드 | 쉬운 건 싼 모델, 어려운 것만 강 모델로 올림 | 요청량 많은 서비스·API | 어려운 케이스 전담/에스컬레이션 | 품질 95% 유지하며 비용 45~85%↓ | 중간 | C |
| ④ 자산 생성 | 강 모델이 프롬프트·규칙·예시·검증셋을 "제작"해 둠 | 반복 작업 세팅 시 | 스캐폴딩 제작자 | 합성 예시로 소형 모델 +4~6점, 때론 교사 모델 추월 | 낮음 | B/C |

> ★ 개인 활용 우선순위: **④(한 번 만들어두면 계속 이득) → ①(즉시 체감) → ②(무거운 리서치일 때) → ③(서비스 만들 때)**

## ③ 패턴별 실전 매뉴얼

### 패턴 ① — 어드바이저 위임 (작은 실행자 + 큰 조언자) ★가장 실용적

**뭐냐**: 싼·빠른 **실행자 모델**(예: Haiku/Sonnet)이 작업을 진행하다가, 방향을 정하기 전·같은 에러에 계속 막힐 때·"다 됐다" 선언 직전 같은 **결정적 순간에만** 강한 **조언자 모델**(Fable 5/Opus)에게 전체 맥락을 넘겨 계획·교정을 받는다. 대부분의 토큰 생성은 싼 모델 가격으로 돌면서 품질은 강 모델에 근접한다. (신뢰도 A)

**핵심 규칙** (공식 문서 확인, 신뢰도 A):
- 조언자 모델은 **실행자보다 같거나 더 똑똑해야** 한다. 조언자는 최소 Sonnet 4.6 이상, 최상은 Fable 5.
- 예: 실행자 Haiku 4.5 → 조언자로 **Fable 5 / Opus 4.8** 지정 가능.
- 조언자는 **전체 대화·툴 결과를 다 읽고** 훈수만 주고, 실행은 싼 모델이 계속한다.
- 언제 부를지는 모델이 알아서 결정 — 매 턴이 아니라 결정점에서만.

**언제**: 긴 호흡의 에이전트(코딩 에이전트·컴퓨터 사용·다단계 리서치)처럼 **대부분 턴은 기계적이지만 계획이 중요한** 작업. 단발 Q&A엔 부적합(계획할 게 없음).

**어떻게 (API)**: 요청의 top-level `model`에 싼 실행자, 툴 정의 안 `model`에 강 조언자. 베타 헤더 `advisor-tool-2026-03-01` 필요.
```json
{
  "type": "advisor_20260301",
  "name": "advisor",
  "model": "claude-fable-5"
}
```
- 잘못된 짝(조언자가 실행자보다 약함)은 400 에러.
- Haiku 실행자가 첫 턴에 조언자를 안 부르면, 다음 턴 전 짧은 리마인더 유저 메시지를 붙이면 통과율 ~7%p 상승(단, Opus 실행자엔 오히려 역효과라 붙이지 말 것).

**어떻게 (Claude Code / 개인 활용 응용)**: 어드바이저 툴은 API 기능이지만, **개념은 Claude Code에서도 그대로 흉내** 낼 수 있다 → 평소엔 싼 모델로 작업하다가 "막혔어요"나 설계 결정 순간에만 **강 모델 세션/서브에이전트에 계획을 물어보고 그 답을 실행 모델에 전달**하는 습관. (내 워크스페이스의 `/stuck` 흐름과 궁합)

**근거 수치**: Haiku + Opus 조언자가 단독 대비 점수 2배 이상(19.7%→41.2%), Sonnet 단독보다 태스크당 85% 저렴 — *단, 이 구체 수치는 3rd-party 요약글 기준(C), 공식 문서는 "튜닝 수치"만 확인.* 공식은 "실행자 결과가 조언자 단독에 근접하면서 토큰 대부분은 실행자 가격"이라고만 명시(A).

### 패턴 ② — 오케스트레이터-워커 (멀티에이전트)

**뭐냐**: 강한 **리드 에이전트**가 질문을 잘게 쪼개 계획을 세우고, **각자 독립된 컨텍스트 창을 가진 서브에이전트**들을 병렬로 띄워 분담시킨 뒤, 결과를 모아 최종 답으로 합친다. 리드는 직접 일하지 않고 **계획·분배·통합**만 한다. (신뢰도 A/B — Anthropic 자사 사례)

**성공의 핵심 = 위임 지시를 구체적으로**:
- 각 서브에이전트에게 **① 구체적 목표 ② 출력 형식 ③ 사용할 도구·출처 ④ 작업 경계**를 명시. "반도체 부족 조사해"처럼 막연하면 중복·누락 발생.
- **복잡도에 맞춰 규모 조절**: 단순 사실확인 1명(툴 3~10회) / 비교 2~4명 / 복잡 리서치 10명+.
- 서브에이전트는 3~5개를 **동시에** 띄우고, 각자 도구도 3개+ 병렬 호출 → 리서치 시간 최대 90%↓.

**언제 쓰나 / 피하나**:
- 쓸 때: 여러 방향을 **동시에 넓게 훑는** 작업, 정보가 컨텍스트 창을 넘칠 때, 가치가 높아 토큰 비용을 정당화할 때.
- 피할 때: 에이전트 간 **강한 의존/공유 맥락**이 필요할 때, **대부분의 코딩 작업**(병렬화 가능한 부분이 적음), 실시간 조율.

**어떻게 (Claude Code)**: 내 워크스페이스에 이미 있는 **서브에이전트**(Explore, general-purpose 등)가 바로 이 패턴. "여러 각도로 조사"류 작업에서 리드(=Fable 5 메인)가 분배하도록 쓰되 — **CLAUDE.md 규칙상 사용자가 명시할 때만** 띄우는 게 원칙이니 남발 금지.

**근거 수치**: 리드 Opus 4 + 서브 Sonnet 4 조합이 단일 Opus 4 대비 **+90.2%**, 단 채팅 대비 **~15배 토큰**. (Anthropic engineering blog, A/B)

### 패턴 ③ — 라우팅 & 캐스케이드 (비용 최적화, 주로 서비스·API)

**뭐냐**:
- **라우팅**: 요청이 들어오면 **선분류**해서 쉬운 건 싼 모델, 어려운 건 강 모델로 한 번에 보냄.
- **캐스케이드**: 일단 **제일 싼 모델부터** 시도 → 확신도가 낮으면 강 모델로 **에스컬레이션**.

**핵심 규칙 & 함정** (신뢰도 C — 실무 블로그/논문):
- 효과: 품질 95% 유지하며 비용 **45~85%↓** (예: GSM8K에서 GPT-4를 14%만 써도 성능 95% 달성).
- **함정 1 — 자기 확신도는 못 믿는다**: LLM이 스스로 매기는 confidence는 부정확 → 내 실제 데이터로 임계값을 실측해야 함.
- **함정 2 — 에스컬레이션이 40% 넘으면** 오히려 오버헤드가 손해. 대부분이 어려운 요청이면 그냥 강 모델 직행.
- **함정 3 — 지연**: 캐스케이드는 싼 모델 지연이 매번 붙음 → 실시간·대화형엔 불리, 비동기·대량 처리엔 유리.
- 구현 순서 추천: ① 시맨틱 캐시(중복 요청 3.4배 빠름) → ② 의도 라우팅 → ③ 확신도 캐스케이드 → ④ 폴백 체인.

**타사 모델 포함**: 라우팅은 본질적으로 **모델 무관** — Claude + GPT + Gemini를 한 라우터 뒤에 두고 태스크별 최적 모델로 보내는 게 정석. Claude API엔 정책 거절 시 다른 모델로 넘기는 **서버측 `fallbacks`** 기능도 있음(신뢰도 A, 내부 문서).

### 패턴 ④ — 자산 생성 (강 모델이 스캐폴딩을 "제작") ★한 번 만들면 계속 이득

**뭐냐**: 강 모델을 매번 추론에 쓰는 대신, 강 모델로 **재사용 자산을 미리 만들어** 두고 그 자산을 싼 모델이 소비하게 한다. 만드는 자산 4종:

1. **프롬프트/시스템 프롬프트 (메타프롬프팅)**: 강 모델이 "내용"이 아니라 **"구조·틀"**을 만들어 줌 → 싼 모델이 그 틀만 따르면 되니 토큰도 절약(신뢰도 C). *구조 우선, 예시 최소.*
2. **few-shot 예시 / 합성 데이터**: 강 모델(교사)이 양질의 문답 쌍·예시를 생성 → 싼 모델이 이를 참고하거나 파인튜닝. 좁은 태스크에선 소형 모델(+4~6점)이 때때로 교사 모델을 추월(신뢰도 B/C, arxiv).
3. **평가셋(eval) / LLM-as-judge**: 강 모델이 테스트 케이스·채점 기준을 만들어 두면, 싼 모델 출력을 자동 검증·에스컬레이션 판단에 사용.
4. **규칙 문서(CLAUDE.md·스킬·커맨드)**: ★내 상황에 가장 직접적 — **Fable 5로 CLAUDE.md·SKILL.md·커맨드를 정교하게 작성**해 두면, 이후 저렴한 모델로 돌려도 워크스페이스가 "똑똑하게" 동작.

**어떻게 (개인/Claude Code 실전)**:
- Fable 5 세션에서 "이 반복 작업용 시스템 프롬프트/스킬을 설계해줘 + 실패 예시 3개 + 검증 체크리스트도" 라고 자산을 뽑아낸다.
- 그 자산을 `.claude/skills/`·`CLAUDE.md`·커맨드로 박아두고, 일상 작업은 더 싼 모델로 실행.
- **결정론적 반복은 코드로**: 같은 추출이 반복되면 Fable 5로 파이썬 스크립트를 짜게 해 `scripts/`에 분리 → 모델을 아예 안 쓰고 같은 입력=같은 출력 보장(내 deep-research 스킬의 원칙과 동일).

## ③-b 내 상황(Claude Code + API)용 실전 레시피 3개

1. **"막힐 때만 Fable"** (①의 개인판): 평소 작업은 기본 모델, 설계 결정·반복 에러·최종 검토 순간에만 Fable 5에게 계획을 물어 실행에 반영. → `/stuck`과 결합.
2. **"자산을 Fable로 굽는다"** (④): 새 스킬·커맨드·CLAUDE.md를 만들 땐 항상 Fable 5로 초안+실패예시+eval을 함께 뽑아 박아둔다. `skillers-suda`의 **풀 모드(eval 서브에이전트)**가 이걸 자동화한 형태.
3. **"검증은 별도 모델 눈으로"** (②+④): 산출물 품질이 중요하면, 만든 모델과 **다른 세션/모델**로 교차 검토(fresh context 검증자). 자기 검토보다 편향을 잘 잡음.

## ③-c 공통 원칙 (모든 X 노하우가 반복하는 것)

- **비싼 모델은 "판단"에, 싼 모델은 "실행"에.** 지능은 결정점에 몰아준다.
- **위임 지시는 구체적으로.** 목표·출력형식·경계·도구를 명시 — 막연하면 다 실패.
- **컨텍스트 격리.** 서브에이전트/조언자는 필요한 것만 보게 해 오염 방지.
- **확신도는 실측.** 모델 자기평가 믿지 말고 내 데이터로 임계값 잡기.
- **자산화하면 복리.** 프롬프트·규칙·eval을 한 번 잘 만들면 계속 이득.
- **결정론 반복은 코드로 빼라.** 모델 대신 스크립트 → 재현성·비용 둘 다 해결.
