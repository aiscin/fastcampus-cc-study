01 · 한 문장 원칙
비싼 모델은 판단에, 싼 모델은 실행에.

지능은 결정점에만 몰아준다. 위임 지시는 구체적으로, 확신도는 실측으로, 반복은 코드로, 자산은 한 번 잘 만들어 복리로.

02 · 4가지 패턴 지도
패턴한 줄 정의강 모델 역할효과(근거)신뢰도
① 어드바이저 위임싼 모델이 막히면 강 모델에게 “훈수”만결정점에서 계획·교정점수 19.7→41.2%, 85%↓*A C*
② 오케스트레이터-워커강 모델이 계획 세워 서브에이전트 병렬 분배총괄 지휘관+90.2% / 토큰 ~15×AB
③ 라우팅·캐스케이드쉬운 건 싼 모델, 어려운 것만 강 모델로어려운 케이스 전담품질 95% 유지·비용 45~85%↓BC
④ 자산 생성강 모델이 프롬프트·규칙·예시·eval 제작스캐폴딩 제작자소형 모델 +4~6점BC

* 어드바이저의 “19.7→41.2%·85%↓”는 3rd-party 요약 기준(C). 패턴 존재·작동은 공식 문서(A).

03 · 이럴 땐 이 패턴
막힐 때 / 설계 결정→ ① 어드바이저. 실행은 싼 모델, 결정점만 Fable에게 물어본다.
넓게 훑는 리서치→ ② 오케스트레이터. 병렬 서브에이전트. 단, 코딩·강한 의존 작업엔 비추.
요청량 많은 서비스→ ③ 라우팅·캐스케이드. 쉬운 건 싼 모델로 흘리고 어려운 것만 승격.
반복 작업 세팅→ ④ 자산 생성. Fable로 프롬프트·CLAUDE.md·스킬·eval을 구워둔다. (한 번 이득이 계속)

04 · 패턴별 매뉴얼
Pattern / Advisor★ 최우선 · 즉시 체감패턴 A

① 어드바이저 위임 — 작은 실행자 + 큰 조언자

뭐냐
싼·빠른 실행자(Haiku/Sonnet)가 일하다 방향 결정·반복 에러·최종 검토 순간에만 강한 조언자(Fable 5/Opus)에게 전체 맥락을 넘겨 계획·교정을 받는다. 토큰 대부분은 싼 값, 품질은 강 모델 근접.
규칙
조언자는 실행자보다 같거나 더 똑똑해야(최소 Sonnet 4.6, 최상 Fable 5). 예: 실행자 Haiku 4.5 → 조언자 claude-fable-5. 호출 시점은 모델이 결정.
어떻게
API: top-level model=싼 실행자, 툴 내부 model=강 조언자, 베타 advisor-tool-2026-03-01. · Claude Code: 평소 싼 모델로 작업하다 “막혔어요”/설계 순간에만 강 세션·서브에이전트에 계획을 묻고 실행에 반영(내 /stuck 흐름과 결합).
근거
Haiku+Opus조언 점수 2배↑(19.7→41.2%), Sonnet 단독보다 85%↓ C* · 패턴·모델호환·튜닝(nudge +7%p)은 공식 A
Pattern / Asset-Gen★ 복리 효과패턴 B/C

④ 자산 생성 — 강 모델이 스캐폴딩을 “제작”

뭐냐
강 모델을 매번 추론에 쓰지 말고, 재사용 자산을 미리 굽는다: ① 시스템 프롬프트(구조 우선=메타프롬프팅) ② few-shot·합성 예시 ③ eval·채점 기준 ④ CLAUDE.md·스킬·커맨드.
왜 좋아
④가 내 상황에 가장 직접적 — Fable로 규칙 문서를 정교하게 써두면 이후 싼 모델로 돌려도 워크스페이스가 똑똑하게 동작. 합성 예시로 소형 모델 +4~6점, 좁은 태스크선 교사 추월 사례도.
어떻게
Fable 세션에 “이 반복작업용 시스템 프롬프트 + 실패 예시 3개 + 검증 체크리스트”를 통째로 뽑아 .claude/skills/·CLAUDE.md에 박기. skillers-suda 풀 모드(eval 서브에이전트)가 이걸 자동화한 형태. 결정론 반복은 Fable로 스크립트 짜 scripts/로 분리.
Pattern / Orchestrator무거운 리서치용패턴 A/B

② 오케스트레이터-워커 (멀티에이전트)

뭐냐
강한 리드가 질문을 쪼개 계획하고, 각자 독립 컨텍스트의 서브에이전트를 병렬로 띄워 분담시킨 뒤 결과를 합친다. 리드는 직접 일 안 하고 계획·분배·통합만.
핵심
위임 지시를 구체적으로(목표·출력형식·도구·경계). 막연하면 중복·누락. 복잡도별 규모: 단순 1명 / 비교 2~4명 / 복잡 10명+. 3~5개 동시 실행.
피할 때
에이전트 간 강한 의존·공유 맥락, 대부분의 코딩, 실시간 조율.
근거
리드 Opus4+서브 Sonnet4가 단일 Opus4 대비 +90.2%, 단 채팅 대비 ~15배 토큰(Anthropic) AB
Pattern / Routing서비스·API용패턴 B/C

③ 라우팅 & 캐스케이드 (비용 최적화)

뭐냐
라우팅=요청을 선분류해 쉬운 건 싼 모델·어려운 건 강 모델로 한 번에. 캐스케이드=제일 싼 모델부터 시도 후 확신 낮으면 강 모델로 승격.
함정
① 자기 확신도는 부정확 → 내 데이터로 임계값 실측. ② 승격률 40% 넘으면 오버헤드 손해. ③ 캐스케이드 지연은 실시간엔 불리, 비동기엔 유리.
근거
품질 95% 유지하며 비용 45~85%↓(RouteLLM: GPT-4를 14%만 써도 95% 달성) BC
05 · 타사 혼합 라우팅 (Claude + GPT + Gemini) — 심화

2026 현재 최고 성과 팀은 한 곳에 락인하지 않고 업무별로 갈아탄다. 모델별 “성향”(벤치 편차 크니 확정은 내 데이터로 실측):

업무 성향강한 쪽비고
코딩·에이전트·긴 신중한 추론Claude하드 리더보드 강세
순수 추론·수학·과학(스텝바이스텝)GPT o-series딥 추론
최장 컨텍스트(1M)·멀티모달·최상위 가성비Gemini코드베이스 통째로
Tool / call-it

OpenRouter

API 하나로 300+ 모델. openrouter/auto가 프롬프트별 최적 모델 선택(Not Diamond·비용/품질 0~10 다이얼). 폴백=프로바이더 자동 + models 배열. 실패 요청 무과금.

Tool / host-it

LiteLLM

자체 호스팅 프록시. 로드밸런싱·폴백체인·감사로그·지출관리. 팀·거버넌스용.

Tool / Claude Code★ 내 상황에 직접

Claude Code Router

로컬 프록시 + config.json 슬롯 매핑. 배경작업→싼 모델, 코딩→Sonnet, 복잡추론→Opus를 수동 /model 없이 자동 분배.

폴백·페일오버 규칙(운영, C): 상태코드 기반(429→즉시 타 프로바이더 · 500→재시도 · 가드레일 위반→대체 모델) · 지연 임계 헤징(2초 안 오면 백업 동시 요청, 먼저 온 것 사용) · 프로액티브 분산(70/30) · 조건부 라우팅(저우선=싼 모델).

06 · 당신의 질문에 답
Q → 업무별로 모델을 자동 지정하는 “훅”을 추가하면?
방향은 정확히 맞다. 단, ‘전환’은 훅이 아니라 라우터/서브에이전트로, ‘검증’은 오프라인 eval로.

업무→최적모델 매칭은 라우팅 그 자체라 좋은 직관입니다. 다만 Claude Code 훅은 세션 모델을 태스크마다 바꾸지 못합니다(훅=컨텍스트 주입·차단·조언·로깅용). ‘지정’은 다른 도구가 맡아야 합니다.

원한 것훅으로 되나?올바른 도구
업무 분류 (이건 긴문서 / 저건 코딩)✔ 됨UserPromptSubmit 훅 (분류·조언 주입 + 로깅)
모델 전환/지정△ 훅 불가Claude Code Router · 서브에이전트 model 핀 · (API) LiteLLM/OpenRouter
적합한지 검증△ 실시간 부정확오프라인 eval 게이트 50~500케이스(규칙 정할 때 1회)

권장 하이브리드 — 훅은 “이 일엔 이 모델이 좋겠다”고 알려주고 로깅, 실제 태우는 건 라우터/서브에이전트, 그 규칙이 맞는지는 eval로 한 번 검증. 싼 모델로 바꿀 때의 “조용한 품질 저하(silent quality regression)”가 숨은 세금이라, 검증 없이 라우팅부터 넣지 말 것. (‘훅 불가’는 현재 이해 — Part 05 Hook 클립/공식 문서로 확인 권장)

07 · 내 워크스페이스 적용 레시피
R1 “막힐 때만 Fable” — 평소 작업은 기본 모델, 설계 결정·반복 에러·최종 검토 순간에만 Fable에게 계획을 물어 실행에 반영. /stuck과 결합. (패턴 ①)
R2 “자산을 Fable로 굽는다” — 새 스킬·커맨드·CLAUDE.md는 항상 Fable로 초안+실패예시+eval을 함께 뽑아 박아둔다. skillers-suda 풀 모드가 자동화판. (패턴 ④)
R3 “지정은 라우터·서브에이전트로” — Claude Code Router config.json으로 배경=싼 모델/코딩=Sonnet/설계=Fable 자동 분배. 또는 서브에이전트에 model을 직접 핀(“리서치=싼모델, 설계=Fable”). (패턴 ③ + 당신의 아이디어)
R4 “검증은 다른 눈으로” — 라우팅 규칙 넣기 전 eval 50~500케이스로 1회 검증. 산출물 품질이 중요하면 만든 모델과 다른 세션/모델로 교차 검토(fresh context 검증자). (패턴 ②·④)

한계 (정보 기준 2026-07-09):

  • 어드바이저 툴은 베타(advisor-tool-2026-03-01) — 파라미터·모델 호환이 바뀔 수 있음, 적용 전 최신 문서 재확인.
  • 구체 수치는 태스크 의존. 특히 “19.7→41.2%·85%↓”는 3rd-party 요약 기준 — 내 워크로드에서 직접 측정 필요.

  • 라우팅 절감 45~85%는 벤치별 편차 큼 → 참고치. distillation 수치는 파인튜닝·오픈소스 소형모델 대상 연구.
