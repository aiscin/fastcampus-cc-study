## 0. 두 대상

## 1. 파일 아키텍처 — 한눈 비교

### A. 플러그인 (멀티 컴포넌트)
```

│   │   ├── SKILL.md               # 본체 7-Phase (575줄)

│   │   │   ├── phase_contracts.md (322)  query_generator.md (405)
│   │   │   ├── tool_strategy.md (422)    citation_rules.md (230)

│   │   │   └── query_schema.json

│   │   │   └── pipelines.py (396)     # 에이전트 프롬프트·합성 파이프라인
│   │   ├── assets/templates/      # 산출물 양식 5종(요약·섹션·참고문헌·README·website.html)

### B. 커스텀 스킬 (단일 파일)
```
.claude/skills/deep-research/
└── SKILL.md  (153줄)   # ⓪범위 ①뼈대 ②키워드분해 ③검색루프 ④교차검증 ⑤보고서 ⑥리서치노트

➡️ **구조 차이의 본질**: A는 "**배포 가능한 제품**"(매니페스트·버전·커맨드·업데이트 점검까지), B는 "**한 파일짜리 개인 도구**".

## 2. 진입점 · 배포 구조

| | A. 플러그인 | B. 커스텀 스킬 |
|---|---|---|
| 진입 | `commands/deep-research.md`(슬래시) + 스킬 description 자동발동 **2중 진입** | SKILL.md `description` 자동발동 **1개** |
| 인자 처리 | `topic / resume / status / query` 4종 라우팅 | 없음 (주제 자연어만) |
| 권한 | 커맨드에 `allowed-tools` 명시(Agent·Web·Bash 등) | 별도 선언 없음 |
| 배포 | 마켓플레이스 설치 + `setup.sh` + 업데이트 점검(.cjs) | 폴더에 파일 두면 끝 |
| 버전관리 | `plugin.json` version + CHANGELOG | 없음(파일 수정 = 버전) |

➡️ A는 **남에게 나눠주는 패키지**라 진입·권한·설치·갱신을 다 갖췄고, B는 **나만 쓰는 파일**이라 그 층이 전부 생략됐다.

## 3. 컨텍스트 로딩 전략 — 점진적 공개(Progressive Disclosure)

- **A**: 본체 SKILL.md(575줄)는 "무엇을·왜"만 담고, "어떻게 자세히"는 `references/` 7개 파일로 **분리해 필요할 때만 읽게** 한다. → 대형 스킬을 다루는 정석 패턴. 단, 본체 자체가 이미 575줄로 무겁다.
- **B**: 153줄 안에 6단계를 **자기완결**로 담는다. 참조 분리 없음 → 한 번에 다 읽힘. 작아서 점진적 공개가 **필요 없는** 크기다.

➡️ 점진적 공개는 "**커야 의미**"가 있다. A는 규모상 필요해서 적용, B는 작아서 불필요 — **둘 다 자기 규모엔 합리적**.

## 4. 스크립트 분리 — 강의 핵심 원칙으로 본 차이 ★

> 강의 원칙: "결정론 단계(API·파싱·상태관리)는 **스크립트로 분리**해 재현성 확보. AI에 맡길 건 문서로 자유도."

| | A. 플러그인 | B. 커스텀 스킬 |
|---|---|---|
| 결정론 분리 | `orchestrator.py`(상태머신)·`pipelines.py`로 **분리** | 분리 없음 — 전 단계 산문(prose)로 AI가 해석 |
| 상태 표현 | `ResearchPhase` Enum 7단계 + `state.json` 스키마 | 없음(단, ⑥ 리서치노트가 *과정 기록*을 대신) |

**해석 (중요)**: 강의 원칙대로면 "결정론은 스크립트로"가 정답이지만, **그 원칙은 결정론 단계가 있을 때 성립**한다.
- A는 상태·단계 관리라는 결정론 로직이 실제로 있어 파이썬 분리가 타당.
- B의 작업(검색·판단·작성)은 **대부분 AI 추론**이라 떼어낼 결정론이 거의 없다 → 스크립트가 없는 게 오히려 맞다.
- 단 B도 **반복되는 결정론**(예: 다이닝코드 URL 호출·영업시간 파싱)이 늘면, 그 부분만 `scripts/`로 빼는 게 강의 원칙에 부합.

➡️ "스크립트 분리 = 무조건 좋음"이 아니라 **결정론이 있는 만큼만 분리**가 핵심.

## 5. 재사용 자산(템플릿·스키마·예시)

| | A. 플러그인 | B. 커스텀 스킬 |
|---|---|---|

➡️ A는 **재현성·일관성**을 템플릿/스키마로 강제(같은 양식 보장). B는 그 일관성을 AI 해석에 맡겨 **변동 가능**하지만 가볍다.

## 7. 결론 — 구조가 말해주는 것

1. **A(플러그인)은 "제품 아키텍처"**: 매니페스트·커맨드·스크립트 상태머신·템플릿·스키마·업데이트 점검·멀티스킬까지 — **재현성과 배포·자산화에 최적화**. 대신 무겁고 로컬 실용 주제엔 과설계.
2. **B(커스텀 스킬)은 "개인 워크플로우 아키텍처"**: 단일 파일에 ⓪범위질문·⑥리서치노트로 **메타 흐름 철학을 구조에 새김**. 가볍고 빠르며 수정·포크 용이. 대신 결정론 강제·배포 층이 없다.
3. **강의가 가르치는 정답은 "A를 베끼기"가 아니라 "B처럼 시작해 필요한 만큼만 A의 부품을 흡수"** 다 —
   - 결정론이 생기면 → `scripts/` 분리(4장)
   - 양식 일관성이 필요하면 → `templates/` 도입(5장)
   - 본체가 커지면 → `references/` 분리(3장, 점진적 공개)
   - 남에게 줄 거면 → `plugin.json`+`command`+`setup`로 패키징(2장)

> **한 문장**: 플러그인은 *완성된 제품의 구조*, 커스텀 스킬은 *성장하는 개인 도구의 구조* — 실습24의 목표는 B를 **필요한 부품만 골라** A 쪽으로 키우는 것.
