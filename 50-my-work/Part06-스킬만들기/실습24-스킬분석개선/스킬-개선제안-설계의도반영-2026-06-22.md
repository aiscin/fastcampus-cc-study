# 설계 의도 → 스킬 개선 제안

> 작성일: 2026-06-22 · 실습24
> 레퍼런스 스킬 설계자의 7가지 가치를 **현재 커스텀 deep-research 스킬**에 적용한 개선안.
> 원칙: 이미 담긴 건 *강화*, 사고방식에서 아직 코드화 안 된 건 *추가*. (베끼기 아닌 흡수)

---

## 0. 진단 — 7가지 의도가 지금 얼마나 담겼나

| # | 설계 의도 | 현재 스킬 | 개선 여지 |
|---|---|---|---|
| 1 | 검색이 아니라 검증 | ✅ ③fetch·④교차검증 강함 | 🔶 검증 *결과를 보이게* (등급) |
| 2 | AI 믿되 가두리 | ✅ 포화 종료 3조건 | 🔶 fetch 예산·비용 인식 부재 |
| 3 | 과정 기록(노트) | ✅ ⑥ 리서치 노트 | 🔶 1회성 — 누적 안 됨 |
| 4 | 사용자를 루프에 | ✅ ⓪ 범위 질문 | 🟢 양호 |
| 5 | 현장 학습 누적 | 🔶 메모 1줄(웹아카이브)뿐 | 🔴 **누적 메커니즘 없음** |
| 6 | 판단 AI / 수집 도구 | ✅ 역할 분리 표 | 🔴 **결정론 반복을 스크립트로 안 뺌** |
| 7 | 의견형 기본값 | ✅ 형식 고정·시각화 X | 🔶 분량·골격 기본값 부재 |

→ **가장 큰 빈칸은 #5(현장 학습 누적)와 #6(스크립트 분리)** — 설계자의 사고방식엔 있는데 스킬엔 아직 안 박힌 부분.

---

## 🥇 Tier 1 — 적은 노력, 큰 효과 (지금 바로)

### T1-1. 검증을 "보이게" — 출처 신뢰도 등급 (의도 #1 강화)
검증은 강한데 *결과가 안 보인다*. 표/출처에 A~E 등급 1열만 추가.
```markdown
## ④ 교차 검증 + 신뢰도 등급
- 각 주장/출처에 등급을 단다: A(공식·1차) B(신뢰매체·플랫폼 본문) C(일반매체·블로그) D(기업/홍보) E(일화·SNS)
- 단일 출처·본문 차단 항목은 그 사실을 표에 명시.
→ ⑤ 보고서 표에 「신뢰도」 열, 출처 목록에 등급 병기.
```

### T1-2. 현장 메모를 "살아있는 로그"로 (의도 #5 — 핵심 빈칸) ★
지금은 "웹아카이브 차단(2026-06)" 한 줄뿐. 이걸 **계속 자라는 섹션**으로 승격해, 리서치 중 막힌 걸 발견할 때마다 **날짜 찍어 append**하게 한다. → 스킬이 쓸수록 똑똑해짐(자산화).
```markdown
## 현장 메모 (계속 갱신 — 막히면 여기 한 줄 추가)
> 리서치 중 새 차단/우회법을 발견하면 날짜와 함께 아래에 덧붙인다.
- 2026-06: web.archive.org WebFetch 차단 → 우회용 금지
- 2026-06: 다이닝코드 프로필 = 차단 적고 영업시간·가격·평점 정형 → 로컬 검증 1순위
- (다음 발견 시 여기에 append)
```

### T1-3. 의견형 기본값 보강 (의도 #7)
"안 물어도 되는 건 고정"을 더 밀어붙인다.
```markdown
- 목적 미입력(질문 생략) 시 → "범용 종합" 기본 골격으로 진행(멈추지 않음).
- 수록 항목 상한: 한 카테고리 20~30개 이내(과밀 방지, 1장 가독성).
- 넓은 주제만 권역/섹션 골격 사용 — 좁은 주제는 단일 표 유지.
```

## 🥈 Tier 2 — 중간 노력 (반복이 쌓이면)

### T2-1. 결정론 반복 → 스크립트 분리 (의도 #6 — 강의 핵심 원칙) ★
역할 분리 표는 "철학"으로만 있고 실제 분리가 없다. **다이닝코드 프로필 호출·파싱**처럼 매번 똑같이 도는 결정론이 잦아지면 `scripts/`로 뺀다.
```
.claude/skills/deep-research/
├── SKILL.md
└── scripts/
    └── fetch_profile.py   # URL → {상호·영업시간·가격·평점} JSON (같은 입력=같은 출력)
```
- SKILL.md ③-c는 "프로필성 페이지는 `fetch_profile.py`로 정형 추출"로 연결.
- 효과: 영업시간·가격 추출이 **AI 해석 변동 → 결정론 고정**(재현성↑). 강의의 "결정론은 코드로" 정합.
- 주의: 분리는 *반복이 실제로 생긴 뒤*. 미리 만들면 과설계(의도 #2의 절제와 충돌).

### T2-2. 노트를 append 친화 구조로 (의도 #3 + #5 연결)
⑥ 리서치 노트의 "미해결/다음에 보강할 것"을 **다음 실행 때 먼저 읽는 입력**으로 쓰게 한다. → 1회성 노트가 *연속 리서치의 메모리*가 됨.
```markdown
## ⑥ 리서치 노트
4. 미해결 / 다음 보강 → (다음 동일주제 리서치 시 ⓪ 직후 이 항목을 먼저 로드)
```

## 🥉 Tier 3 — 큰 노력 (공유·배포 단계)

### T3-1. fetch 예산 명시 (의도 #2 강화)
"가두리"를 비용까지 확장. 한 바퀴 fetch 2~4개(이미 있음) + **세션 총 fetch 상한**(예: 12개)을 둬 폭주·비용을 막는다.

### T3-2. 최소 패키징 (남에게 줄 때만)
배포하려면 `plugin.json` + 슬래시 커맨드 1개만 추가. **전면 7-Phase·13파일은 끝까지 채택 안 함**(로컬 실용 주제엔 과설계 — 통제 비교 35 vs 46이 근거).

## 우선순위 한눈에

```
지금 바로 (Tier 1):  T1-2 현장 로그 ★  >  T1-1 등급화  >  T1-3 기본값
반복 쌓이면 (Tier 2): T2-1 스크립트 분리 ★  >  T2-2 노트 연속화
배포할 때 (Tier 3):   T3-1 fetch 예산  >  T3-2 최소 패키징
```

★ = 설계자의 사고방식엔 있었지만 스킬엔 아직 없던, **이번 분석의 진짜 수확**.

## 한 문장

> 설계자는 이미 "검증·기록·사용자 참여"는 스킬에 박았지만, **자기 머릿속 '현장에서 배운다(#5)'와 '결정론은 떼어낸다(#6)'는 아직 글로만 남겼다.** 이 둘을 *살아있는 로그*와 *스크립트 분리*로 코드화하면, 스킬이 **쓸수록 똑똑해지고 매번 같은 결과를 내는** 도구로 한 단계 올라선다.
