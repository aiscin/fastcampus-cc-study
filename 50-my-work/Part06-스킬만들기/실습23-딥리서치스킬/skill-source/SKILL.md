# deep-research — 딥리서치 보고서 스킬

주제 하나를 받아 **출처가 달린 마크다운 보고서 한 장**으로 정리한다.
검색만 하는 얕은 리서치가 아니라, 유망 URL을 골라 **본문을 직접 가져와(fetch)** 읽고
서로 다른 출처를 **교차검증**한 뒤 보고서를 쓴다.

> 이 흐름은 "어떻게 워크플로우를 구성해야 할까?"를 먼저 조사한 뒤
> 사용자가 보강해서 정의한 것이다. 단계를 임의로 건너뛰지 말 것.

---

## ⓪ 범위 확인 — 모호하면 먼저 묻는다

주제를 받으면 아래 **3개 항목**이 명확한지 점검한다.
모호한 항목만 골라 `AskUserQuestion`으로 묻고 사용자가 선택하게 한다.
**혼자 가정하고 검색을 시작하지 않는다.**

| 항목 | 묻는 것 |
|---|---|
| **범위** | 어디까지·얼마나 넓게 |
| **목적** | 왜 필요한지·어디에 쓸지 |
| **기준** | 무엇을 우선으로 볼지 |

- 결과형식은 **마크다운 보고서로 고정** — 묻지 않는다.
- 3개 항목이 이미 주제에 다 드러나면(예: "수원역 5천원 이하 김밥집") 질문을 생략하고 ①로 바로 간다.
- 질문은 항목당 객관식 2~4개 + "직접 입력" 여지.

---

## ① 보고서 뼈대 잡기

⓪에서 좁혀진 범위로 **최종 보고서의 목차를 먼저** 만든다.
이 뼈대가 이후 검색의 길잡이가 된다. (예: 지역별 / 기준별 / 후보 리스트 / 한계)

---

## ② 키워드 분해 + 검색 쿼리

**분해** — 주제를 3~5개 **축**으로 펼친다.
- 넓은 축 → 좁은 축 순서
- 축끼리 의미가 겹치지 않게
- ⓪ 답변에 근거한 축만 (사용자가 안 고른 관점은 만들지 않음)

**쿼리** — 축을 조합해 검색어를 만든다. 한 번에 다 넣지 않는다.
- (1) 넓은 쿼리 먼저 → 전체 지형·자주 언급되는 후보 파악
- (2) 축 × 축 조합 → 빈칸 채우기
- (3) 검증용 쿼리 → 후보를 다른 각도로 재확인

한국 주제는 **검색어를 한국어로** 명시한다(WebSearch는 미국 기준 결과라 영어권 결과가 섞일 수 있음).

---

## ③ 반복 검색·요약 루프 ⟲

한 바퀴 안에서 **검색 → 선별 → 본문 가져오기**를 돈다.

```
③-a  WebSearch       쿼리 던져 후보 URL + 스니펫 수집
③-b  URL 선별         스니펫 보고 깊이 볼 가치 있는 URL 2~4개 선별
                      (출처 종류를 섞는다: 블로그·뉴스·공식 — 한 종류 쏠림 금지)
③-c  WebFetch        고른 URL 본문을 직접 가져와 핵심 추출
                      (스니펫엔 없는 영업시간·실후기·가격 등 디테일 확보)
```

검색 결과는 **원본을 통째로 쌓지 말고 핵심만 추려** 누적한다.

**루프 횟수 — 포화로 멈춘다(카운터로 세지 않는다)**
- 기본 3~4바퀴, 넓은 주제는 최대 6바퀴
- 아래 중 하나면 종료:
  1. 직전 바퀴 대비 **새 정보 < 20%** (수확 체감)
  2. **핵심 주장마다 출처 2개 이상** 확보
  3. **상한선**(기본 4 / 깊게 6) 도달

WebFetch가 막히는 사이트(로그인·봇 차단·JS 렌더링)는 건너뛰고 ③-b에서 고른 다른 URL로 대체한다.

---

## ④ 교차 검증

가져온 본문들을 **서로 대조**한다.
- 충돌하는 정보 식별(예: 한쪽은 "평점 4.5", 다른 쪽은 "폐업")
- 신뢰도 낮은 정보는 걸러내거나 "확인 필요"로 표시
- 한 출처에만 있는 주장은 다른 출처로 재확인

---

## ⑤ 마크다운 보고서 + 출처

①에서 짠 뼈대에 검증된 내용을 채워 **보고서 한 장**으로 묶는다.
한국어로 정리하며, **시각화(차트·그래프·이미지)는 만들지 않는다.**

보고서 구성:
1. **핵심 한 줄 요약**
2. **카테고리별 표** (뼈대 목차대로)
3. **출처 목록** — 모든 주장은 출처로 추적 가능해야 함 (마크다운 링크)
4. **한계** — 정보 기준 시점, 못 가져온 출처, 확인 필요 항목

저장 위치: 호출한 작업 폴더(또는 사용자가 지정한 곳)에 `{주제}-{날짜}.md`.

---

## 현실 한계 (보고서 "한계"에 반영)

- **WebSearch 미국 기준** — 한국 로컬 주제는 한국어 검색어 명시, 부족분은 fetch로 보완
- **WebFetch 차단 가능** — 로그인·봇차단·JS 페이지는 본문 못 가져올 수 있음 → 다른 URL로 대체
- **실시간성 한계** — fetch한 페이지가 최신이 아닐 수 있음 → "정보 기준 시점" 명시

## 역할 분리 — 판단은 AI, 수집은 도구

| AI(추론) | 도구(결정론) |
|---|---|
| 어떤 축으로 쪼갤지 | 검색 `WebSearch` |
| 어떤 검색어를 만들지 | 본문 수집 `WebFetch` |
| 어떤 URL이 유망한지 | 범위 질문 `AskUserQuestion` |
| 충돌 정보 판단 | 파일 저장 `Write` |
