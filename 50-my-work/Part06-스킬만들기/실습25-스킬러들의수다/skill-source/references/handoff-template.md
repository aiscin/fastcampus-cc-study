# 핸드오프 문서 템플릿 (Step 6)

받는 쪽(클로드코드 또는 사람)이 다시 캐묻지 않고 곧장 구현에 착수할 수 있도록, 아래 형식으로 `handoff-{한줄제목}.md`를 작성한다.

```markdown
# 핸드오프 — {한 줄 제목}

생성: {날짜} · ideation-funnel

## 개요
- **정의**: {한 줄 정의}
- **왜**: {해결할 불편 / 만드는 이유}

## 레퍼런스 정리
- 재활용할 기존 자산: {스킬/도구명 또는 "없음"}
- 차별점: {기존과 다른 점}

## 핵심 기능 (우선순위)
1. {기능 1} ★MVP
2. {기능 2} ★MVP
3. {기능 3} (나중)

## 입력 / 출력
- **입력**: {사용자가 주는 것 — 텍스트/파일/URL}
- **출력**: {받는 결과물 형태}

## 구현 단계
1. {단계 — 필요한 도구·파일·API}
2. {단계}
3. {단계}

## 첫 실행
```
{바로 시작할 첫 명령어 또는 첫 프롬프트}
```

## 수용 기준 (이게 되면 완성)
- [ ] {체크 1}
- [ ] {체크 2}

## 다음 행동
{예: 이 문서로 skillers-suda 호출해 스킬화 / 바로 클로드코드에 붙여넣어 제작 시작}
```

## 원칙

- **빈칸을 남기지 않는다** — `{...}`를 추측이 아니라 인터뷰·명세에서 확정된 값으로 채운다. 모르는 칸은 비워두지 말고 Step 5로 돌아가 묻는다.
- **첫 실행은 진짜 실행 가능해야** — "잘 만들어보자" 같은 추상 문장이 아니라, 복붙하면 동작이 시작되는 명령어/프롬프트를 넣는다.
- **수용 기준은 관찰 가능하게** — "좋은 결과"가 아니라 "X를 입력하면 Y 파일이 생긴다"처럼 확인 가능한 문장으로 쓴다.
