# 아이디어 깔때기 (Ideation Funnel)

> 막연한 "뭔가 만들고 싶다"를 "지금 바로 클로드코드로 만들 수 있는 한 가지"로 좁혀준다.

이 스킬은 만들기 **전 단계**를 담당한다. 머릿속 뭉게구름을, 그대로 붙여넣어 "이거 만들어줘"라고 쓸 수 있는 명세 한 장으로 바꾸는 것이 목표다. 분석만 하고 끝나면 실패다 — 항상 다음 행동(Next Step)으로 이어져야 한다.

## 핵심 원칙

- **인터뷰 먼저, 후보 나중** — 의도를 모르면 엉뚱한 걸 구체화한다. 단 질문은 최대 3개로 압축한다. 질문이 많으면 사용자가 지쳐 이탈하기 때문이다.
- **바닥부터 만들지 않는다** — 비슷한 게 이미 있으면 재활용하거나 차별점을 잡는다. 그래서 후보를 내기 전에 레퍼런스를 먼저 스캔한다.
- **중간에 멈춰도 자산** — 각 단계 결과를 draft 파일에 누적 저장한다. 사용자가 중간에 그만둬도 정리된 내용이 남아 이어갈 수 있어야 한다.
- **거창함 경계** — 결과가 너무 크고 복잡하면 사용자가 "이건 나 혼자 못 하겠다"며 의욕을 잃는다. 항상 개인이 지금 만들 수 있는 MVP 범위로 좁힌다.

## 워크플로우

### Step 1: 발산 — 의도 좁히기
**타입**: prompt

사용자의 한 줄 생각에서 핵심 키워드와 맥락을 뽑는다. 그다음 의도를 좁히는 질문을 **최대 3개**까지만 AskUserQuestion으로 던진다. 기본 3축은 "왜 만들고 싶은가 / 누구를 위한 것인가 / 어떤 불편을 해결하는가"다. 사용자가 이미 일부를 말했으면 그 항목은 건너뛰고 빈 칸만 묻는다.

답을 받으면 **이번 실행 전용 폴더**를 먼저 만든다 — 반복 실행 때 문서가 섞이지 않게 하기 위해서다. 워크스페이스 최상위에 `ideation/{YYYYMMDD-HHMM}-{한줄제목-슬러그}/` 형태로 만들고, 그 안에 `resource/` 하위 폴더도 같이 만든다(레퍼런스 검토·리서치 산출물 보관용). 타임스탬프를 넣어 같은 주제를 또 돌려도 겹치지 않게 한다. 슬러그는 아이디어를 영문/숫자·하이픈으로 짧게 줄인 것. 실행 폴더 안에 `ideation-draft.md`를 만들고 "원문 + 좁힌 의도"를 첫 섹션으로 적는다. 이후 모든 단계는 이 draft에 결과를 이어 붙인다(아래 Drafts 참조) — 사용자가 중간에 멈춰도 정리된 내용이 남게 하기 위해서다.

이 실행 폴더 경로는 한 번 정하면 Step 6까지 그대로 쓴다. draft와 최종 핸드오프가 같은 폴더에 모이도록.

입력 방어 — 이 단계에서 비정상 입력을 먼저 처리하는 이유는, 모호하거나 범위가 큰 입력을 그대로 구체화하면 뒤 단계가 전부 헛돌기 때문이다:
- **빈 입력**: "어떤 걸 만들고 싶은지 한 줄이라도 적어줘"로 안내하고 멈춘다. 구체화 단계로 넘어가지 않는다.
- **도메인 무관 입력**("저녁 메뉴 추천"): "이 스킬은 클로드코드로 만들 아이디어를 정리하는 용도예요"라고 범위를 알리고 재입력을 청한다.
- **범위 초과 입력**("AGI 만들어줘", "전 세계가 쓸 SNS"): "혼자 클로드코드로 지금 만들 수 있는 범위로 줄이면?" 스코핑 질문을 넣어 개인 MVP로 축소한다.
- **장문 입력**(여러 아이디어 뒤섞임): "핵심 아이디어를 한 문장으로 요약해줘"를 먼저 청해 초점을 잡는다.

### Step 2: 레퍼런스 검토 — 이미 있나 + 잘되는 건 어떻게 하나
**타입**: rag + api_mcp + review

후보를 내기 전에 두 가지를 본다 — (가) 비슷한 게 이미 있나(재활용·차별점), (나) 이 도메인에서 잘되는 건 어떻게 하나(베스트프랙티스). 바닥부터 만드는 낭비를 막고, 검증된 패턴을 빌려오기 위해서다. 검토 산출물은 모두 `resource/`에 저장한다.

**1단계 — 로컬 스캔 (빠름, 항상)**
- Glob으로 로컬 스킬 `.claude/skills/*/SKILL.md`와 플러그인 스킬 `~/.claude/plugins/cache/*/*/*/skills/*/SKILL.md`를 찾고, frontmatter(`name`, `description`)만 빠르게 읽는다. 본문 전체는 읽지 않는다(컨텍스트 절약). 플러그인 경로는 마켓·플러그인·버전 폴더가 3겹 끼므로 `*/*/*` 깊이를 꼭 지킨다 — 얕게 잡으면 0건이 나온다.
- Step 1에서 좁힌 의도와 역할이 겹치는 기존 스킬을 추린다. 단, ideation-funnel 자신과 코스 가이드 스킬(`part0N-*`)은 노이즈이므로 제외한다.

**2단계 — 외부 레퍼런스 딥리서치 (깊음, 기본 ON·건너뛰기 가능)**

검색 스니펫만 긁고 끝내면 "그럴듯한데 근거 없는" 레퍼런스가 된다. 그래서 외부 레퍼런스는 **deep-research 방법론**으로 검토한다 (상세: `deep-research` 스킬 / `references/reference-scan-guide.md`):
- 유망 URL 2~4개를 골라 **본문을 직접 fetch**해 읽는다(스니펫에서 멈추지 않음).
- 서로 다른 출처를 **교차검증**하고, 주장·출처마다 **신뢰도 등급(A~E)**을 단다 (A 공식/1차 ~ E 일화/SNS).
- 무엇을 리서치할지는 의도에서 정한다 — 보통 "이 도메인에서 잘되는 사례·구조·템플릿", "흔한 실패", "필요한 도구".
- 산출물 2종을 `resource/`에 저장: 보고서 `resource/{주제}-research-{날짜}.md` + 리서치 노트(과정) `resource/{주제}-research-note-{날짜}.md`.
- 비용 주의 — fetch는 한 바퀴 2~4개. 가벼운 아이디어면 사용자에게 "깊은 리서치 건너뛸까요?"로 생략 가능.

**3단계 — 정리 (항상)**
- 두 결과를 한 묶음으로 보여준다: "이미 비슷한 게 있음 → 재활용/차별점" + "잘되는 패턴 → 이렇게 빌려옴(등급 표시)".
- 정리본을 `resource/reference-scan-{날짜}.md`로 저장하고, 요약은 `ideation-draft.md`에도 이어 붙인다.

`references/reference-scan-guide.md`에 스캔 범위·딥리서치 적용법·유사도 판단 기준이 있다.

### Step 3: 구체화 — 후보 3개
**타입**: prompt

레퍼런스 스캔 결과를 반영해 "만들 거리" 후보를 **정확히 3개**, 각 **한 줄 설명 + 예상 결과물 + 난이도 태그(간단/보통/복잡)**로 제안한다. 후보가 많으면 결정 장애를 부르므로 3개로 제한한다. 기존 스킬과 겹치는 후보에는 "○○ 재활용" 또는 "○○와 이렇게 다름"을 한 줄 덧붙인다.

### Step 4: 선택 — 방향 확정
**타입**: review

AskUserQuestion으로 후보 중 하나를 고르게 한다(또는 마음에 드는 요소 조합). 잘못된 방향으로 달리는 것을 막는 체크포인트다. 선택을 한 문장으로 되짚고 "이렇게 이해했어요" 확인을 받는다. 사용자가 "아니야"라고 하면 어느 단계부터 틀렸는지 지점을 골라 부분 재실행한다 — 처음부터 전체를 다시 돌리지 않는다.

### Step 4.5: 전문가 팀 검토 (기본 ON·건너뛰기 가능)
**타입**: prompt (병렬 서브에이전트) + review

명세를 굳히기 전에, 고른 방향을 **여러 전문가 관점**으로 한 번 검증하고 개선 방향을 받는다. 혼자 명세를 쓰면 본인 한계가 그대로 박히기 때문이다. kkirikkiri의 *"동적 전문가 팀 + Critic 필수 + 근거 기반"* 알맹이만 빌리되, 영속 팀·공유메모리 같은 무거운 배관은 쓰지 않는다 — 일회성 가벼운 검토다.

1. **전문가 동적 구성** — 고정 페르소나를 쓰지 않는다. 아이디어 도메인에 맞춰 3~4명을 그때그때 정한다(archetype 매칭: Researcher/Analyst/Builder/Writer/Designer/Critic/Leader). **Critic 1명은 항상 포함** — "잘됐다"가 아니라 *약점·실패 시나리오를 찾아라(refute)* 프롬프트로 띄운다(거수기 방지). 매칭 기준은 `references/expert-review-guide.md`.
2. **근거 주입** — 각 전문가 프롬프트에 Step 2 레퍼런스 검토(`resource/`)와 좁힌 의도를 함께 넘긴다. 눈감고 평하지 않게.
3. **병렬 스폰** — Agent 도구로 한 메시지에서 동시에 띄운다(`subagent_type: general-purpose`, `model: sonnet`). 순차 스폰은 앞 의견에 오염되므로 금지.
4. **종합 + 개선 채택** — 의견을 모아 합의·충돌·개선안을 한 묶음으로 보여주고, AskUserQuestion으로 "어떤 개선을 명세에 반영할지" 고르게 한다. 채택 결과와 전문가 검토 요약은 `resource/expert-review-{날짜}.md`에 저장하고, 요약은 draft에도 이어 붙인다.

비용 주의 — 전문가 3~4명이면 서브에이전트 그만큼 돈다. 가벼운 아이디어면 사용자에게 "전문가 검토 건너뛸까요?"로 생략 가능. 채택한 개선안은 다음 Step 5 명세에 반영된다.

### Step 5: 추가 인터뷰 + 명세 생성·다듬기
**타입**: prompt + review

고른 후보를 명세로 굳히기 전에, 빠진 디테일을 AskUserQuestion으로 채운다. 후보 한 개를 골랐어도 결과물 형태·핵심 기능 우선순위·제약 같은 세부는 아직 비어 있기 때문이다.

1. **추가 인터뷰** — AskUserQuestion으로 디테일을 묻는다. 최대 2개, 옵션마다 description 필수, 추천 옵션은 첫 번째 + "(추천)". 예: 결과물 형태(텍스트/파일/세트), 꼭 넣을 핵심 기능, 손대면 안 되는 제약. 후보 분석에서 이미 명확한 항목은 묻지 않는다.
2. **명세서 생성** — 인터뷰 답을 반영해 명세 draft를 만든다: 한 줄 정의 / 핵심 기능 3개 / 입력·출력 / 범위(MVP vs 나중) / 제약.
3. **의도대로 다듬기** — 명세를 보여주고 "이대로 좋아요 / 수정할 부분 있어요 / 기능 더 추가"를 AskUserQuestion으로 묻는다. 수정·추가 요청이 있으면 명세를 고쳐 다시 보여주고, 사용자가 만족할 때까지 이 루프를 반복한다. 분석만 하고 넘어가지 않고 사용자 손에 맞을 때까지 맞추는 것이 이 단계의 핵심이다.

확정된 명세를 `ideation-draft.md`에 이어 붙인다 — 여기까지만 해도 사용자에게 남는 산출물이 되도록.

### Step 6: 핸드오프 문서 저장
**타입**: generate

확정된 명세를, 클로드코드(또는 다른 사람)가 바로 구현에 착수할 수 있는 **상세 핸드오프 문서**로 저장한다. 한 줄 Next Step으로 끝내지 않는 이유는, 받는 쪽이 다시 캐묻지 않고 곧장 만들 수 있어야 아이데이션이 실제 결과물로 이어지기 때문이다.

핸드오프 문서에 반드시 담는다:
- **개요** — 한 줄 정의 + 왜 만드는지(해결할 불편)
- **레퍼런스 정리** — Step 2 스캔 결과(재활용할 기존 자산 / 차별점)
- **핵심 기능** — 우선순위 매긴 기능 목록(MVP 표시)
- **입력 / 출력** — 사용자가 뭘 주고 뭘 받는지, 결과물 형태
- **구현 단계** — 클로드코드가 따라갈 단계별 분해(필요한 도구·파일·API 포함)
- **첫 실행** — 바로 시작할 첫 명령어 또는 첫 프롬프트
- **수용 기준** — "이게 되면 완성"이라고 볼 체크 항목 2-3개
- **다음 행동** — 보통 "이 문서로 skillers-suda 호출해 스킬화" 또는 "바로 클로드코드에 붙여넣어 제작"

진행 내내 누적해 온 `ideation-draft.md`를 바탕으로, 최종 핸드오프 문서를 **Step 1에서 만든 이번 실행 폴더** 안에 `handoff-{한줄제목}.md`로 확정 저장한다(draft와 같은 폴더). 아이데이션 전 과정(원문 → 좁힌 의도 → 레퍼런스 스캔 → 고른 후보 → 명세 → 핸드오프)이 한 문서에 담겨, 그대로 넘기면 구현이 시작되는 상태여야 한다. draft는 같은 폴더에 남겨 둔다 — 실행 단위로 폴더가 분리돼 있어 섞이지 않으므로 굳이 지우지 않는다.

`references/handoff-template.md`에 핸드오프 문서 형식이 있다.

## Settings (가변 요소)

| 설정 | 기본값 | 변경 방법 |
|------|--------|-----------|
| 외부 레퍼런스 딥리서치 | ON (deep-research 방법론) | 가벼운 아이디어면 "딥리서치 건너뛰기"로 로컬 스캔만 |
| 전문가 팀 검토 (Step 4.5) | ON (3~4명 + Critic) | 가벼운 아이디어면 "전문가 검토 건너뛰기" |
| 후보 개수 | 3개 | 요청 시 조정(단 5개 초과 금지 — 결정 장애 방지) |
| 저장 위치 | 워크스페이스 최상위 `ideation/` | 다른 상위 경로 지정 가능 — 단 실행마다 하위 폴더 분리는 유지 |

## 저장 구조 — 실행마다 폴더 분리

반복 실행해도 문서가 섞이지 않게, 스킬을 호출할 때마다 워크스페이스 최상위 `ideation/` 아래에 **실행 전용 폴더 한 개**를 만들고 그 안에만 저장한다.

```
ideation/
└── {YYYYMMDD-HHMM}-{슬러그}/         ← 이번 실행 (Step 1에서 생성, 타임스탬프로 고유)
    ├── ideation-draft.md              ← 진행 중 작업본
    ├── handoff-{제목}.md               ← 최종 핸드오프 문서
    └── resource/                       ← 레퍼런스 검토·리서치·전문가 검토 원자료
        ├── reference-scan-{날짜}.md             (Step 2)
        ├── {주제}-research-{날짜}.md             (Step 2 deep-research 보고서)
        ├── {주제}-research-note-{날짜}.md        (Step 2 리서치 노트 — 과정 기록)
        └── expert-review-{날짜}.md              (Step 4.5 전문가 검토 요약)
```

파일별 역할:

| 파일 | 언제 | 무엇 |
|------|------|------|
| `ideation-draft.md` | 진행 중 (Step 1·2·4.5·5에서 append) | 원문·좁힌 의도·스캔 요약·검토 요약·명세를 단계마다 이어 붙인 작업본. 중간에 멈춰도 남는다. |
| `resource/*` | Step 2·4.5 | 레퍼런스 검토·deep-research 보고서·노트·전문가 검토 원자료. draft엔 요약만, 원본은 여기. |
| `handoff-{제목}.md` | 완성 시 (Step 6) | draft를 정리·확정한 최종 핸드오프 문서. 받는 쪽이 바로 구현 착수. |

진행 중에는 draft에 쌓고, 끝나면 같은 폴더에 handoff로 확정한다. 실행마다 폴더가 갈리므로 어제 한 것과 오늘 한 것이 섞이지 않고, 폴더 이름의 타임스탬프로 시간순 정렬도 된다.
