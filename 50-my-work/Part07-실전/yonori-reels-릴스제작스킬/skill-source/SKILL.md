# yonori-reels — 여행 릴스 제작 파이프라인

> @travel_yonori 전용. 프로파일 문서 하나를 모든 단계가 참조해 릴스 몇 개를 만들어도 톤과 전략이 통일된다.

## 설정 (기본값)

| 설정 | 기본값 | 변경 방법 |
|------|--------|-----------|
| 계정 | travel_yonori | 사용자가 요청 시 |
| **나레이션** | **미포함 — 사용자 본인 목소리를 추후 직접 녹음** (조립은 영상+자막+현장음까지만, 녹음용 TTS 가이드 트랙은 별도 파일) | "TTS 넣어줘" 명시 시에만 TTS 포함 |

## 진입점 라우팅

요청을 보고 필요한 단계부터 시작한다. 어느 진입점이든 Step 0(프로파일 확보)을 먼저 통과한다.

| 요청 | 시작 지점 |
|------|----------|
| "주제 추천해줘" | Step 3 |
| "○○ 대본 써줘" (주제 확정됨) | Step 4 |
| "클립 편집해줘 / 영상 만들어줘" | Step 5 |
| "프로파일 갱신" | Step 1→2 강제 재실행 |
| "성과 확인 / 지난 릴스 어땠어" | Step 6 |

## 워크플로우

### Step 0: 프로파일 확보 (게이트)
**타입**: prompt

`data/tone-profile.md`(이 스킬 폴더 기준)를 확인한다:
- 없으면 → Step 1→2 실행 후 원래 목적지로 진행
- 있고 frontmatter의 `updated` 날짜가 30일 이내 → 그대로 사용
- 30일 초과 → "프로파일이 N일 지났어요. 갱신할까요?" 확인 후 진행 (낡은 톤으로 대본을 만드는 것을 막기 위해)

### Step 1: 계정 데이터 수집
**타입**: script

```bash
python3 scripts/fetch_profile.py --username travel_yonori --out data/profile-raw.json
```

- 성공 → Step 2로
- HTTP 401/403/429 (차단) → 사용자에게 최근 캡션 9~12개 수동 붙여넣기를 요청하는 폴백. **단, 폴백에는 조회수가 없으므로 성과 신호 분석(신호 지문·루브릭의 송신유발력 데이터 근거)이 비활성화됨을 명확히 고지**하고 톤 분석만 수행한다.

### Step 2: 톤앤매너 프로파일 생성
**타입**: prompt → generate

`data/profile-raw.json`을 분석해 `data/tone-profile.md`를 생성한다. 필수 섹션 (형식 상세: `references/script-rules.md` §5):

1. **톤 이원화**: 나레이션·자막 = 반말 / 피드 캡션 = 해요체 — 실측 캡션 예시 인용
2. **어미 분포표**: 캡션 어미 빈도 실측. 반말 나레이션 어미는 실측 샘플이 없으면 **"미확인" 라벨**을 붙이고, 첫 대본 생성 때 사용자 확인을 강제한다 (같은 어미 반복 "로봇 리듬" 방지)
3. **성과 신호 지문**: 게시물별 `조회수 / viral ratio(조회수÷팔로워) / 추정 승리신호(송신·저장·무신호)` 표. 신호 분류가 조회수 나열보다 다음 소재 예측력이 높다
4. **훅 라이브러리**: 실측 릴스 첫 문장을 승리 훅 / 사망 훅으로 분리 저장 + 금지 패턴
5. **가설 카운터**: "송신형(DM 공유 유발) 우위 가설 — 지지 n건 / 반증 n건" (Step 6이 갱신)

생성 후 사용자에게 요약을 보여주고 확인받는다.

### Step 2.5: 벤치마크 계정 구조 흡수 (주 1회 증분)
**타입**: api_mcp → generate

`data/benchmarks/INDEX.md`를 확인한다. 마지막 조사가 **7일 초과**면 주간 조사를 제안한다:

1. **증분 조사**: 한 번에 **신규 2~3계정**만 조사하거나 기존 계정의 변화(새 릴스 구조·성과)를 추적한다. 누적 목표 10개+, 이후에도 계속 확장. 전체 10개 일괄 재조사 금지 (API rate limit + 토큰). 후보 기준: ① 같은 니치(여행 정보·커플 여행·저장형 코스) ② 팔로워 1천~10만 ③ 최근 90일 활동. 대형 셀럽 계정은 구조 참고용 최대 2개까지만. 발굴은 WebSearch(+차단 시 insane-search) 우선, API 호출은 세션당 1~2회.
2. **구조 분해**: 계정마다 대표 릴스 2~3개를 보고 6필드를 기록 — `훅 첫 문장 패턴 / 컷 길이·전환 리듬 / 자막 스타일 / 나레이션 유무·톤 / 저장·송신 장치 / 캡션 구조`. viral ratio 높은 릴스 우선.
3. **날짜별 누적 저장**: 조사 결과를 `data/benchmarks/{YYYY-MM-DD}.md`에 쌓고, `INDEX.md`(계정당 1줄 + 유효 공통 구조 요약)를 갱신한다. **대본 생성 시엔 INDEX만 읽는다** — 날짜별 상세는 특정 계정을 깊게 볼 때만.
4. **녹이기**: 공통 구조를 뽑아 `tone-profile.md` 훅 라이브러리와 `references/script-rules.md`에 반영할 항목을 제안하고, 사용자 승인 후 반영한다. **베끼기가 아니라 구조 흡수** — 문장·소재를 복사하지 않고 "왜 먹히는가" 패턴만 가져온다.

### Step 3: 주제 선정
**타입**: api_mcp → review

**모드 분기** (상세 절차·쿼리 템플릿·루브릭: `references/topic-rubric.md`):

- **빠른 모드 (기본)**: `data/trend-cache-{YYYYMM}.md`(TTL 14일) + 즉석 WebSearch. 에이전트 스폰 없음.
- **딥 모드**: deep-research 스킬 재호출로 트렌드 리포트 재작성. 발동 조건 = 캐시 만료 / 사용자가 "제대로 조사해서" / 캐시에 없는 신규 테마 / 직전 릴스 2개 연속 viral ratio < 3x. deep-research 실패 시 WebSearch 직접 조사로 강등 + 사용자 고지.

**후보 채점**: 후보 3~5개를 100점 루브릭(송신 유발력 40 / 계정 적합성 25 / 차별각 20 / 제작 실현성 15)으로 채점. ≥70 통과, 55~69 수정 후 재채점, <55 폐기. 채점은 기본적으로 메인 스레드 단일 패스 — **검증 에이전트 스폰은 상위 2후보 점수차 ≤5점이거나 트렌드vs계정적합 하드 충돌일 때만** (비용 대비).

후보마다 `트렌드 소스 / 이 계정만의 각 / 예상 송신 트리거` 3필드 + 예상 훅 문장 필수 (트렌드 복붙 방지). 송신형 소재 ≥1개 권장. 최종 선택은 AskUserQuestion으로 사용자가 한다.

### Step 4: 릴스 대본 생성
**타입**: generate

작법 규칙 전체를 `references/script-rules.md`에서 읽고 따른다. 산출물 `script.md`를 결과물 폴더에 저장:

1. **타이밍표** (9컬럼): `장면 | 타임코드 | 나레이션 | 발음표기 | 음절수 | 예상발화(초) | 장면 지시문 | 자막 | 트리거 신호`
   - 예상발화 공식(실측 검증): `음절수 ÷ 6.0 + 0.25 × 쉼표수 + 0.15` — 장면 길이와 ±20% 벗어나면 경고
2. **리텐션 6장치 체크리스트** 통과 확인 (송신 훅 / 3초 오픈루프 / 패턴 인터럽트 / 저장 미끼 / 루프 클로징 / 송신 CTA)
3. **캡션(해요체) + 해시태그** — 프로파일 어미 분포 준수
4. 나레이션 총 음절이 분량 상한(30초≈140~160음절) 초과 시 경고

### Step 5: 영상 조립 (목소리 미포함이 기본)
**타입**: script → review

**기본 = 무음성 모드**: 사용자가 본인 목소리를 추후 직접 녹음하므로, 조립은 **영상 편집(컷 길이·자막 번인·현장음)까지만** 한다. TTS는 최종 영상에 넣지 않는다 — 대신 녹음 참고용 가이드 트랙(`*_guide.m4a`)을 별도 생성한다. 장면 길이는 타이밍표의 발화 공식 값(`duration`)을 쓴다.

1. **클립 확인**: 사용자에게 클립 폴더/파일을 받는다.
   - 해당 여행의 **클립 인덱스**(`50-my-work/reels/_index/{여행명}.csv`)가 있으면 인덱스 조회로 후보를 좁힌다. 없으면 구축을 제안한다 (1회 구축, 이후 재사용 — 절차: `references/clip-index-guide.md`, 기기별 시간 규칙 포함)
   - **클립 0개** → "촬영 가이드 모드": 타이밍표의 장면 지시문으로 촬영 리스트만 출력하고 종료
   - **클립 과다(장면 수의 2배+)** → 장면 지시문 기준으로 1차 자동 선별 후 사용자 확인
2. **장면-클립 매핑**: 매핑표를 보여주고 확인받는다. 해상도 1080p 미만 소스는 **⚠️저화질 라벨**을 붙이고 고화질 원본이 따로 있는지 사용자에게 확인한다 (업스케일은 최후 수단)
3. **원본 클립 사본 폴더 (필수 법칙, 2026-07-07)**: 매핑이 확정되면 사용할 클립을 결과물 폴더 안 `clips/`로 복사한다 (`50-my-work/reels/{YYYYMMDD-주제}/clips/`). **용량 절약: 전체가 아니라 사용 구간만 트림해 저장** (`ffmpeg -ss {시작} -t {장면길이+5초} -c copy` — 재인코딩 없음, 원본 파일명 유지). 동시에 `clips/sources.md`에 `사본 파일명 → 원본 전체 경로 / 원본 길이 / 트림 구간`을 기록해 원본을 항상 역추적할 수 있게 한다. **편집(매니페스트)은 반드시 이 사본을 참조한다** — 임시폴더·외장드라이브 원본 직접 참조 금지
4. **매니페스트 작성**: 타이밍표를 `manifest.json`으로 변환 (스키마: `scripts/build_reel.py` 상단 docstring). 무음성 모드는 `"narration": "none"` + 장면별 `"duration"`(발화 공식 값) + `"guide_audio": true`
5. **조립 실행**:
   ```bash
   python3 scripts/build_reel.py --manifest manifest.json --out final.mp4
   ```
   스크립트가 처리하는 것 (전문가 실측 반영): 클립을 filter_complex(scale **lanczos**+crop+setsar+fps)로 1080x1920/30fps 정규화 — **색보정·필터 절대 금지, 기하 변환만**(사용자 규칙) → 장면 길이에 맞춰 트림/연장 → 재인코딩 concat(CRF 18) → **ASS 자막 번인**(이 환경 ffmpeg은 drawtext 미지원 — ASS 단일 경로, Malgun Gothic) → libx264/yuv420p/AAC 출력. TTS 포함 모드일 때만: edge-tts 장면별 생성 → 뒤 0.9초 고정 묵음 트림 → 24kHz 모노를 44.1kHz 스테레오로 리샘플 → 현장음과 믹스
6. **결과 확인**: 출력 파일 정보(길이·해상도)와 가이드 트랙 경로를 보여주고, 사용자 육안 확인을 요청한다. 목소리 녹음이 끝나면 "내 목소리 합쳐줘"로 재호출 → 녹음 파일을 받아 최종 믹스한다. 이때 **실측 발화속도(총음절÷발화초)를 계산해 tone-profile에 기록** — 이후 대본의 발화 공식이 TTS 상수(6.0) 대신 개인 실측값을 쓴다 (`references/script-rules.md` §1)

### Step 6: 성과 회수 (측정·학습 루프)
**타입**: script → generate

다음 실행 시(또는 사용자 요청 시) fetch_profile.py를 재호출해:
1. 지난 릴스의 최신 조회수 → viral ratio 계산 → 프로파일 성과 신호 지문 표에 append
2. 추정 신호 분류 후 가설 카운터 갱신 ("송신형 우위: 지지 n / 반증 n")
3. 릴스 3건 이상 쌓이면 가설 재검증 결과를 보고하고, 루브릭 배점·쿼터 조정을 제안한다

스킬이 쓰일수록 이 계정의 데이터로 똑똑해지는 구조다.

## Scripts
- **`scripts/fetch_profile.py`** — 인스타 웹 프로필 API 수집 (로그인 불필요, 차단 시 상태코드 분기)
- **`scripts/build_reel.py`** — 정규화→조립→ASS 번인 파이프라인 (기본 무음성 + 가이드 트랙, TTS 모드 겸용)
- **`scripts/index_clips.py`** — 여행 촬영본 인덱스 생성 (기기별 시간 규칙 적용, 여행당 1회)

## 토큰 효율 규칙 (동일 퀄리티 · 소모 최소화 — 2026-07-08)
1. **압축 카드 원칙**: 각 단계 산출물은 요약 카드(표·1줄 결론)로 저장하고, 후속 단계는 카드만 읽는다. 원시 데이터(profile-raw.json, 날짜별 벤치마크 상세, listing.tsv)를 대본·조립 단계에서 다시 읽지 않는다
2. **클립 선별은 인덱스 조회**: 프레임 추출 육안 확인은 최종 매핑 후보만 — 장면당 1컷, 360px 축소본, 같은 프레임 재확인 금지
3. **벤치마크는 주간 증분**: 신규 2~3계정만. 대본 생성 시 INDEX.md 1파일만 읽기
4. **딥 리서치·에이전트는 조건부**: deep-research는 Step 3 딥 모드 발동 조건에서만, 검증 에이전트는 점수차 ≤5점 조건에서만 (기존 규칙 유지)
5. **조립은 백그라운드 + 로그 요약만 읽기**: ffmpeg 출력 전문을 컨텍스트에 넣지 않는다

## 주의 (사용자 확정 필수 법칙 — 2026-07-07~08)
1. **목소리는 넣지 않는다** — 대본 이후 영상 편집까지만. 나레이션은 사용자가 직접 녹음 (TTS는 가이드 트랙 별도 파일로만)
2. **보정 절대 금지** — 색보정·필터 없이 원본 화질 그대로, 저화질 소스는 업스케일(lanczos)만
3. **벤치마크 구조 흡수** — **주 1회 증분** 조사(신규 2~3계정), `data/benchmarks/`에 날짜별 누적, 구조를 대본 규칙에 녹인다
4. **원본 클립 사본 필수** — 사용 구간만 트림한 사본을 결과물 폴더 `clips/`에 저장(+`sources.md`로 원본 역추적)하고 그 사본으로만 편집한다
5. **토큰 효율** — 위 "토큰 효율 규칙" 5개 항상 적용 (퀄리티 저하 없는 범위)

## 기타 주의
- 여행 일정·코스 계획 요청은 이 스킬이 아니라 trip-advisor를 사용한다
- 인스타 API는 비공식 엔드포인트 — 응답 구조가 바뀌면 fetch_profile.py 수정 필요. 반복 호출을 자제한다 (세션당 1~2회)
- viral ratio는 sends/saves 실측이 아닌 프록시 — 사용자가 인스타 인사이트 캡처를 주면 추정을 실측으로 승격해 프로파일에 기록한다
