# 전문가 팀 검토 가이드 (Step 4.5)

명세 직전, 고른 방향을 여러 전문가 관점으로 검증하고 개선안을 받는 단계. kkirikkiri의 동적 팀 합성에서 알맹이만 빌렸다 — 영속 팀·공유메모리·세션 디렉토리 같은 무거운 배관은 쓰지 않는 **일회성 가벼운 검토**.

## 1. archetype 7종 — 도메인에 맞춰 3~4명 동적 선택

고정 페르소나(기획자·사용자…)를 메뉴처럼 고르지 않는다. 아이디어의 검증 방식 시그널을 보고 매칭한다.

| 검증 시그널 | archetype | 무엇을 보나 |
|------------|-----------|------------|
| "동작하나?" / 코드·시스템 산출 | **Builder** | 구현 가능성·도구 |
| "전달되나?" / 텍스트·청중 의식 | **Writer** | 카피·톤·메시지 |
| "쓸 수 있나?" / 시각·UX | **Designer** | 구성·사용성 |
| "출처 있나?" / 외부 정보 | **Researcher** | 사실성·레퍼런스 |
| "패턴 있나?" / 분류·통계 | **Analyst** | 구조·우선순위 |
| "반박 가능한가?" / 검증·감사 | **Critic** | 약점·실패·리스크 |
| 조율 (직접 실행 X) | **Leader** | 범위·합의 |

**규칙:**
- 아이디어 도메인에 맞는 전문가로 **이름을 구체화**한다. 예: 정보형 여행 릴스 → "숏폼 마케터(Analyst)", "여행 카피라이터(Writer)", "릴스 영상 기획자(Designer)", "콘텐츠 비평가(Critic)".
- **Critic 1명은 항상 포함**. refute 프롬프트로 띄운다(아래 §2).
- 한 사람에 두 archetype 강제 금지. 매칭 모호하면 Researcher 기본값.
- 3~4명 권장(많으면 종합이 흐려지고 비용↑).

## 2. 스폰 규칙

- **Agent 도구로 한 메시지에서 병렬 스폰**(`subagent_type: general-purpose`, `model: sonnet`). 순차 스폰은 앞 전문가 의견에 오염되므로 금지.
- 각 프롬프트에 **근거를 주입**: 좁힌 의도 + Step 2 레퍼런스 검토(`resource/`의 핵심 수치·등급). 눈감고 평하지 않게.
- 출력 형식 통일: "강점 / 약점 / 개선안 2~3개 / 한줄 요약".

**Critic refute 프롬프트(거수기 방지):**
> "이 방향의 **약점과 실패 시나리오를 찾아라**. 잘된 점 칭찬 말고, 이게 망한다면 왜 망하는지, 놓친 리스크(표절·과욕 범위·근거 부족·사용자 이탈)를 구체적으로 짚어라."

## 3. 종합 + 개선 채택

1. 4명 응답을 **합의 / 충돌 / 개선안**으로 묶어 보여준다(이모지 + 역할 + 한줄 요약 + 개선안).
2. 충돌이 있으면 그 지점을 선택지로 만든다.
3. AskUserQuestion으로 "어떤 개선을 명세에 반영할지" 고르게 한다(multiSelect 가능).
4. 채택 결과 + 검토 요약을 `resource/expert-review-{날짜}.md`에 저장, 요약은 draft에도 append.
5. 채택한 개선안은 다음 Step 5 명세에 반영한다.

## 4. 버린 것 (kkirikkiri 대비 — 의도적 경량화)

- 공유 메모리 파일(TEAM_PLAN/PROGRESS/FINDINGS) ❌ — 일회성이라 불필요
- 세션 디렉토리·team_name·Ralph 루프·Workflow substrate ❌
- 도메인 카드 100~150줄 합성 ❌ — 짧은 인라인 프롬프트로 충분
- 외부 CLI 모델 배정 ❌ — sonnet 병렬이면 충분

→ 남긴 것: **동적 전문가 + Critic 필수 + 근거 기반 + 병렬 독립**. 이게 다각도 검증의 본질이다.
