# Idea Shaper — 아이디어 구체화기

> 막연한 한 문장을 실행 가능한 1페이지 명세서 + AI에게 그대로 줄 수 있는 핸드오프 프롬프트 문서로 바꿔줍니다.

코딩 지식이 없어도 됩니다. "이런 거 만들고 싶어" 한 마디면 시작합니다. 이 스킬은 막연한 욕구를 클로드코드가 곧장 실행할 수 있는 형태로 다듬는 **사전 정제 단계**입니다.

## 두 산출물의 역할 (반드시 구분)

- **명세서.md = 구현을 담당하는 문서.** 아이디어가 무엇이고 무엇을 어떻게 만들지(목적·대상·기능·MVP 범위·참고 사례)를 담는다. "이걸 만든다"의 내용 자체.
- **핸드오프.md = AI에게 그 명세서를 만들도록 지시하는 프롬프트 위주 문서.** 새 대화에서 클로드코드(또는 다른 AI)에게 그대로 붙여넣으면 명세서대로 구현을 시작하게 만드는 지시문이 본체다. 설명·표가 아니라 **바로 실행되는 프롬프트**가 중심.

---

## 핵심 원칙

- **질문은 최대 3개** — 인터뷰가 길어지면 사용자가 지친다. 핵심 불확실 항목만 골라 묻는다.
- **언제 멈추고 생성할지가 관건** — 명확화가 충분하면 더 묻지 말고 명세 생성으로 넘어간다.
- **출력 템플릿은 고정** — 사용자 답변 길이가 들쭉날쭉해도 산출물 품질이 일정하도록 섹션 구조를 고정한다.
- **항상 다음 행동과 연결** — "예쁜 문서"에서 멈추지 않는다. 반드시 바로 구현에 착수할 수 있는 핸드오프 문서로 끝낸다.
- **외부 데이터는 검토 후 사용** — 웹 검색 결과를 그대로 명세에 넣지 않는다. 관련성을 거른 뒤 요약해서 쓴다.

---

## 산출 위치 규칙

모든 산출물은 **워크스페이스 최상위의 `ideation/` 폴더 아래, 실행마다 분리된 하위 폴더**에 저장한다. 반복 실행해도 문서가 섞이지 않게 하기 위해서다.

```
{워크스페이스 최상위}/ideation/
└── {YYYY-MM-DD}-{아이디어-슬러그}/        ← 이번 실행 전용 폴더
    ├── resources/                        ← 리서치·레퍼런스 검토 원본
    │   ├── research-notes.md             ← 검색·인출 요약 (deep-research식)
    │   └── sources.md                    ← 출처 목록 + 등급(S~D)
    ├── 명세서.md
    └── 핸드오프.md
```

- 슬러그는 아이디어를 짧은 kebab-case로 (예: `book-log`, `독서기록`).
- 같은 날 같은 슬러그가 이미 있으면 뒤에 `-2`, `-3`을 붙여 새 폴더를 만든다. 기존 폴더를 덮어쓰지 않는다.
- 워크스페이스 최상위는 현재 작업 디렉토리 기준. `ideation/`이 없으면 만든다.
- Step 5 직전에 이번 실행 폴더를 먼저 만들고, 이후 모든 파일은 그 안에 쓴다.

---

## 워크플로우

### Step 1: 의도 추출 (prompt)

사용자의 막연한 한 문장을 "무엇 / 왜 / 누구를 위해" 3축으로 분해한다. 컨텍스트가 극히 적어도 일단 추론하되, **첫 파싱 결과를 사용자에게 확인받는다** — 방향이 틀린 채로 진행하면 전체가 어긋나기 때문이다.

확인은 짧은 자연어로 한다: "이런 걸 만들고 싶으신 거 맞나요? — [무엇]을 [누구]를 위해 [왜]." 맞으면 다음으로, 아니면 사용자가 바로잡게 한다.

### Step 2: 명확화 인터뷰 (review)

불확실한 항목만 골라 **AskUserQuestion 도구로 최대 3개** 질문한다. 텍스트로 선택지를 나열하지 않는다 — 반드시 AskUserQuestion을 호출한다.

- 한 번에 하나씩, 옵션마다 설명을 붙인다. 추천 옵션은 첫 번째 + "(추천)".
- 물어볼 후보: 대상 사용자 / 핵심 목적 / 사용 상황 / 출력 형태 / MVP 범위 중 **가장 불확실한 것만**.
- 사용자가 "잘 모르겠어요 / 알아서 해주세요"를 고르면 → 가장 합의된 방향을 기본값으로 잡고 강제 진행한다. 무한 질문 루프를 만들지 않는다.

### Step 3: 레퍼런스 리서치 (api_mcp) — deep-research 방식 적용

단발 검색으로 끝내지 않는다. deep-research 스킬의 방법론을 가볍게 적용해 더 자세히 조사한다. 먼저 이번 실행 폴더의 `resources/`를 만들고, 조사 내용을 거기에 저장한다.

**① DATE-AWARE 다중 쿼리** — 검색어를 3~5개 만든다. 각 쿼리에 **연도·최신 신호**(예: `2026`, `latest`)를 하나 이상 넣고, **한국어와 영어를 함께** 돌린다 (한국어만으로는 결과가 빈약할 때가 많다). 관점을 나눠 만든다: 유사 제품·사례 / 사용된 도구·API / 흔한 실패·한계.
- 예: `독서 기록 앱 2026` / `book reading tracker app open source 2026` / `book metadata API free`

**② 검색 → 본문 인출** — WebSearch로 후보를 모으고, 핵심 URL은 WebFetch로 **본문까지 인출**한다 (검색 요약만 믿지 않는다). 광고·중복·로그인 페이지는 거른다.

**③ 출처 등급 간이 적용** — 공식·1차 자료(S/A: 공식 문서·제조사·주류 매체)를 개인 블로그·커뮤니티(C/D)보다 우선한다. C/D는 참고만 하고 단독 근거로 쓰지 않는다.

**④ resources/에 저장** — 조사 내용을 두 파일로 남긴다:
- `resources/research-notes.md` — 쿼리별 검색·인출 요약, 발견한 핵심 사실
- `resources/sources.md` — 인용한 URL 목록 + 등급(S~D) 표시

> 아이디어가 크거나 시장·기술 조사가 깊게 필요하면, 여기서 멈추지 말고 정식 `deep-research` 스킬을 호출해 그 산출물을 `resources/`에 함께 두는 것도 좋다. 이 단계는 그 경량 버전이다.

### Step 4: 검토 (review)

`resources/`에 모은 내용을 그대로 쓰지 않는다. 사용자 아이디어와 **관련성 높은 것만 골라** 한두 줄로 요약한다. 무관하거나 신뢰도 낮은(C/D 단독) 결과는 버린다. 이 요약이 Step 4.5 전문가 검토와 Step 5 명세서의 "참고 사례"로 들어간다.

### Step 4.5: 전문가 팀 검토 (review)

명세서를 쓰기 전에, 리서치 결과 + 아이디어를 전문가 팀에게 다각도로 검토받아 개선 방향을 뽑는다. 이 단계는 항상 실행한다.

**① 도메인 기반 전문가 동적 정의** — 고정 역할을 쓰지 않는다. 아이디어 도메인을 보고 그에 맞는 전문가 3~4명을 매번 새로 정의한다. (예: 독서 기록 앱 → 제품기획자 / UX디자이너 / 데이터설계자 / 그로스마케터. 예: 사내 업무 자동화 → 업무프로세스 분석가 / 자동화 엔지니어 / 보안검토자.)

**② 병렬 스폰** — `Agent` 도구로 정의한 전문가를 **하나의 메시지에서 동시에** 스폰한다 (`subagent_type: "general-purpose"`, `model: "sonnet"`). 순차 스폰 금지 — 앞 결과에 오염되지 않게 독립적으로 받는다. 각 전문가 프롬프트에 **아이디어 + Step 4 리서치 요약**을 함께 넣고, 다음을 요구한다:
- 본인 관점에서 이 아이디어의 강점·약점 각 1~2개
- 구체적인 개선안 1~2개 (실행 가능하게)
- 마지막 줄 "한줄 요약"

**③ 종합** — 4명의 응답을 모아 합의된 개선 방향과 의견이 갈린 지점을 자연어로 정리해 보여준다. 갈린 지점은 사용자에게 노출한다.

**④ 반영 확인** — AskUserQuestion으로 "개선안 반영 / 일부만 반영 / 원안 유지 / 다시 검토"를 받는다. 선택 결과를 다음 Step 5 명세서에 반영한다.

> kkirikkiri식 "도메인 동적 팀 + 병렬 + 종합"을 가볍게 가져온 단계다. 정식 Agent Teams(공유메모리·검증 루프)는 아이데이션엔 과하므로 쓰지 않는다.

### Step 5: 명세 생성 (generate)

먼저 위 "산출 위치 규칙"대로 이번 실행 폴더 `ideation/{YYYY-MM-DD}-{슬러그}/`를 만든다. 그 안에 `assets/spec-template.md`의 고정 구조로 **명세서.md**(구현을 담당하는 문서)를 만든다. **Step 4.5에서 사용자가 반영하기로 한 개선안을 명세서에 녹인다.**

생성 직전, 상세도를 한 번 확인할 수 있다 (가변 요소 — 아래 Settings). 기본값은 "표준"이므로 사용자가 따로 말하지 않으면 묻지 않고 표준으로 만든다.

화면에도 핵심 요약(목적 + 핵심 기능 3~5개 + MVP 범위)을 함께 보여준다.

### Step 6: 다듬기 루프 (review)

명세서를 보여준 뒤 AskUserQuestion으로 묻는다: "수정할 부분 있어 / 기능 추가하고 싶어 / 이대로 좋아". 사용자가 만족할 때까지 반복한다. 수정 요청은 명세서에 반영해 다시 보여준다.

### Step 7: 핸드오프 문서 생성 (generate)

사용자가 명세에 만족하면, 같은 실행 폴더에 **핸드오프.md**를 `assets/handoff-template.md` 구조로 만든다.

핸드오프는 **AI에게 명세서대로 만들라고 지시하는 프롬프트 위주 문서**다. 설명·표가 본체가 아니라, 새 대화에서 클로드코드(또는 다른 AI)에게 **그대로 복사·붙여넣으면 바로 구현을 시작하게 만드는 지시문**이 본체다. 따라서:

- 문서의 중심은 **하나의 완결된 실행 프롬프트** — 무엇을·어떤 스택으로·어떤 산출물을·어떤 순서로 만들지를 명세서 내용에 근거해 한 덩어리로 담는다.
- 그 프롬프트만 읽고도 AI가 첫 구현에 착수할 수 있어야 한다 (명세서를 따로 안 봐도 될 만큼 자족적으로).
- 기술 스택·파일 구조·구현 단계는 프롬프트를 보조하는 맥락으로만 간결히 덧붙인다.

마지막에 "이 핸드오프.md의 프롬프트를 새 대화에서 클로드코드에게 그대로 주면 명세서대로 만들기 시작합니다"라고 안내한다.

## 엣지 케이스 처리

검수자 분석 기반 — 입력 품질이 들쭉날쭉한 것이 최대 위험이므로 아래를 명시적으로 처리한다.

| 상황 | 대응 |
|------|------|
| 극도로 짧은 입력 ("앱 만들고 싶어") | Step 2 유도 질문으로 맥락 확보. 빈 입력으로 강제 진행 안 함 |
| 범위가 너무 넓음 ("세상 바꾸는 AI 서비스") | 3개 이상 하위 영역으로 분해 후 "어디부터?" 선택지 제시 |
| 이미 완성된 상세 아이디어 | "이미 잘 정리돼 있네요, 검증·보완 모드로 갈까요?" 안내 후 불필요한 질문 생략 |
| 비현실적 아이디어 | 실현 불가 요소를 짚되 "가장 가까운 현실적 대안" 함께 제안 |
| 중간 방향 전환 | "새로 시작 / 지금까지 내용 일부 유지" 명시적으로 물어 맥락 오염 방지 |
| 무의미한 입력 ("ㅁㄴㅇ") | "어떤 아이디어를 구체화하고 싶으세요? 한 문장이라도 좋아요" 재요청 |
| 중간 실패 | 그때까지 정리된 내용을 텍스트 블록으로 즉시 출력해 보존 |

## Assets

- **`assets/spec-template.md`** — 명세서(구현 담당 문서) 고정 구조 (목적/대상/핵심기능/MVP범위/참고사례)
- **`assets/handoff-template.md`** — 핸드오프(AI 지시 프롬프트) 구조 — 중심은 복사용 실행 프롬프트 한 덩어리

## Settings

| 설정 | 기본값 | 변경 방법 |
|------|--------|-----------|
| 출력 상세도 | 표준 | Step 5 직전 "간단 / 표준 / 상세" 중 선택 가능 (요청 시) |
| 레퍼런스 스캔 | 켬 | "검색 없이 빨리"라고 하면 Step 3~4 생략 |
