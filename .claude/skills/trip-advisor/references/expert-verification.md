# 전문가 교차검증 단계 — 4인 적대 리뷰

가이드 **초안**이 나오면 최종본으로 굳히기 전에, 서로 다른 관점의 전문가 4명이
"이 가이드의 결함을 찾아라(refute)" 자세로 교차검증한다. 딥리서치의 adversarial-verify
+ 끼리끼리의 "만든 인스턴스 ≠ 검토 인스턴스" 원칙을 따른다.

- 검증은 판단 작업이라 **코드가 아니라 서브에이전트**로 돈다 (Agent/Task 스폰).
- 검토자는 **초안을 쓴 세션과 다른 인스턴스**로 띄운다. "잘 썼나 봐줘"(rubber-stamp) 금지 — "틀렸다고 가정하고 반박하라".
- 외부 CLI(codex/agy) 있으면 검토자로 우선. 없으면 **Opus 적대 인스턴스**.

---

## 스폰 방법

가이드 초안(`draft`)과 `collect_places.py` 결과(JSON), TourAPI 보강 데이터를 4명에게 각각 넘긴다.
한 메시지에서 4개 Agent 호출을 병렬로.

```
Agent(subagent_type="general-purpose", model="opus",
      description="정확성 검증",
      prompt="<아래 1번 전문가 프롬프트> + 초안 + collect_places 결과")
... (2·3·4번도 동일하게 병렬)
```

각 전문가는 **검증 리포트(JSON)** 를 돌려준다:
```json
{
  "lens": "정확성",
  "verdict": "PASS | FIX | BLOCK",
  "issues": [
    {"severity": "BLOCK|FIX|NOTE", "place": "○○", "problem": "...", "fix": "..."}
  ]
}
```

- `BLOCK` 1개라도 있으면 → 최종본으로 굳히지 말고 **수정 후 재검증**.
- `FIX` → 반영 권장, 반영 후 최종.
- 모두 `PASS` → 그대로 최종.

---

## 전문가 1 — 정보 정확성 검증가 (Fact-Checker)

> archetype: Critic · "출처 있나? 일치하나?"

- 각 장소의 이름·주소·좌표가 `collect_places.py` 의 TourAPI 매칭값과 **실제로 일치**하는지 대조.
- `api_found=false`(🔸블로그만)인데 ⭐필수처럼 단정해 쓴 곳이 있으면 BLOCK.
- 가이드에 적힌 영업시간·요금이 **출처 없이 지어낸 값**이면 FIX (→ `tour_api.py detail`/`detailIntro2`로 재확인 지시).
- 폐업·이전 가능성(오래된 블로그 출처만 있는 곳) 표시 누락 시 NOTE.

**refute 프롬프트 핵심**: "이 가이드의 모든 사실이 틀렸다고 가정하고, 출처로 반박할 수 없는 항목을 찾아라."

## 전문가 2 — 현지 동선 가이드 (Route Realist)

> archetype: Critic/Designer · "현실적으로 돌 수 있나?"

- 하루 일정이 과하지 않은지 (관광지 4곳+ 빡빡함 → FIX).
- 좌표(`mapx`/`mapy`) 기준 **왔다 갔다 하는 동선**(같은 날 도시 양 끝 왕복)이면 FIX.
- 이동수단(차/대중교통) 대비 이동시간이 비현실적이면 FIX.
- 식사 시간대에 맛집이 배치됐는지, 마지막에 야경/숙소로 자연스럽게 끝나는지 NOTE.

**refute 프롬프트 핵심**: "실제로 이 코스대로 다녀봤더니 못 지킨다고 가정하고, 어디서 무너지는지 찾아라."

## 전문가 3 — 광고·편향 심판 (Bias Auditor)

> archetype: Critic · "광고 아닌가? 한쪽으로 쏠렸나?"

- `verdict=⚠️광고의심`(grade E) 장소가 코스 핵심에 올라왔으면 BLOCK.
- `mentions=1`(단일 출처, 특히 협찬 의심)인 곳이 ⭐필수급으로 다뤄졌으면 FIX.
- 특정 동네·브랜드에 과하게 쏠렸으면 FIX (지역 분산 권고).
- "블로그에서만 본 곳"이 "방문 전 확인 권장" 표시 없이 섞였으면 FIX.

**refute 프롬프트 핵심**: "이 가이드가 협찬·광고에 낚였다고 가정하고, 근거 약한 추천을 골라내라."

## 전문가 4 — 안전·운영 점검가 (Ops Verifier)

> archetype: Critic/Researcher · "가서 헛걸음 안 하나?"

- 휴무일·영업시간·예약 필요·계절 운영(겨울 휴장 등) 정보 누락 시 FIX (→ `detailIntro2` 조회 지시).
- 가족/아이/부모님 동행인데 접근성·편의(주차·유모차) 언급 없으면 NOTE.
- 축제가 **여행 기간과 실제로 겹치는지** 날짜 재확인 (안 겹치면 BLOCK).
- 야간·외진 곳 일정의 안전·교통 막차 관련 주의 누락 시 NOTE.

**refute 프롬프트 핵심**: "이 일정대로 갔다가 문 닫았거나 못 들어갔다고 가정하고, 헛걸음할 지점을 찾아라."

## 종합 (Lead)

4명의 리포트를 모아:
1. `BLOCK` 모음 → 반드시 수정. 수정 후 해당 전문가만 재검증. **재검증은 최대 2회**까지 — 2회 후에도 BLOCK 이 남으면(예: TourAPI에 그 데이터가 아예 없음) 그 항목을 "⚠️ 미검증 — 방문 전 직접 확인" 으로 표기하고 사용자 판단에 맡긴다 (무한 루프 방지).
2. `FIX` 모음 → 반영.
3. 최종 가이드 하단에 **검증 요약** 1줄 추가:
   `> ✅ 전문가 4관점(정확성·동선·광고편향·운영) 교차검증 완료 — 수정 N건 반영`
4. 검증 리포트를 `실습26-공공API스킬/verify-report-{지역}-{날짜}.md` 로 남긴다 (재현성).
