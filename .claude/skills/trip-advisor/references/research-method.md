# 여행 정보 리서치 방법 — deep-research 차용판

블로그·유튜브에서 여행 코스를 모을 때 따르는 방법. 딥리서치 스킬의 자료 조사 방식
(날짜 인식 검색 · 하위 주제 분해 · 본문 확인 · 교차검증 · 출처 등급 · 환각 방지 · 우회)을
**여행 도메인에 맞게** 각색한 것. SKILL.md의 "1. 블로그·유튜브 코스 수집" 단계가 이 문서를 따른다.

---

## 1. 날짜 인식 검색어 (여행 정보는 쉽게 상한다)

여행 정보는 가게 폐업·축제 종료·리뉴얼로 **금방 낡는다.** 검색어에 반드시 연도·최신성 키워드를 넣는다.

| 나쁜 예 | 좋은 예 |
|---|---|
| 부산 여행 코스 | 부산 여행 코스 2026 최신 |
| 부산 맛집 | 부산 맛집 2026 / 부산 신상 맛집 올해 |
| 흰여울문화마을 | 흰여울문화마을 2026 운영 후기 |

- 검색 템플릿: `{지역} {하위주제} {올해연도} {최신/요즘/신상}`
- 1년 넘은 글의 정보(특히 가격·영업시간)는 **API로 재확인** 후 채택.

## 2. 하위 주제로 쪼개기 (한 번에 "여행" 통검색 금지)

딥리서치의 "3~5개 하위 주제 분해"를 여행에 적용. 한 번에 던지지 말고 갈래별로 검색한다.

| 하위 주제 | 검색어 예 |
|---|---|
| 핵심 명소 | `{지역} 가볼만한곳 2026`, `{지역} 여행 코스 추천` |
| 맛집 | `{지역} 맛집 2026`, `{지역} 현지인 맛집` |
| 카페·뷰 | `{지역} 감성카페 2026`, `{지역} 오션뷰 카페` |
| 야경·액티비티 | `{지역} 야경 명소`, `{지역} 액티비티` |
| 동행 맞춤 | `{지역} 아이랑 / 부모님 / 커플 코스` (인터뷰 결과 반영) |

> **맛집·숙박은 TourAPI 먼저.** `tour_api.py food/stay` 가 사진·메뉴·영업시간·체크인을 검증된 채로 준다. 블로그 검색은 "현지인 픽·신상" 보강용으로만 — 맛집을 블로그로만 찾으면 API 미확인(🔸)이 많아 정보가 빈약해진다.

## 3. 스니펫에서 멈추지 말고 본문까지 (WebFetch)

검색 결과 제목·요약만으로 장소를 뽑지 않는다. **유망한 글 3~6개는 `WebFetch`로 본문을 읽어** 실제 추천 장소와 동선을 확보한다.

- 유튜브: 영상 설명란·고정 댓글에 코스 타임라인이 정리된 경우가 많다 → 그 텍스트를 활용.
- 본문에서 뽑을 것: **장소 이름**(주소·좌표는 무시 — API가 채움), 사람들이 짠 **동선 흐름**, 한 줄 평.

## 4. 교차검증 — 2개 이상에서 겹치면 채택 (mentions)

딥리서치의 "핵심 주장은 최소 2개 출처"를 여행에 적용. **여러 글/영상에 반복 등장하는 장소일수록 신뢰**한다.
이 "몇 개 글에 나왔나"가 곧 `collect_places.py`의 `mentions` 값.

- 3개+ 등장 → 강하게 채택 / 1개(특히 광고성) → 후보로만.
- 출처가 어디였는지 기록해 둔다 (아래 7. 리서치 노트).

## 5. 출처 신뢰도 등급 — 여행 블로그용 A~E

딥리서치의 A~E 등급을 여행 콘텐츠에 맞게 다시 정의한다.

| 등급 | 여행 정보에서 | 예 |
|---|---|---|
| **A** | 공식·검증 | 한국관광공사, 지자체 관광포털, TourAPI |
| **B** | 언론·대형 매체 | 신문 여행면, 방송 맛집 코너 |
| **C** | 경험 기반 개인 후기 | 솔직 방문 후기 블로그, 브이로그 |
| **D** | 정보성 약함 | 단순 정보 나열, 오래된 글(1년+) |
| **E** | 신뢰 낮음 | **협찬/광고 표기 글**, 출처·날짜 없음 |

→ A·B·C 우선, **E(광고성)는 mentions 가산에서 제외**하거나 점수를 깎는다.

### 광고글 위험 신호 (Red Flags)
- "협찬받았습니다 / 원고료 지원 / 광고 포함" 문구
- 단점이 하나도 없음, 같은 문장 여러 블로그에 복붙
- 날짜·작성자 없음, 방문 사진 없이 보도자료만

## 6. 환각 방지 — 모든 장소는 출처에 근거

딥리서치의 "근거 없는 주장 금지"를 그대로 적용.

- 블로그에서 **실제로 본 장소만** 후보에 올린다. 머릿속 지식으로 장소를 지어내지 않는다.
- 후보 → **반드시 `collect_places.py`로 TourAPI 검증** (좌표·주소 확보). API에 없으면 `🔸블로그만` 으로 표기하고 "방문 전 확인 권장" 명시.
- 영업시간·요금은 블로그 값을 그대로 쓰지 말고 `tour_api.py detail`/`detailIntro2`로 재확인.

## 7. 접근 실패 시 우회 (WebFetch 막힐 때)

딥리서치 Fallback 순서를 여행에 맞게 축약.

1. 모바일 User-Agent 로 `curl` 재시도
2. 그래도 막히면 검색 스니펫 + 다른 글로 대체
3. 인스타/유튜브 임베드는 본문 대신 **검색 결과 요약 + 댓글**로 보완
4. 막힌 URL 은 리서치 노트에 `실패`로 기록 (다음에 헛수고 방지)

## 8. 리서치 노트 남기기 (재현성)

딥리서치의 `sources.jsonl` 을 가볍게 차용. 가이드와 함께 **출처 기록**을 남겨, 나중에 "이 코스 어디서 봤더라" 추적 가능하게.

`실습26-공공API스킬/research-notes-{지역}-{날짜}.md` 에 표로:

```markdown
| 장소 | 등장 글 수 | 출처(대표 URL) | 등급 | API 검증 |
|---|---|---|---|---|
| 흰여울문화마을 | 4 | blog1, youtube2 | C | ✅ |
| ○○카페 | 1 | blog3(협찬) | E | ✅ |
```

→ `collect_places.py` 에 후보를 넘길 때 `sources`·`grade` 를 함께 실으면 이 노트가 자동으로 채워진다.
