# trip-advisor — 블로그·유튜브 × TourAPI 여행 가이드 스킬

**요즘 블로그·유튜브가 고른 실제 여행 코스**를 수집하고, 그 장소들을 **공공데이터포털 TourAPI(KorService2)**로 검증·정형화해서, 지역·기간에 맞는 **날짜별 여행 가이드 마크다운**을 만든다.

핵심 발상 — 블로그/유튜브는 "어디가 요즘 좋은지·실제 동선"을 알지만 정보가 지저분하고, TourAPI는 정확한 주소·좌표·운영시간을 주지만 트렌드를 모른다. **블로그가 고르고, API가 검증·보강한다.**

## 역할 분담 (핵심)

| 일 | 담당 | 이유 |
|---|---|---|
| 블로그/유튜브 검색·장소명 추출·교차검증, 동선 판단, 글쓰기 | 이 문서 (AI) | 맥락·판단이 필요 |
| 장소 후보 → TourAPI 검증·정형화·신뢰도 점수 | `scripts/collect_places.py` (코드) | 매번 같은 결과 |
| 축제·숙박·상세 등 직접 조회 | `scripts/tour_api.py` (코드) | 주소·키·응답 구조 고정 |
| 초안 가이드 적대 교차검증 (정확성·동선·광고·운영) | 전문가 4 서브에이전트 (AI, 검토 인스턴스 분리) | 만든 인스턴스가 자기 검토하면 못 잡음 |

> 모든 스크립트는 **프로젝트 루트에서** 실행한다 (예시의 `.claude/skills/...` 상대경로 기준).

### 사전 점검 — API 키 (시작 직후 1회)

`scripts` 가 프로젝트 루트 `.env` 의 `TOURAPI_SERVICE_KEY` 를 자동으로 읽는다. `.env` 가 없거나 값이 `YOUR_KEY_HERE` 면 **2단계에서 멈추므로 먼저 안내한다:**

```
TourAPI 키가 필요해요. 공공데이터포털(data.go.kr)에서:
1) "국문 관광정보 서비스" 검색 → 활용신청 (자동 승인, 반영까지 30분~2시간)
2) 마이페이지에서 "일반 인증키(Decoding)" 복사
3) 프로젝트 루트 .env 에 한 줄: TOURAPI_SERVICE_KEY=복사한키
```

키 호출이 `401 Unauthorized` 면 → 신규 키 반영 대기(최대 1~2시간)이거나 신청 서비스가 "국문 관광정보"가 아닌 경우. 잠시 후 재시도 안내.

## 진행 순서

### 0. 인터뷰 — 핵심 3~4개만 (AI)

- **이 스킬은 한국 내 여행 전용** (TourAPI 지원 지역). 지역이 해외면 즉시 "이 스킬은 국내 여행만 지원해요"로 안내하고 종료한다.
- 핵심 3~4개만 한 번에 묻고, 안 답하면 아래 기본값으로 조용히 진행한다 (질문 폭탄 금지).

| 항목 | 물어볼 것 | 무응답 기본값 |
|---|---|---|
| 지역 / 기간 | 예: 부산 2박3일 (Day = 박+1) | (필수 — 없으면 1회 되물음) |
| **여행 날짜/시기** | 언제 가요? (예: 7월 둘째 주, 9/20) → **축제·제철 음식·계절 명소** 맞춤 | **미정** (아래 처리) |
| 동행 | 혼자/커플/아이/부모님 → contentTypeId 영향 | **혼자** |
| 취향 | 맛집/자연/액티비티/핫플 | **명소+맛집** |
| 이동수단 | 차/대중교통 → 동선 반경 | **대중교통** |

**여행 날짜를 받으면** 그 날짜를 살려 맞춤 추천한다:
- `tour_api.py festival {YYYYMMDD}` 로 **여행 기간에 실제로 열리는 축제**만 골라 해당 Day 에 넣는다.
- **계절·제철 반영** — 봄(벚꽃·딸기), 여름(빙수·계곡·물놀이), 가을(단풍·축제), 겨울(눈·온천·방어). 검색어와 코스에 계절 키워드를 섞는다.
- 날짜가 **미정**이면 축제는 "날짜 정해지면 `festival` 재확인" 으로 남기고, 제철 추천은 **현재 시기 기준**으로 하되 "○월 기준" 임을 밝힌다.

기본값으로 진행할 땐 결과 상단에 "동행=혼자·이동=대중교통·9월 기준으로 짰어요" 처럼 적용 기준 한 줄을 남겨 사용자가 왜 그렇게 나왔는지 알게 한다.

### 1. 블로그·유튜브 코스 수집 (AI) — `references/research-method.md` 따름

딥리서치 방식을 여행에 맞게 각색한 `references/research-method.md` 의 8원칙을 따른다. 요약:

- **날짜 인식 검색어** — 여행 정보는 빨리 낡으니 연도·최신 키워드 필수: `"{지역} 여행 코스 2026 최신"`, `"{지역} 신상 맛집 올해"`
- **하위 주제로 쪼개 검색** — 명소 / 맛집 / 카페·뷰 / 야경·액티비티 / 동행 맞춤 (한 번에 "여행" 통검색 금지)
- **스니펫에서 멈추지 말 것** — 유망한 글 3~6개는 `WebFetch` 로 본문(유튜브는 설명란·고정댓글)까지 읽어 장소·동선 확보
- **교차검증** — 여러 글에 겹쳐 나온 장소일수록 신뢰 → 등장 수 = `mentions`
- **출처 등급(A~E)** — 공식/언론(A·B) > 경험 후기(C) > 오래됨(D) > **협찬·광고(E)**. 광고 위험 신호("협찬받았습니다", 단점 0, 날짜 없음)는 E.
- **환각 방지** — 블로그에서 실제로 본 장소만 후보로. 머릿속 지식으로 지어내지 않는다.

각 장소를 **이름 + 등장 수 + 등급 + 출처**로 후보 JSON 을 만든다 (등급·출처는 선택):
```json
[
  {"name": "흰여울문화마을", "mentions": 4, "grade": "C", "sources": [{"url": "blog1"}, {"url": "youtube2"}]},
  {"name": "감천문화마을",   "mentions": 5, "grade": "B"}
]
```

### 2. 장소 검증·정형화·점수 — `scripts/collect_places.py` (코드)

위 후보 JSON 을 넘기면, 각 장소를 TourAPI 로 조회해 좌표·주소·사진·contentid 를 붙이고, 등장 수·출처 등급을 합친 신뢰도 점수로 정렬해 돌려준다.

```bash
echo '[{"name":"흰여울문화마을","mentions":4,"grade":"C"},{"name":"감천문화마을","mentions":5,"grade":"B"}]' \
  | python3 .claude/skills/trip-advisor/scripts/collect_places.py --area 부산
```

결과의 `verdict`:
- `⭐필수` — 여러 글 등장 + API 매칭 → 코스 중심
- `○추천` — API 매칭 (등장 1~2회)
- `⚠️광고의심` — 출처 등급 E(협찬·광고) → 점수 깎임, 채택 신중
- `🔸블로그만` — API 에 없음(신상 카페 등) → **버리지 말고 "블로그 정보로만" 따로 표기**

이 결과를 그대로 **리서치 노트**(`실습26-공공API스킬/research-notes-{지역}-{날짜}.md`)에 표로 남긴다 (장소·등장 수·출처·등급·API검증). 나중에 "이 코스 어디서 봤더라" 추적용 — 재현성 확보.

### 3. 보강 조회 — `scripts/tour_api.py` 호출

지역명은 시군구까지 자동 해석된다 (예: "구미" → areaCode 35 + sigunguCode 4).

```bash
S=.claude/skills/trip-advisor/scripts/tour_api.py

# 지역 관광지 / 음식점 / 숙박 — 모두 사진(firstimage) 포함
python3 $S area "구미" --rows 15
python3 $S food "구미" --rows 15      # ★ 검증된 맛집 + 사진 (블로그 의존 줄이기)
python3 $S stay "구미" --rows 15      # ★ 숙박 여러 곳 + 사진

# 여행 시작일 이후 열리는 축제 (인터뷰에서 받은 여행 날짜 YYYYMMDD) — 기간에 겹치는 것만 채택
python3 $S festival "20260701" --rows 10

# 상세 — 타입별로 다른 정보가 나온다 (--type)
python3 $S detail "126508"                  # 공통: 개요·주소·좌표·대표사진
python3 $S intro "2766562" --type 39        # 음식점: 대표메뉴·취급메뉴·영업시간·쉬는날
python3 $S intro "143052"  --type 32        # 숙박: 체크인/아웃·객실수·편의시설
python3 $S intro "126508"  --type 12        # 관광지: 이용시간·쉬는날·요금·주차
python3 $S images "126508" --rows 5         # 추가 사진 (있을 때)
```

각 목록 결과는 `{"total": N, "items": [...]}` JSON. `items` 의 `title`·`addr1`·**`firstimage`(사진 URL)**·`contentid`·`mapx`/`mapy` 를 쓴다.

**★ 맛집·숙박은 TourAPI(`food`/`stay`)를 먼저** 쓴다 — 사진·메뉴·영업시간·체크인이 검증된 채로 나온다. 블로그는 "현지인 픽" 보강용으로만. (음식점을 블로그로만 찾으면 대부분 `🔸API미확인`으로 빠져 정보가 빈약해진다.)
**코스에 확정한 장소는 `intro` 로 운영정보를, 목록의 `firstimage` 로 사진을 반드시 챙긴다.**

### 4. 일정에 배치하기 (AI)

- `⭐필수` → `○추천` 순으로 코스의 뼈대를 잡는다. `🔸블로그만` 은 "API 미확인" 표시와 함께 후보로.
- 블로그에서 메모한 **실제 동선**(아침→점심→카페→야경)을 우선 참고하되, 좌표(`mapx`/`mapy`)로 가까운 곳끼리 같은 Day 에 묶어 보정한다.
- Day 수만큼 나누고, 하루에 관광지 2~3곳 + 식사/카페 1곳 정도.
- 기간 안에 열리는 축제가 있으면 해당 Day 에 끼워 넣고, 숙소는 1~2곳 추천으로 따로 정리한다.

### 6. 전문가 교차검증 (서브에이전트) — `references/expert-verification.md` 따름 ★ 항상 자동

초안을 최종본으로 굳히기 전에 **서로 다른 관점의 전문가 4명**을 띄워 "결함을 찾아라(refute)" 자세로 교차검증한다. (초안을 쓴 인스턴스 ≠ 검토 인스턴스 — rubber-stamp 방지)

| 전문가 | 보는 것 |
|---|---|
| 정보 정확성 검증가 | 장소·주소·운영정보가 TourAPI와 일치하나, 지어낸 값 없나 |
| 현지 동선 가이드 | 하루가 빡빡하지 않나, 왔다갔다 동선 아닌가, 이동시간 현실적인가 |
| 광고·편향 심판 | 협찬(⚠️광고의심)·단일출처가 핵심에 올랐나, 한쪽 쏠림 없나 |
| 안전·운영 점검가 | 휴무·영업시간·예약·계절 운영 누락 없나, 축제 날짜 실제로 겹치나 |

각 전문가는 `PASS / FIX / BLOCK` 판정 + 이슈 목록(JSON)을 돌려준다. 4개 Agent 호출을 **병렬로**, 검토자는 **Opus 적대 인스턴스**(codex/agy 있으면 그쪽 우선). 상세 프롬프트·판정 스키마는 `references/expert-verification.md`.

### 7. 반영 · 최종 저장 (AI)

- `BLOCK` 1건이라도 있으면 → 수정 후 **해당 전문가만 재검증**. 통과할 때까지 반복.
- `FIX` → 반영. `NOTE` → 가능하면 보강.
- 통과한 최종 가이드를 `50-my-work/Part06-스킬만들기/실습26-공공API스킬/{지역}-여행가이드-{YYYYMMDD}.md` 에 저장하고, 하단에 검증 요약 한 줄:
  `> ✅ 전문가 4관점(정확성·동선·광고편향·운영) 교차검증 완료 — 수정 N건 반영`
- 검증 리포트는 `verify-report-{지역}-{날짜}.md` 로 따로 남긴다 (재현성).

## 주의

- TourAPI는 `http://` 도 동작하지만 KorService1(구버전)은 종료 예정 → **KorService2** 사용 (스크립트에 반영됨).
- `.env` 키가 인증 안 되면(401 Unauthorized) 신규 발급 키 반영 대기(최대 1~2시간)이거나 신청 서비스가 "국문 관광정보"가 아닐 수 있다.
- 디코딩 키를 `.env` 에 넣는다 (requests 가 자동 인코딩).
- 키워드 검색에 **지역명 접두어·`areaCode` 파라미터를 넣으면 0건** (KorService2 특성) — 장소명만 검색하고 주소로 후처리 필터.
