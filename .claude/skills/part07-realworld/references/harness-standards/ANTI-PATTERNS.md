# 안티패턴 모음 — 실제로 워크스페이스를 죽이는 것들

> 운영 실측에서 일반화한 실패 패턴. 각 항목은 "증상 → 왜 생기나 → 처방" 구조다.
> 본문 문서(01~10)에 흩어진 것을 한 장으로 모았다.

## 1. 선제 과설치 (premature harness)

- **증상**: 위임 카드 N종, 공유메모리 파일들, 정교한 의무 조항이 첫날부터 깔려 있는데
  몇 주가 지나도 전부 빈 껍데기다.
- **왜**: "있으면 좋겠지"는 증거가 아니다. 라우팅 신호 없는 카드는 호출되지 않고,
  쓰는 사람 없는 구조는 채워지지 않는다.
- **처방**: 기본 0에서 시작, 실측 신호(실제 위임 기록, 실제 반복 작업)가 생길 때만 한 칸 승격. (→ 01 ladder)

## 2. 보일러플레이트 description

- **증상**: 위임은 활발한데(수십 회) 전문 카드는 단 한 번도 선택되지 않고
  범용 에이전트로만 흐른다.
- **왜**: 라우팅은 description만 본다. "{역할}을 담당함" 식 문구에는 사용자 발화와
  매칭될 신호가 없다.
- **처방**: 트리거 발화 2~3개 + NOT-trigger + 산출물 계약. 못 채우면 생성 보류. (→ 03)

## 3. 못 지킬 의무 (dead-letter obligation)

- **증상**: "모든 X 전후에 Y 갱신 의무" 조항이 있지만 Y는 항상 비어 있다.
- **왜**: 운영 규모에 안 맞는 의무는 "형식적 기록 vs 규칙 무시"의 양자택일을 강요하고,
  무시가 반복되면 **다른 규칙들의 신뢰도까지 무너진다**.
- **처방**: 의무 대신 자동 기록(훅). 자동이 안 되면 권장으로 격하. 지킬 수 있는 의무만 의무로.

## 4. 꺼진 측정기의 0건

- **증상**: "로그가 비어 있으니 이 기능은 사용되지 않았다"는 결론.
- **왜**: opt-in 수집기가 등록조차 안 된 상태였다면, 빈 로그는 미사용의 증거가 아니라
  측정 부재의 증거다.
- **처방**: 결론 전에 측정 도구의 가동 여부부터 확인. 1차 데이터(세션 기록)로 교차 검증. (→ 08 §4)

## 5. 컨텍스트 무한 증식

- **증상**: 생성 시 가볍던 CLAUDE.md+rules가 몇 주 만에 수십 KB — 매 세션 고정 비용.
- **왜**: 변경이력이 본문에 누적되고, 특정 작업에만 필요한 절차가 "중요하니까" rules로
  들어가고, "rules로 빼면 절감"이라는 착각이 겹친다 (rules도 전부 자동 로드된다).
- **처방**: 배치 3분류 — 불변식만 상시 / 절차는 스킬 / 상황 의존은 조건부 훅 주입.
  변경이력은 정본 분리 + 최근 3행. (→ 07)

## 6. 규칙 산포 → 드리프트

- **증상**: 같은 규칙이 CLAUDE.md·rules·커맨드 십수 곳에 복사돼 있고, 개정 시 일부만 고쳐져
  버전이 갈린다.
- **왜**: "중요하니까 여기도 적어두자"의 누적.
- **처방**: 정본 1곳 + "위반=사고급" 불변식만 사용처 1줄 인라인 + 나머지 포인터. (→ 07 §4)

## 7. 수량 하드코딩

- **증상**: 폴더 트리 주석의 "커맨드 11종"이 실제(10종)와 어긋난 채 방치.
- **왜**: 개수는 변할 수밖에 없는데 문서는 스스로 안 변한다.
- **처방**: 개수 대신 역할 서술. 점검 루틴에 "선언↔디스크 드리프트" 항목.

## 8. 문서 규율만으로 비가역 자원 보호

- **증상**: "정본 시트를 수정하기 전엔 반드시 백업" 규칙이 문서에 있었는데도
  모델이 정렬 명령을 실행해 데이터가 어긋났다.
- **왜**: 문서는 모델이 "기억하고 따라줄 때만" 작동한다. 긴 세션에서 규칙은 잊힌다.
- **처방**: 위반=사고인 규칙은 훅으로 승격. 차단+이유+절차 안내+의식적 우회 마커,
  단 PII처럼 비가역인 것은 우회 없음. (→ 05)

## 9. 검증 없는 완료 선언

- **증상**: "끝났습니다" 후에 확인해 보면 검증 항목이 미통과 상태.
- **왜**: 모델의 완료 판단은 자기 평가다. 자기 평가는 낙관적이다.
- **처방**: "완료로 보지 않는 조건"을 결정론 스크립트로 작성하고 Stop 게이트로 강제.
  (루프 가드 필수 — → 05 §2-③)

## 10. 사고 후 즉흥 복구

- **증상**: 오작동이 난 뒤에야 "어떻게 되돌리지"를 고민하고, 복구하다가
  범위 밖 파일까지 건드려 2차 사고.
- **왜**: 복구 절차가 사전에 없으면 사고 순간의 판단은 조급하고 범위는 번진다.
- **처방**: recovery 계약을 실행 전에 작성 — 실패 루프 / 재시도 한계 / scope 잠금 /
  되돌리기 규칙. 복구 지점(세션당 1커밋)이 없으면 되돌리기 자체가 불가능하다. (→ 05)

## 11. PII의 첫 커밋

- **증상**: 고객 명단이 git 이력에 들어갔고, 지우려니 이력 재작성(filter-repo + force-push)이
  필요하다.
- **왜**: .gitignore보다 git init/add가 먼저였다.
- **처방**: ① gitignore 3중 방어 → ② init → ③ check-ignore 실측 → ④ 첫 add 순서 엄수.
  커밋 PII 게이트는 우회 없음. (→ 06)

## 12. 검증기의 양치기 소년화

- **증상**: 검증 경고가 너무 많거나 오탐이 섞여서 아무도 경고를 안 읽는다.
- **왜**: 광범위 패턴(예: 모든 "key" 문자열을 시크릿으로 의심)과 숙성 없는 신규 검사.
- **처방**: 정밀 패턴 + placeholder 제외, 신규 검사는 warning-only로 숙성 후 승격. (→ 09 §3)
