# 01. 하네스 개념 — 7-layer 모델과 strength ladder

> 하네스(harness) = AI가 의도대로 동작하게 하고, 오작동하면 복귀하게 하는 통제 장치의 총합.
> 비유: 말의 마구(馬具) — 힘은 말이 내고, 방향과 안전은 마구가 잡는다.

## 1. 세 가지 하네스

| 종류 | 정의 | 목표 |
|------|------|------|
| **기본 하네스** | 정적 통제 패키지 — instruction surface(CLAUDE.md), 커맨드/스킬, 허용/금지, 완료 기준, 검증 체크리스트 | "다시 설명하지 않아도 같은 방식으로 움직이게" |
| **런타임 하네스** | 동적 통제 계층 — 훅(pre/post-tool), stop gate, 도구 allowlist, state/trace | "문서의 규칙이 실행 중에도 지켜지게" |
| **메타하네스** | 하네스를 생성·검증·개선하는 상위 하네스 | "목적별 하네스를 찍어내고 진화시키게" |

**2대 원칙**
1. **One Workspace, One Agent** — 워크스페이스 하나 = 특화 에이전트 하나. 목적이 섞이면 쪼갠다.
2. **Recovery-First** — 모든 설계는 두 질문으로 수렴한다: "오작동을 어떻게 막나? 났을 때 어떻게 돌아오나?"

## 2. 7-layer 모델

아래에서 위로 갈수록 "구조"에서 "통제"로. **L0~L2가 뼈대, L3~L6이 하네스.**

```
L6  운영 루프      변경이력 · 정기 검증 · 세션 마무리 커밋 · 사고→룰 역반영      ← 진화
L5  기계 강제      hooks (예방/검출/복귀) · permissions                         ← 강제
L4  위임           서브에이전트 카드 · 공유 상태(orchestration)                  ← 분업
L3  라우팅         커맨드/스킬 description (트리거 발화 + 경계)                  ← 분기
──────────────────────────────────────────────────────── 하네스 ↑ / 뼈대 ↓
L2  지침 표면      CLAUDE.md · rules (매 세션 로드 = 컨텍스트 예산의 대상)
L1  구조           폴더 넘버링(작업물 생애주기) · 메타 기록 폴더(_meta/)
L0  정체성         One Workspace One Agent · 허용/금지 · 인접 워크스페이스 경계
```

### 레이어별 설계 질문

- **L0**: 이 에이전트는 무엇이고, 무엇을 절대 안 하는가?
- **L1**: 산출물은 어디서 태어나서 어디로 졸업하는가?
- **L2**: 이 문장은 매 세션 필요한 불변식인가? (아니면 L2에 두면 안 된다 → 07 참조)
- **L3**: 사용자가 뭐라고 말하면 발동해야 하고, 발동하면 안 되는 경우는? (→ 03)
- **L4**: "이 역할을 언제 부르는가?"에 트리거 발화로 답할 수 있는가? (못 하면 만들지 마라)
- **L5**: 이 규칙이 위반되면 사고인가? (Yes → 문서가 아니라 훅으로 → 05)
- **L6**: 오늘의 실수가 내일의 표준이 되는 경로가 있는가?

## 3. 쌓는 순서 — 대원칙

> **뼈대(L0~L2)는 처음부터 완성형으로, 하네스(L3~L6)는 현재의 증거만큼만.**

- 뼈대가 부족하면 나중에 비싸다 (정체성·폴더 구조 변경은 이사 수준의 비용).
- 하네스가 과하면 지금 당장 썩는다 — 안 쓰는 위임 카드, 빈 공유메모리, 사문화된 의무 조항은
  컨텍스트 비용과 "규칙은 무시해도 된다"는 학습만 남긴다.

## 4. Strength ladder — 하네스를 단계적으로 올리기

```
basic ──────→ ready(light) ──────→ ready(full) ──────→ enforced
커맨드+구조만   위임 카드 1~2종        다(多) slot 분업      차단 훅 + 게이트
               (살 있는 것만)          + 공유 상태/검증 루프
```

### 승격 신호 (언제 한 칸 올리나)

| 전이 | 신호 |
|------|------|
| basic → light | 위임할 역할의 도메인 살(정체성·방법·트리거 발화)을 채울 수 있을 때만 |
| light → full | 위임이 **실제로 돌고 있다는 기록** — 계획이 아니라 위임 로그의 실측 |
| 문서 규율 → 훅 | 사고가 났거나, 사고 1회가 비가역인 자원이 식별됐을 때. 단 **PII/시크릿은 사고를 기다리지 않고 처음부터** |
| **강등** | 카드/스캐폴드가 일정 기간 사용 0회 → 축소·아카이브. 하네스는 쌓기만 하는 게 아니라 실측으로 가지치기한다 |

### 흔한 실패 패턴 (실측에서 일반화)

- **선제 과설치**: "있으면 좋겠지"로 위임 카드 N종을 미리 깔아둠 → 라우팅 신호(description) 없이는
  단 한 번도 호출되지 않는 dead code가 된다. 같은 워크스페이스에서 트리거 발화를 가진 커맨드는
  잘 돌고 generic 카드는 죽는다 — **차이는 개수가 아니라 description 품질**.
- **못 지킬 의무**: "모든 위임 전후 공유 파일 갱신 의무" 같은 조항은 1인 운영에서 지켜지지 않고,
  지켜지지 않는 규칙은 다른 규칙의 신뢰도까지 갉아먹는다. 의무는 지킬 수 있는 만큼만.
- **꺼진 측정기**: opt-in 로그가 등록되지 않은 채 "로그가 비었으니 미사용"이라고 결론내리는 것 —
  측정 도구가 꺼져 있으면 0건은 아무것도 증명하지 못한다.

## 5. 메타하네스의 닫힌 루프

워크스페이스를 "찍어내는" 쪽을 만들 거라면, 생성보다 진화 루프가 본체다:

```
생성 → 운영 → 실측 감사 → (피드백 자체를 검증) → 표준 반영 → 회귀 테스트 → 변경이력 기록
```

- **피드백 문서도 검증 대상이다** — 운영자의 보고("한 번도 안 쓰임")를 세션 기록 같은
  1차 데이터로 재검증한 뒤 표준에 반영하라. 보고는 종종 과장되거나 측정이 잘못돼 있다.
- **표준 변경엔 반드시 회귀가 따른다** — 기존 산출물을 재생성/재검증해 등급 유지 확인.
- **변경 이력이 없으면 진화가 아니라 드리프트다.**
