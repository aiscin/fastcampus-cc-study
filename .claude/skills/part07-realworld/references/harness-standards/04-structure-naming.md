# 04. 구조와 네이밍 — 폴더가 생애주기를 말하게 하라

> 폴더 구조는 장식이 아니라 **작업물의 생애주기 선언**이다. 에이전트는 구조를 보고
> "어디서 만들고 어디에 완성본을 두는지"를 학습한다.

## 1. 10단위 폴더 넘버링

```
00-system/          시스템 설정·운영 문서
10-planning/        기획·계획
20-knowledge-base/  지식 베이스 (참조 자료)
30-templates/       템플릿
40-drafts/          작업 중 (여기서 태어나서)
50-output/          완성본 (여기로 졸업한다)
60~80-*/            도메인별 커스텀 (src, tests, data 등)
90-archive/         아카이브 (은퇴)
```

원칙:
- **두 자리 + 하이픈 + 소문자 영어**: `40-drafts`. 정렬이 곧 워크플로우 순서가 된다.
- **40 → 50 흐름이 핵심**: "작업 중"과 "완성"의 물리적 분리가 품질 게이트 역할을 한다.
  완성 기준을 통과한 것만 50으로 옮긴다는 규칙 하나가 산출물 신뢰도를 만든다.
- **90은 삭제 대신 은퇴**: 폐기하는 실험·구버전은 지우지 말고 90으로. 반면교사 자료가 된다.
- **도메인 예외는 선언하고 쓴다**: 개발형 워크스페이스의 `src/`, `tests/` 같은 비넘버링 폴더는
  허용하되, 구조 정의 문서에 명시한다 (암묵 예외 금지).
- **메타 기록은 `_meta/`**: 변경이력 정본, 일탈 기록, recovery 계약 등 "워크스페이스 자신에 대한
  기록"은 산출물과 분리해 `_meta/`에 둔다.

## 2. 파일 네이밍

| 유형 | 규칙 | 예 (가상 — 베이커리) |
|------|------|------|
| 시스템 파일 | UPPERCASE | `README.md`, `CHANGELOG.md` |
| 지식 베이스 | 숫자 접두사 | `01-ingredients.md`, `02-suppliers.md` |
| 템플릿 | `TEMPLATE_` 접두사 | `TEMPLATE_order.md` |
| 작업 단위 | 의미 단위 케밥 | `weekly-settle-plan.md` |
| 분석/결과물 | 타임스탬프 접미사 | `stock-audit-20260611.md` |

- 구분자는 하이픈(케밥). 공백·한글 파일명은 셸 스크립트 호환성을 깨므로 지양
  (본문은 한국어, 파일명은 영어가 안전).
- 타임스탬프는 `YYYYMMDD` — 정렬 가능해야 한다.

## 3. 빈 폴더와 추적

- 의미 있는 빈 표준 폴더에는 `.gitkeep`을 둬서 구조가 git에 보존되게 한다.
- 단, "오래 비어 있는 폴더"는 그 자체로 신호다 — 설계와 실사용이 어긋났다는 뜻이니
  정기 점검에서 축소 후보로 본다 (01의 강등 원칙).
