# 05. 훅과 Recovery-First — 모델의 협조 없이 지켜지는 규칙

> 훅(hook) = 특정 이벤트에 끼워 실행되는 **결정론적 코드**.
> 문서 규율은 모델이 "읽고 따라주길" 기대하는 것이고, 훅은 잊든 무시하든 **기계가 강제**하는 것.
> "이 규칙이 위반되면 사고인가?" — Yes면 문서가 아니라 훅이다.

## 1. 이벤트를 알아야 활용한다 (Claude Code 기준, 공식 문서 2026-06 fetch)

훅 이벤트는 흔히 아는 몇 개가 아니라 **30종**이다. 전체 목록과 stdin 스키마는 공식 문서
(code.claude.com/docs/en/hooks)가 정본이고, 여기서는 하네스 설계에 중요한 것만 추린다.

### exit code 의미 (공통)

| exit | 의미 |
|------|------|
| 0 | 성공 — stdout의 JSON이 적용됨 (additionalContext 등) |
| 2 | 블로킹 — stderr가 모델에게 전달. **차단 여부는 이벤트별로 다름** |
| 기타 | 논블로킹 에러 |

### 하네스 관점 핵심 이벤트

| 이벤트 | 발화 시점 | exit 2 | 하네스 역할 |
|--------|----------|--------|------------|
| `PreToolUse` | 도구 호출 직전 | **차단** | 예방 — 비가역 명령 차단/인자 교정 |
| `ConfigChange` | 세션 중 설정 변경 | **차단** (변경 무효화) | 예방 — **방어선 자체 보호** |
| `UserPromptSubmit` | 프롬프트 제출 시 | **차단** | 예방 |
| `PostToolUse` | 도구 성공 후 | 비차단 (stderr→모델) | 검출 — 쓰기 직후 검증 |
| `PostToolUseFailure` | 도구 실패 후 | 비차단 | 검출→복귀 — 실패 순간 복구 절차 주입 |
| `FileChanged` | 감시 파일 외부 변경 | 비차단 | 검출 — 정본 외부 변조 감지 |
| `Stop` | 모델이 응답을 마칠 때 | **차단** (정지 방지) | **복귀 — 검증 미통과 "완료 선언" 차단** |
| `SessionStart` | 세션 시작/재개 | 비차단 (stdout=컨텍스트) | 복귀 — 미완 작업 복구 컨텍스트 주입 |
| `SessionEnd` | 세션 종료 | 비차단 | 복귀 — 핸드오프 기록 |
| `PreCompact`/`PostCompact` | 컨텍스트 압축 전/후 | 차단/비차단 | 복귀 — 컨텍스트 증발 전 스냅샷 |
| `SubagentStop` | 서브에이전트 종료 | **차단** | 검출 — 위임 결과 검증 |

훅 타입도 command만이 아니다: `prompt`(LLM 판단 훅), `agent`(읽기 도구를 쓰는 검사 훅),
`http`, `mcp_tool`. handler 단위 `if` 필드(`"if": "Bash(rm *)"`)와 `async` 실행도 지원.

### 권한 우회 모드에서의 보증 (공식 원문 요지)

> PreToolUse 훅의 deny는 **bypassPermissions 모드(승인 생략)에서도 도구를 차단한다.**
> 사용자가 권한 모드를 바꿔도 우회할 수 없는 정책이다. 역은 성립하지 않는다 —
> 훅의 allow는 settings의 deny를 뚫지 못한다. **훅은 강화만 가능, 완화 불가.**

→ 승인 생략 모드로 운영하는 워크스페이스에서는 allowlist가 무효이므로 **훅이 유일한 기계 방어선**이다.

## 2. Recovery-First — 예방/검출/복귀 3단계

하네스의 최우선 목표: **목적대로 동작하게 하고(예방·검출), 오작동해도 복귀하게 한다(복귀).**
훅 설계는 항상 세 질문을 함께 묻는다.

### ① 예방 — "비가역 행위가 무엇인가?"

- 워크스페이스가 수정하는 **외부 정본**(스프레드시트·DB·SaaS)의 위험 명령 패턴을 식별해
  PreToolUse에서 차단한다.
- 차단 설계 4요소: **차단(exit 2) + 이유 + 절차 안내 + 의식적 우회 마커**
  (예: 백업·승인 절차를 마쳤으면 `GUARD_OK=1`을 붙여 재실행). 단 **PII/시크릿 커밋은 우회 없음** —
  git 이력은 비가역이다 (→ 06).
- ConfigChange 훅으로 **세션 중 훅 설정 변조를 무효화**한다 — 방어선을 끄는 가장 쉬운 경로를
  막는 메타 방어. (세션 밖에서 사용자가 직접 수정하는 권리는 보존된다.)

### ② 검출 — "오작동을 언제 아는가?"

- 핵심 데이터 쓰기 후 PostToolUse에서 정합성 스크립트 자동 실행.
- PostToolUseFailure에서 "같은 방식으로 즉시 재시도 금지 + recovery 계약 참조"를 주입 —
  실패를 무시한 강행을 막는다.

### ③ 복귀 — "어떻게 돌아오는가?"

리커버리는 사고 후에 즉흥으로 정하지 않고 **실행 전에 계약으로 작성한다**:

```
_meta/recovery.md (복귀 계약)
├── 기본 원칙: 복구 지점 우선 / 같은 방식 재시도 금지 / 모르면 멈춤
├── 실패 루프: 멈춤 → 상태 파악 → 영향 기록 → 되돌리기·수습 → 재발 방지
├── 재시도 한계: 같은 작업 3회 실패 시 사용자 보고
├── scope 잠금: 복구 중 "고치는 김에" 금지 — 2차 사고의 주범
└── 되돌리기 규칙: 상황별 복구 수단 표 (로컬=git, 외부 정본=사전 백업, ...)
```

기계 강제 연결:
- **Stop 훅**: "완료로 보지 않는 조건"을 결정론 스크립트(예: `recovery-checks.sh`)로 옮겨 적고,
  실패하면 모델의 "끝났습니다" 선언 자체를 차단(exit 2)하고 실패 루프를 지시한다.
  **무한루프 주의**: Stop 훅 입력의 `stop_hook_active`가 true면 즉시 통과시켜야 한다
  (이미 한 번 정지를 막은 상태 — 재차단하면 영원히 안 끝난다).
- **SessionStart 훅**: 미커밋 잔여물 등 "복구가 필요한 상태"를 감지했을 때**만**
  복구 컨텍스트를 주입한다. 조건 미충족 시 출력 0 — 상시 비용이 없다 (→ 07의 조건부 주입).
- **세션 마무리 커밋**(/wrap류 루틴): 세션당 1커밋이 곧 복구 지점이다. 커밋이 없으면
  "로컬 파일 오염 → git으로 복구" 경로 자체가 존재하지 않는다.

## 3. 훅 작성 수칙

1. stdin으로 JSON이 들어온다 — 반드시 소비하고, 파싱 실패 시 exit 0 (훅 버그로 작업을 막지 마라).
2. 차단 메시지에는 **이유 + 다음 행동**을 쓴다. 모델은 stderr를 읽고 행동을 교정한다.
3. 같은 이벤트의 훅들은 병렬 실행된다 — 도구 인자를 수정(updatedInput)하는 훅은 이벤트당 1개만.
4. 훅은 등록 전 손으로 시뮬레이션하라: `echo '{...}' | bash my-hook.sh; echo $?`
   차단 케이스 / 통과 케이스 / 우회 마커 케이스를 전부.
