# 07. 컨텍스트 경제 — 매 세션 로드되는 것의 예산

> CLAUDE.md와 `.claude/rules/*.md`는 **매 세션 무조건 로드**된다. 여기 적힌 모든 글자는
> 세션마다 반복 지불하는 고정 비용이다. 운영하다 보면 반드시 비대해진다 —
> 생성 시 적정했던 워크스페이스가 단기간에 수십 KB 상시 비용으로 불어나는 패턴이 전형적이다.

## 1. 배치 3분류 — 무엇을 어디에 두나

| 분류 | 배치 | 로드 시점 | 예 |
|------|------|----------|-----|
| 매 세션 불변식 | CLAUDE.md / rules | 항상 | 정체성, 정본(SSOT) 선언, 금지, 안전 가드레일 |
| 작업별 절차 | 스킬 본문 + `references/` | 해당 작업 시에만 (lazy) | 발주서 작성 절차, 정산 셀맵 상세 |
| 상황 의존 컨텍스트 | 훅 additionalContext | 조건 충족 시에만 | 미커밋 잔여물 있을 때만 복구 경고 |

판단 질문 하나면 된다: **"이게 매 세션 로드돼야 하는가?"**
- 아니오 + 특정 작업에서만 필요 → 스킬로.
- 아니오 + 특정 상황에서만 필요 → 조건부 훅 주입으로 (→ 05).

흔한 착각: **"rules로 빼면 절감"** — rules도 전부 자동 로드된다. 파일을 나누는 것과
로드를 줄이는 것은 다른 문제다.

## 2. 예산 가이드

- CLAUDE.md + rules 합계 **~35KB 가이드라인** (모델·환경에 따라 조정하되 상한을 정해두는 것이 핵심).
- 단 **전역 가드레일(PII·정본·안전)은 예산을 이유로 절대 빼지 않는다** — 다이어트 대상은
  절차와 상세이지 안전이 아니다.
- rules 신설 전 항상 위의 판단 질문을 거친다. 신설보다 스킬 references 추가가 답인 경우가 많다.

## 3. 변경이력 — 무한 누적 차단 구조

변경이력 표를 CLAUDE.md 본문에 두면 운영할수록 자란다. 처음부터 이렇게 분리한다:

```
_meta/changelog.md      ← 정본 (전체 이력 누적)
CLAUDE.md ## 변경 이력   ← 최근 3행 + "정본: _meta/changelog.md" 포인터만
```

## 4. 정본 + 인라인 안전 불변식 + 포인터

같은 규칙을 여러 파일에 풀어 쓰면 한 곳만 고쳐져 드리프트한다 (한 규칙이 십수 곳에
중복되면 사실상 수정 불가능해진다). 표준 구조:

1. **정본 1곳** — 규칙 전문은 단일 파일에만.
2. **인라인 안전 불변식 1줄** — "위반 = 사고급"인 핵심(금액·PII 금지, 멱등성 등)만
   사용처마다 1줄 인라인. 모델이 포인터 추적을 생략할 위험에 대한 보험.
3. **포인터** — 그 외 사용처는 정본 경로만 가리킨다.

판단 기준: **"이 한 줄이 누락된 채 실행되면 사고인가?"** Yes → 인라인 1줄. No → 포인터만.

## 5. 다이어트 절차 (이미 비대해진 경우)

1. CLAUDE.md + rules 총 바이트를 측정한다 (현황 없이는 다이어트도 없다).
2. 각 섹션에 분류 3종 라벨을 붙인다 — 불변식 / 작업별 / 상황별.
3. 작업별 → 스킬 신설·이관. 상황별 → 훅 주입으로. 변경이력 → 정본 분리.
4. 중복 규칙 → 정본+포인터로 통합.
5. 재측정하고, 상한 초과 시 반복. 단 가드레일은 손대지 않는다.

## 스스로 점검

- [ ] CLAUDE.md + rules 합계가 몇 KB인지 알고 있는가?
- [ ] 최근 신설한 rule에 "매 세션 필요한가?" 질문을 던졌는가?
- [ ] 변경이력이 본문에서 자라고 있지 않은가?
- [ ] 같은 규칙이 3곳 이상에 전문 복사돼 있지 않은가?
