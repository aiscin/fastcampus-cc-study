# 09. 검증 원리 — "잘 됐는지"를 눈이 아니라 기계로

> 닫힌 루프의 절반은 검증이다. 검증이 없으면 생성은 한 번의 행운이고,
> 검증이 LLM의 감상이면 채점 자체를 신뢰할 수 없다.

## 1. 결정론 검증 원칙

- **판정은 grep/카운트/스크립트로** — "CLAUDE.md에 정체성 섹션이 있는가"는 모델에게 묻지 말고
  `grep -c '## .*정체성'`으로 센다. 같은 대상 → 항상 같은 판정이어야 검증이다.
- LLM은 검증의 **보조**(애매한 품질 판단의 의견)로만 쓰고, 등급 판정에는 넣지 않는다.
- 검증 스크립트의 흔한 함정: 카운터 증가 방식이 셸 옵션(`set -e`)과 충돌해 중간에 조용히
  종료하는 류의 버그 — 검증기가 끝까지 돌았는지부터 확인하라.

## 2. P0/P1/P2 가중치 모델

모든 검사 항목을 동급으로 두면 "사소한 것 10개 통과 + 치명 1개 실패"가 좋은 점수를 받는다.

| 등급 | 의미 | 예 | 가중치 예 |
|------|------|-----|----------|
| **P0** | 치명 — 하나라도 실패 시 낙제(F) | instruction surface 존재, .gitignore에 .env 패턴, 정체성 섹션 | 4점 |
| **P1** | 중요 — 90% 이상 통과 요구 | 폴더 넘버링 준수, 트리거 경계 명시, 변경이력 구조 | 2점 |
| **P2** | 권장 — 점수 반영, 낙제 사유 아님 | .gitkeep, 민감 파일명 부재 | 1점 |

등급 규칙 예: F = P0 1건이라도 실패 / A = P0 100% + P1 ≥ 90%.
**핵심은 숫자가 아니라 "치명과 권장을 구분한다"는 사고방식**이다.

## 3. warning-only → 승격 패턴

새 검사 항목을 바로 등급에 반영하면 기존 산출물이 무더기 낙제하고, 검사 자체의
오탐(false positive)도 걸러지지 않은 상태다. 2단계로 도입한다:

```
신설 → warning-only (등급 미반영, 별도 카운터) → 한 주기 숙성 → 오탐 0 확인 → P1/P2로 승격
```

반대로, 오탐을 내는 검사는 즉시 정밀화한다 — 검증기가 한 번 "양치기 소년"이 되면
이후 모든 경고가 무시된다.

## 4. 검증기 자신도 검증하라 (self-check)

- 검증기를 만든 워크스페이스(메타 쪽) 자신에게 검증기를 돌려 통과하는지 본다.
  자기 기준을 자기가 못 지키면 기준이 틀렸거나 검증기가 틀렸다.
- 검증 강화 후에는 반드시 **양성/음성 픽스처**로 확인: 일부러 깨뜨린 샘플이 잡히는가(true positive),
  정상 샘플이 통과하는가(false positive 0).
- 검증 대상 폴더에 과거 실험 산출물이 쌓이면 self-check가 오염된다 — 검사 제외 경로를 명시 관리.

## 5. 조건부 기대값 — 생성 옵션을 아는 검증

워크스페이스마다 생성 옵션(위임 카드 유무, 하네스 강도)이 다르면 기대값도 달라야 한다.
"카드가 없네 → 경고"는 카드 없이 생성하기로 **결정한** 워크스페이스에는 오탐이다.
→ 생성 시점의 옵션을 메타 파일(`_meta/` 안 구조 기록)에 남기고, 검증기는 그걸 읽어
기대값을 조정한다. 메타 기록이 없는 구버전은 보수적 기본값으로.

## 6. 무엇을 검사할 것인가 — 항목 설계 가이드

검사 항목은 01~07의 각 원리에서 도출한다. 영역별 최소 커버리지:

- **존재**: instruction surface, README, .gitignore (+ .env 패턴)
- **구조**: 폴더 넘버링, 선언된 예외 외 비넘버링 폴더 없음
- **정체성**: 허용/금지 섹션, 단일 에이전트 원칙 명시
- **라우팅**: 커맨드 description에 트리거 발화 존재 (보일러플레이트 마커 부재)
- **보안**: 시크릿 정밀 패턴 스캔, 민감 파일명 스캔
- **하네스 계약**: 강도 선언과 실제 산출물의 일치 (recovery 선언 시 동반 훅 존재 등)
- **이력**: 변경이력 구조 (정본 분리 + 포인터)
