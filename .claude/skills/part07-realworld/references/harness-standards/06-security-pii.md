# 06. 보안 — PII/시크릿 3중 방어와 비가역 안전 순서

> 보안 사고의 특수성: **git 이력은 비가역이다.** 한 번 커밋된 PII는 `filter-repo` + force-push
> 없이 제거할 수 없다. 그래서 PII 방어만큼은 "사고 후 학습"이 아니라 "처음부터 기계 강제"다.

## 1. .gitignore 3중 방어

PII/시크릿을 다루는 워크스페이스는 한 겹으로 막지 않는다:

```gitignore
# ── 0단: 시크릿 기본 ──
.env
.env.*
!.env.example
*.pem
*.key
secrets/
credentials/

# ── 1단: 키워드 (자기 도메인의 PII 파일명 패턴) ──
*고객명단*
*연락처*
*계약서*
*신분증*

# ── 2단: 전용 폴더 (PII는 여기에만 두기로 약속) ──
pii/
private/

# ── 3단: 문서 확장자 catch-all ──
# 산출물과 겹치지 않는 워크스페이스만 활성화. 겹치면 경로를 좁혀 적용.
*.pdf
*.xlsx
*.docx
```

- 1단 키워드는 **자기 도메인 언어로** 작성한다 (위는 예시 — 실제 파일명에 쓰일 단어로).
- 3단은 양날이다: 산출물이 PDF인 워크스페이스에서 켜면 완성본이 추적 안 된다.
  도메인에 따라 활성화 여부를 결정하고, 켜지 못하면 1·2단을 더 촘촘히.

## 2. git init 안전 순서

순서를 어기면 첫 커밋에 PII가 박힌다:

```
① .gitignore 먼저 작성 (3중 방어 포함)
② git init
③ git check-ignore -v {PII 경로 샘플}   ← 차단되는지 실측 확인
④ 그 다음에야 첫 git add
```

## 3. 커밋 PII 게이트 (기계 강제)

- 커밋 전 `git status --porcelain` 결과에서 PII 패턴(키워드·확장자)을 스캔, 발견 시 커밋 중단.
- 기계 강제: PreToolUse 훅으로 `git add/commit` 명령을 가로채 staged + 미추적 경로를 검사하고
  매칭 시 exit 2. **이 게이트에는 우회 마커를 두지 않는다** — 다른 가드는 "절차 후 의식적 우회"를
  허용하더라도, 비가역인 PII만은 예외 없음.
- 세션 마무리 커밋 루틴(/wrap류)에도 같은 게이트를 1단계로 넣는다 — 훅이 꺼진 환경의 이중화.

## 4. 시크릿 취급 수칙

- 키/토큰은 `.env`로만, 코드·문서에는 `.env.example`의 placeholder만.
- 민감 파일명 자체도 검색 대상이다: `credentials*`, `secret*`, `password*`, `*.pem`, `id_rsa*`
  패턴 파일이 작업 트리에 보이면 경고하는 검증 항목을 둔다.
- 검증 스크립트의 시크릿 grep은 **정밀 패턴**으로 (`sk-`, `api_key=` 등 형태 한정 + placeholder 제외).
  광범위 패턴은 false positive로 검증 자체를 불신하게 만든다.

## 5. 인접 워크스페이스와 PII 경계

PII를 다루는 워크스페이스와 그렇지 않은 워크스페이스가 이웃하면:
- PII가 있는 쪽 **한 곳에만** 증빙류를 모은다 (양쪽 보관 금지).
- 참조는 단방향 ID로만 (파일 복사 금지).
- 경계 규칙을 양쪽 CLAUDE.md L0(정체성)에 한 줄씩 명시한다.
