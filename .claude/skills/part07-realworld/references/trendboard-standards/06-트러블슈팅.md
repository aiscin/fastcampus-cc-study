# 06. 트러블슈팅 — 증상별 해결 / 테스트 채널

> 막혔을 때 여는 문서. **증상 → 십중팔구 원인 → 확인 → 해결** 순.
> 원칙: **추측 말고 로그로.** 에러를 그대로 붙여넣고 "고쳐줘" — 그게 바이브코딩 디버깅이다.

## 0. 30초 분류 — 어디서 막혔나

| 화면이 | 십중팔구 | 가서 볼 곳 |
|---|---|---|
| 아예 안 뜸 | 프론트 빌드·env | §1 |
| 뜨는데 **전부 빔** | 뷰 컬럼명 / 수집 안 됨 | §2 · §3 |

| 로컬은 되는데 **배포가 안 됨** | Vercel env | §6 |

> 브라우저는 **DevTools → Console / Network**. 여기 안 보고 추측하면 시간만 간다.

## 2. 화면이 전부 비었다 ★

**가장 흔한 원인: 뷰 컬럼명이 바뀌었다.**

```sql
select video_id, snapshot_at, channel_avg_views from video_details limit 1;
```

| 결과 | 뜻 |
|---|---|
| **에러** (`column does not exist`) | 뷰가 `id`·`collected_at`으로 만들어짐 → **원인 확정** |
| 빈 결과, 에러 없음 | 뷰는 정상 — 수집이 안 된 것 (§3) |
| 행 나옴 | 뷰·데이터 정상 — 프론트 문제 |

**해결**: `_spec/schema.sql`의 뷰 정의를 그대로 다시 적용.
```
video_details 뷰가 사양이랑 다른 것 같아. _spec/schema.sql의 뷰 정의대로 다시 만들어줘 — v.id는 video_id로, s.collected_at은 snapshot_at으로.
```

> kkirikkiri가 "더 깔끔한 이름"으로 바꿔버리는 일이 잦다. 조용히 실패해서 제일 찾기 어렵다.

## 3. 수집이 안 된다 (videos·snapshots 빔)

| 증상 | 원인 | 해결 |
|---|---|---|
| **403** `accessNotConfigured` | **YouTube Data API v3 미Enable** ★ | Google Cloud → Library → Enable |
| **403** `quotaExceeded` | 쿼터 소진 (`search.list`?) | §8 / 다음날 리셋(PT 자정) |
| **400** `keyInvalid` | Secret 오타 | `supabase secrets list` |
| `YOUTUBE_API_KEY is not defined` | Secret 미등록 | 대시보드 Secrets 또는 `supabase secrets set` |
| 채널은 되는데 영상 0 | `UC`→`UU` 변환 누락 | `"UU" + id.slice(2)` 확인 |
| 함수 500 | 로그 확인 | Edge Functions → Logs |

> **403이 압도적 1위**다. 키는 멀쩡한데 **API를 Enable 안 했다.** 키를 새로 만들지 말고 Enable부터.

**확인:**
```sql
select count(*) from channels;          -- 0이면 채널 등록부터
select count(*) from videos;            -- 0이면 collect-data 문제
select count(*) from video_snapshots;   -- 0이면 INSERT 문제
```

```
collect-data 로그 보고 왜 영상이 안 들어오는지 확인해줘. Edge Function 로그 그대로 보여줘.
```

## 4. Score가 전부 0 ★

**대부분 정상이다.** `avg_views`가 아직 0이라서다.

```sql
select name, avg_views from channels;
```

| `avg_views` | 뜻 |
|---|---|
| **0** | **정상** — 수집 1~2회 더 돌면 채워진다 (신규 채널) |
| > 0인데 Score 0 | 공식 문제 (아래) |

`avg_views`는 **최근 30일 발행 영상들의 최신 스냅샷 views 평균**이다.
→ 30일 내 영상이 없는 채널은 계속 0 (**정상** — 활발한 채널로 테스트).

**공식 확인** (`avg_views > 0`인데 Score가 이상하면):
```ts
// 맞음
outlierScore = avgViews > 0 ? (views / avgViews) * 10 + vsRatio : vsRatio;
// 틀림 — +vsRatio 누락
outlierScore = (views / avgViews) * 10;
```
```
Outliers의 outlierScore 계산이 (views/avg)*10+vsRatio인지 확인해줘. vsRatio는 (views/subs)*100이고, avg가 0이면 vsRatio만 쓰는 폴백이어야 해.
```

## 5. 키워드·요약이 안 뜬다

| 증상 | 원인 | 해결 |
|---|---|---|
| 배지가 아예 없음 | **Admin에 Gemini 키 미저장** ★ | Admin 로그인 → 키 저장 |
| `generate-tags` 404 | 함수 미배포 / `summarize-videos` 호출 | **`generate-tags`** 배포·호출 |
| **429** | Gemini rate limit | 15초 대기 후 재시도 / 1.5초 sleep / batch↓ |
| 일부만 태깅됨 | **정상** — 요약 없는 영상만 batch(20) | 여러 번 호출 |
| JSON 파싱 에러 | Gemini가 마크다운 감쌈 | ```` ```json ```` 제거 후 파싱 |

```sql
select count(*) from content_summaries;   -- 0이면 한 번도 성공 못 함
```

> Admin 로그인이 안 되면 기본 비번 **`trendboard2026`** (첫 로그인 후 즉시 변경).
> 비번을 잊었으면: `update admin_settings set value = crypt('새비번', gen_salt('bf')) where key='admin_password';`

## 6. 배포했는데 데이터가 안 보인다

| 증상 | 원인 | 해결 |
|---|---|---|
| 로컬 OK, Vercel 빔 | **Vercel 환경변수 미등록** ★ | `VITE_` 2개 등록 → **재배포** |
| 등록했는데 그대로 | **재배포 안 함** | 기존 빌드엔 반영 안 됨 → Redeploy |
| 빌드 실패 | TS 에러 | 로그 그대로 붙여넣기 |
| CORS | 함수 CORS 헤더 | OPTIONS 프리플라이트 응답 확인 |
| 404 (새로고침 시) | SPA 라우팅 | `vercel.json` rewrite → `/index.html` |

> **비밀 키를 Vercel에 넣어서 해결하려 하지 말 것.** 프론트는 anon만으로 충분하다 —
> 데이터가 안 보이는 건 키가 부족해서가 아니라 **`VITE_` 2개가 없거나 재배포를 안 해서**다.

## 7. 자동 갱신이 안 된다

```sql
select * from cron.job;                                                  -- 등록됐나
select * from cron.job_run_details order by start_time desc limit 5;     -- 돌았나
```

| 증상 | 원인 | 해결 |
|---|---|---|
| `cron.job` 비어 있음 | 미등록 | `cron.schedule` 재실행 |
| `schema "cron" does not exist` | 확장 미활성 | `CREATE EXTENSION pg_cron;` |
| `net` 함수 없음 | `pg_net` 미활성 | `CREATE EXTENSION pg_net;` |
| 등록됐는데 안 돎 | Free 티어 제약 | **cron-job.org 폴백** (`05` §3) |
| 돌았는데 401 | service_role 오타 | Authorization 헤더 확인 |
| 갱신은 되는데 태깅 안 됨 | **체인 끊김** | collect-data 끝에서 generate-tags 호출 확인 |
| 프로젝트 일시중지 | 7일 무요청 | Restore → cron이 돌면 재발 방지 |

> **cron은 `collect-data` 1개만** 걸어야 한다. 3개 걸면 순서가 어긋나 태깅 전에 집계한다.

## 8. 쿼터가 터졌다

**`search.list`를 쓰고 있을 가능성이 가장 크다** (100 units — 정상의 100배).

```
YouTube API 호출에서 search.list 쓰는 데 있는지 찾아줘. 있으면 playlistItems.list(UC→UU 재생목록)로 바꿔줘.
```

| 호출 | 비용 | |
|---|:-:|---|
| `playlistItems.list` | 1 | ✅ |
| `videos.list` (50개 조인) | 1 | ✅ |
| `channels.list` | 1 | ✅ |
| `search.list` | **100** | ❌ |

> 리셋은 **태평양시(PT) 자정** — 한국시간 오후 4~5시경. 정상 구현이면 하루 500~1,000 units다.
> `videos.list`를 영상마다 1회씩 부르고 있어도 터진다 → **50개 콤마조인** 확인.

## 9. kkirikkiri 결과가 불완전하다

> **정상이다.** kkirikkiri는 **코드 생성**까지고, 연결·키·배포·체인 검증은 클로드코드와 한다.
> "한 방 완성"이 아니다 — 굴리며 고치는 게 본체.

가장 자주 어긋나는 것 (`03` DO/DON'T로 재교정):

| # | 어긋남 | 고치기 |
|:-:|---|---|
| 1 | `outlierScore`에 `+vsRatio` 누락 | §4 |
| 2 | 뷰 컬럼 개명 | §2 |
| 3 | `summarize-videos` 생성 | `generate-tags`로 |
| 4 | Trends가 `trend_topics` 읽음 | `video_details` 실시간 집계 |
| 5 | 스냅샷 UPDATE | INSERT로 |
| 6 | 지표를 DB 컬럼에 저장 | 프론트 계산으로 |
| 7 | `search.list` 사용 | §8 |

```
_spec/03-기술스택과규칙.md의 DO/DON'T 기준으로 지금 코드를 점검해줘. 어긋난 것만 목록으로 보여주고, 하나씩 고치자.
```

## 10. 테스트 채널 고르기

**조건 3가지:**
1. **최근 30일 내 영상이 있어야** (없으면 `avg_views` = 0 → Score 0)
2. **업로드가 잦아야** (수집이 눈에 보임)
3. **구독자 규모가 섞여야** ★ — 100만·10만·1만이 섞이면 V/S Ratio와 Ch.Avg 배율이 **왜 필요한지**가 화면에서 드러난다

> 본인 **관심 분야**로 5개 고르는 게 제일 좋다 — 결과가 실제로 쓸모 있어야 계속 쓴다.
> 채널 URL(`youtube.com/@handle`)을 Channels 페이지에 붙여넣으면 메타데이터는 자동으로 붙는다.

**피할 채널:**
- 몇 달째 업로드 없는 채널 (`avg_views` 0)
- 영상 수천 개 채널 (초기 수집 50개 상한이라 되긴 하지만 느림)
- 라이브 위주 채널 (duration 파싱이 지저분)
