# 05. 배포와 자동화 — Vercel + 매시간 pg_cron + 보안

> **clip-08의 사양 정본** (04 STEP 8~9의 상세).
> 로컬에서 도는 대시보드를 **공개 URL + 매시간 알아서 도는 서비스**로.

---

## 0. 배포의 그림 — 층이 두 개다

```
[프론트 — Vercel]        공개 URL, anon 키만          ← 사람이 본다
        │ (anon, RLS 통과)
[백엔드 — Supabase]      Edge Function + Secret       ← 비밀 키는 전부 여기
        ↑
[pg_cron 매시간]         collect-data 1콜 → 체인 자동  ← 내가 안 켜도 돈다
```

> **비밀 키가 프론트로 내려오지 않는다**는 게 이 구조의 전부다.
> 브라우저는 anon으로 DB를 읽고, YouTube·Gemini 호출은 **백엔드에서만** 일어난다.

### 단순 재배포가 아니라 "시계열 운영"

| | 단순 재배포 | **TrendBoard 운영** |
|---|---|---|
| 무엇이 도나 | 정적 페이지 다시 올림 | **매시간 조회수 찍어 시계열 누적** |
| 효과 | 최신 내용 | **쌓일수록 Outlier가 정확해짐** |
| 자동화 위치 | 내 컴퓨터 | **Supabase pg_cron (클라우드 — 안 켜도 됨)** |

> 백엔드가 있는 원본이라 **서버 cron이 정답**이다. 내 노트북 crontab이 아니다 — 노트북은 닫히니까.

## 1. GitHub + Vercel (STEP 8)

### 올리기 전 확인 ★

```bash
cat .gitignore    # .env.local, .env, node_modules 있어야 함
git status        # .env.local이 목록에 뜨면 절대 커밋 금지
```

> 한 번 커밋되면 히스토리에 영원히 남는다. **push 전에** 확인한다.
> 이미 올렸다면 → 키를 **폐기하고 새로 발급**한다 (히스토리 삭제만으론 부족 — 이미 노출됐다).

```
이 프로젝트를 GitHub에 올리고 Vercel로 배포해줘. .env.local·node_modules가 .gitignore에 있는지 먼저 확인하고, Vercel 환경변수에 VITE_SUPABASE_URL·VITE_SUPABASE_PUBLISHABLE_KEY 넣는 법도 알려줘.
```

### Vercel 환경변수 — `VITE_` 2개만 ★

| 변수 | 값 |
|---|---|
| `VITE_SUPABASE_URL` | Supabase Project URL |
| `VITE_SUPABASE_PUBLISHABLE_KEY` | anon 키 |

> ⚠️ **여기에 비밀 키를 넣으면 공개된다.** Vercel의 프론트 환경변수는 빌드 때 번들에 박혀
> **브라우저로 그대로 나간다.** "서버 환경변수니까 안전하겠지"가 이 프로젝트에서 제일 위험한 오해다.
> YouTube·Gemini·service_role은 **Supabase 쪽**(§2)에 넣는다.

**설정 위치**: Vercel → 프로젝트 → Settings → Environment Variables → **추가 후 재배포**

| ✅ 검증 | 공개 URL 접속 / 다른 기기(휴대폰)에서 접속 / DevTools Network에 비밀 키 없음 |

> 빌드 프리셋은 **Vite** 자동 감지 (`npm run build` → `dist`).
> 데이터가 안 보이면 → 환경변수 넣고 **재배포**했는지 (기존 빌드엔 반영 안 됨).

## 2. Edge Function 배포 + Secret (STEP 9-a)

```
Edge Function들(fetch-channel, collect-data, generate-tags, extract-trends, generate-strategy, manage-settings, verify-admin)을 MCP로 배포하고, YOUTUBE_API_KEY만 Secret으로 설정하는 법 알려줘(SUPABASE_URL·SERVICE_ROLE_KEY는 자동 주입이라 제외).
```

### Secret은 `YOUTUBE_API_KEY` 하나뿐

| Secret | 설정 |
|---|---|
| `YOUTUBE_API_KEY` | **수동** ← 이 프로젝트에서 MCP로 안 되는 유일한 지점 |
| `SUPABASE_URL` | 자동 주입 |
| `SUPABASE_SERVICE_ROLE_KEY` | 자동 주입 |

**넣는 법 (둘 중 하나):**
- 대시보드: **Edge Functions → Secrets → Add new secret**
- CLI: `supabase secrets set YOUTUBE_API_KEY=AIza...`

> `gemini_api_key`는 Secret이 **아니다** — 앱 **Admin 페이지**에서 DB(`admin_settings`)에 저장한다.

> ℹ️ 위 목록이 **7개**인 건 `generate-strategy`(P2 선택)까지 포함해서다. Strategy 화면을 안 만들었으면
> **핵심 6종**만 배포하면 된다 — clip-07의 "6종"과 어긋나는 게 아니다. (`03` §3)

| ✅ 검증 | 함수 배포(핵심 6종 + 선택 시 strategy) / Secret 등록 / collect-data 수동 호출 성공 |

## 3. 매시간 pg_cron ★ (STEP 9-b)

**cron이 부르는 건 `collect-data` 하나뿐이다.** generate-tags·extract-trends는 체인으로 따라온다.

```
매시간 정각에 collect-data를 부르는 Supabase pg_cron 등록 SQL 만들어줘. 안 되면 cron-job.org로 외부 cron 거는 법도 알려줘.
```

### 등록 SQL

```sql
-- 확장 (schema.sql에 이미 있음)
CREATE EXTENSION IF NOT EXISTS pg_cron;
CREATE EXTENSION IF NOT EXISTS pg_net;   -- cron에서 HTTP 호출

-- 매시간 정각에 collect-data 호출
SELECT cron.schedule(
  'trendboard-hourly-collect',
  '0 * * * *',                                  -- 매시간 0분
  $$
  SELECT net.http_post(
    url     := 'https://<PROJECT_REF>.supabase.co/functions/v1/collect-data',
    headers := jsonb_build_object(
      'Content-Type',  'application/json',
      'Authorization', 'Bearer <SERVICE_ROLE_KEY>'
    ),
    body    := '{}'::jsonb
  );
  $$
);
```

> `<PROJECT_REF>`·`<SERVICE_ROLE_KEY>`는 본인 것으로. 이 SQL은 **DB 안에서만** 살고
> 프론트·깃에 나가지 않으므로 여기 service_role이 들어가는 건 **정상**이다.

**확인·관리:**
```sql
SELECT * FROM cron.job;                                   -- 등록 목록
SELECT * FROM cron.job_run_details ORDER BY start_time DESC LIMIT 10;   -- 실행 이력
SELECT cron.unschedule('trendboard-hourly-collect');      -- 해제
```

| ✅ 검증 | `cron.job`에 1건 / **다음 정시 후** `job_run_details` 성공 + `video_snapshots`에 새 행 |

> 검증은 **다음 정시까지 기다려야** 한다. 라이브에서 못 기다리면 `collect-data`를 수동 호출해
> "체인이 돈다"만 보여주고, cron 결과는 강사가 미리 찍어둔 실행 로그로.

### 폴백 — Free 티어에서 pg_cron이 안 될 때

`pg_cron`/`pg_net`이 막히면 **외부 cron으로 동일 효과**:

1. [cron-job.org](https://cron-job.org) 가입
2. **Create cronjob** → URL: `https://<PROJECT_REF>.supabase.co/functions/v1/collect-data`
3. Method **POST**, Header `Authorization: Bearer <SERVICE_ROLE_KEY>`
4. 주기 **매시간**

> ⚠️ 외부 cron은 service_role 키를 **3rd party에 맡기는** 것이다. 개인 프로젝트면 수용 가능한
> 트레이드오프지만, pg_cron이 되면 pg_cron이 낫다 (키가 DB 밖으로 안 나감).

### 왜 매시간인가

| 주기 | |
|---|---|
| 매시간 (`0 * * * *`) | **권장** — 급상승을 놓치지 않고 쿼터도 넉넉 |
| 6시간 (`0 */6 * * *`) | 쿼터 아끼기 |
| 10분 | ❌ 과함 — 조회수는 그렇게 안 빨리 변한다 |

> 주기는 자유지만 **체인 진입점은 언제나 `collect-data` 1콜**이다.

## 4. 보안 체크리스트 ★

- [ ] `.gitignore`에 `.env.local`·`.env`·`node_modules`
- [ ] 깃 히스토리에 키 없음 (`git log -p | grep -i "AIza\|service_role"`)
- [ ] Vercel 환경변수는 **`VITE_` 2개만**
- [ ] `YOUTUBE_API_KEY`는 **Supabase Secret**에만
- [ ] `gemini_api_key`는 **DB**에만 + anon select로 **안 보임**(RLS)
- [ ] Admin 비번 **`trendboard2026`에서 변경 완료**
- [ ] (권장) YouTube 키에 API 제한 걸기

**키가 새면**: 즉시 **폐기 후 재발급** (Google Cloud / AI Studio에서 삭제 → 새 키). 히스토리 정리는 그다음.

> `service_role`은 **RLS를 통째로 우회**한다. 이게 프론트로 새면 남이 내 DB를 전부 읽고 지울 수 있다.
> 그래서 자동 주입 + Secret + cron SQL 안에서만 살게 설계돼 있다.

## 5. 운영 — 알아둘 것

| | |
|---|---|
| **Supabase 일시중지** | Free는 **7일 무요청 시 중지** — 매시간 cron이 돌면 자동 방지 |
| **쿼터** | 채널 10개 × 24회/일 ≈ 500~1,000 units (한도 10,000) |
| **Gemini 429** | 15초 대기 후 재시도 / 호출 간 1.5초 sleep / 요약 **없는** 영상만 |
| **스냅샷 증가** | 영상 100개 × 24회/일 ≈ 2,400행/일 — Free 500MB로 몇 달 OK |
| **데이터 정리** | 오래된 스냅샷은 일 1건으로 롤업하는 cron 추가 가능 (선택) |
