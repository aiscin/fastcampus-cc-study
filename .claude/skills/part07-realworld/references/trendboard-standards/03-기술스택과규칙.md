# 03. 기술 스택과 규칙 — kkirikkiri 가드레일

> **kkirikkiri에게 코드를 맡기기 전에 못박는 문서.** 에이전트 팀은 사양이 비면 그럴듯하게 지어낸다 —
> 지어내면 안 되는 것들이 여기 있다. 구체화(STEP 2)에서 이 DO/DON'T를 통째로 넣는다.

---

## 1. 스택 (바꾸지 않는다)

| 영역 | 선택 | 왜 이것 |
|---|---|---|
| 프론트 | **Vite + React + TypeScript** | |
| UI | **Tailwind + shadcn/ui**, 다크 기본 | 지표 테이블이 눈에 덜 피곤 |

| 영상 데이터 | **YouTube Data API v3** | |
| AI | **Google Gemini** | 무료 티어 |
| 배포 | **Vercel** (+ GitHub) | |

> 대체 스택 제안 금지 — Next.js·Prisma·Firebase 등으로 갈아타면 `schema.sql`·Edge Function 사양이
> 전부 무의미해지고, 수강생마다 다른 앱이 된다.

## 2. 환경변수 — 어디에 무엇이 사는가 ★

**이 표가 이 문서에서 제일 중요하다.** 키가 잘못된 층에 가면 보안 사고다.

| 변수 | 사는 곳 | 공개? | 설정 방법 |
|---|---|:-:|---|
| `VITE_SUPABASE_URL` | 프론트 (`.env.local`, Vercel) | ✅ 공개 | MCP 자동 |
| `VITE_SUPABASE_PUBLISHABLE_KEY` | 프론트 (`.env.local`, Vercel) | ✅ 공개 | MCP 자동 (=anon) |
| `YOUTUBE_API_KEY` | **Edge Function Secret** | ❌ 비밀 | **수동** (대시보드 / `supabase secrets set`) |
| `SUPABASE_URL` | Edge Function | — | **자동 주입** |
| `SUPABASE_SERVICE_ROLE_KEY` | Edge Function | ❌ 비밀 | **자동 주입** |
| `gemini_api_key` | **DB** (`admin_settings`) | ❌ 비밀 | Admin 페이지에서 저장 |

**규칙 3줄:**
1. **`VITE_` 접두사 = 브라우저에 그대로 박힌다.** 비밀 키에 `VITE_`를 붙이는 순간 공개된 것이다.
2. **Vercel 환경변수엔 `VITE_` 2개만.** YouTube·Gemini·service_role은 절대 Vercel 금지.
3. **Gemini 키만 env가 아니라 DB.** Admin에서 코드 수정 없이 교체하려고 그렇게 설계했다.

> `SUPABASE_URL`·`SERVICE_ROLE_KEY`는 Edge Function 런타임이 **알아서 넣어준다** — 수동 설정하면 안 된다.
> 수동으로 넣는 Secret은 **`YOUTUBE_API_KEY` 하나뿐이고, 이 프로젝트에서 MCP로 안 되는 유일한 지점**이다.

`.gitignore`에 반드시: `.env.local`, `.env`, `node_modules`

## 3. Edge Function — 핵심 6종 + 선택 1종

> **6이냐 7이냐**: 핵심은 **6종**이고, `generate-strategy`는 **P2(선택)** 이다.
> clip-07에서 "6종"이라 하고 clip-08 배포 목록에 7개가 나오는 건 이 차이다 — 틀린 게 아니다.
> Strategy 화면을 안 만들면 6종으로 끝내도 앱은 완전히 돈다.

| 함수 | 트리거 | 하는 일 | |
|---|---|---|:-:|
| `fetch-channel` | 채널 등록 | `channels.list` → channels UPSERT | 핵심 |
| **`collect-data`** ★ | **cron(매시간)** / 등록 직후 | 영상·스냅샷 수집 + `avg_views` 갱신 → **체인 시작** | 핵심 |
| `generate-tags` | collect-data 체인 / Admin | Gemini 태깅 → content_summaries | 핵심 |
| `extract-trends` | collect-data 체인 (7·30) | 키워드 집계 → trend_topics | 핵심 |
| `manage-settings` | Admin | admin_settings R/W (service_role) | 핵심 |
| `verify-admin` | Admin | 로그인·비번 변경 (RPC) | 핵심 |
| `generate-strategy` | Strategy 페이지 | Gemini 전략 → strategy_results | **선택(P2)** |

> ⚠️ `summarize-videos`는 **레거시** — `generate-tags`로 대체됐다. 만들지 않는다.

**체인 (HTTP, Bearer service_role):**
```
cron → collect-data ─┬─> generate-tags
                     ├─> extract-trends {7}
                     └─> extract-trends {30}
```

> **cron이 부르는 건 `collect-data` 하나뿐이다.** 나머지는 알아서 따라온다 —
> cron을 3개 걸면 순서가 어긋나서 "요약 없는 영상"을 태깅 전에 집계하게 된다.

## 5. DO / DON'T ★

> 구체화·kkirikkiri 입력에 **그대로** 넣는다.

### DO

- ✅ `playlistItems.list`로 영상 목록 (`UC` → `UU` 재생목록)
- ✅ `videos.list`는 **50개 콤마조인** 1회 (`part=snippet,statistics,contentDetails`)
- ✅ `videos`는 `platform_video_id` **UPSERT**, `video_snapshots`는 **INSERT**
- ✅ 지표(Score·배율·V/S)는 **프론트 JS 계산**
- ✅ `video_details` 뷰 컬럼명 **그대로** (`video_id`·`snapshot_at`)
- ✅ `outlierScore = (views/avg)*10 + vsRatio`
- ✅ Gemini 키·모델·프롬프트는 **`admin_settings`에서 읽기**
- ✅ 요약 **없는** 영상만 태깅 (batch 기본 20)
- ✅ Gemini 429 → **15초 대기 후 재시도**, 호출 간 **1.5초 sleep**
- ✅ `views`·`keywords` **NULL 방어** (신규 영상은 비어 있음)
- ✅ 실패는 **토스트 + 로그**로 드러내기

### DON'T

- ❌ **`search.list` 금지** (100 units — 쿼터 즉사)
- ❌ **목업·더미 데이터 금지** — 실제 API·실제 DB만
- ❌ `service_role`·`YOUTUBE_API_KEY`·`GEMINI_API_KEY`를 **프론트·깃·Vercel에** 금지
- ❌ 비밀 키에 **`VITE_` 접두사** 금지
- ❌ `outlierScore = (views/avg)*10` (**`+vsRatio` 누락**)
- ❌ 뷰 컬럼을 `id`·`collected_at`으로 **개명** 금지
- ❌ `summarize-videos` 생성 금지 (레거시)
- ❌ Trends가 `trend_topics`를 읽게 하기 금지 (**뷰에서 실시간 집계**)
- ❌ 스냅샷 **UPDATE** 금지 (시계열이 사라짐)
- ❌ 지표를 **DB 컬럼으로** 저장 금지
- ❌ 스택 교체(Next.js·Firebase 등) 금지
- ❌ 로그인·결제·멀티플랫폼 (`01-PRD.md` §6 Out of Scope)
- ❌ 에러를 **조용히 삼키기** 금지 (`catch {}`)

## 6. YouTube 쿼터 — 하루 10,000 units

| 호출 | 비용 | |
|---|:-:|---|
| `playlistItems.list` | 1 | ✅ |
| `videos.list` (50개 조인) | 1 | ✅ |
| `channels.list` | 1 | ✅ |
| **`search.list`** | **100** | ❌ |

채널 10개 × 24회/일 ≈ **500~1,000 units** — 넉넉하다. `search.list`를 쓰면 같은 작업이 **5만 units**가 된다.

**수집 모드:**
- **초기** (영상 0개 또는 `force_full`): 최대 `max_videos_per_channel`(기본 **50**)개
- **증분** (평소): 이미 가진 영상을 만나면 그 지점까지만 (`hitExisting`)

> 증분 덕분에 매시간 돌아도 쿼터가 거의 안 든다.

## 7. 보안 3줄 요약

1. **프론트엔 anon만.** 나머지는 전부 백엔드.
2. **Admin 기본 비번 `trendboard2026` — 첫 로그인 즉시 변경.**
3. **`gemini_api_key`는 RLS로 anon 차단** — 저장 후 anon select로 안 보이는지 실측.

> 개인용 RLS라 공개 URL이면 남이 채널을 지울 수도 있다. 이건 **의도된 MVP 트레이드오프**이고,
> 다중 사용자로 갈 땐 `auth.uid()` 정책으로 다시 짜야 한다 (`02-데이터모델.md` §4).
