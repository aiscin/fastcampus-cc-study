> **구체화(STEP 2)와 검증(Phase 2)에서 사양으로 쓰는 문서.**
> 실행 가능한 원본은 `assets/schema.sql` — 이 문서는 "왜 이렇게 생겼나"를 설명한다.
> 공식·컬럼명은 실제 동작 코드 기준 (`BUILD-PROCESS.md` 분석 결과 반영).

## 0. 한눈에

```
channels ──< videos ──< video_snapshots     (매시간 지표 시계열)
                    └──1 content_summaries  (Gemini 태깅, 영상당 1개)
trend_topics       (extract-trends 집계·보관)
strategy_results   (generate-strategy 결과)
admin_settings     (키·비번·모델·프롬프트)

video_details 뷰 = videos + channels + 최신 snapshot + content_summaries
                   → 프론트 분석 4페이지가 읽는 유일한 소스
```

**기억할 3가지:**
1. **`video_details` 뷰가 0순위** — 없으면 어떤 화면도 데이터가 없다.
2. **지표는 DB가 아니라 프론트에서 계산** — 뷰는 원천값만 준다.
3. **자동화 진입점은 `collect-data` 하나** — 나머지는 체인으로 따라온다.

## 1. 테이블 7

### `channels` — 모니터링 대상 채널

| 컬럼 | 타입 | 메모 |
|---|---|---|
| `id` | UUID PK | |
| `platform_channel_id` | TEXT **UNIQUE** | `UCxxxx` — uploads 재생목록의 씨앗 |
| `name`, `thumbnail_url`, `url` | TEXT | `fetch-channel`이 채움 |
| `subscriber_count` | BIGINT | **V/S Ratio의 분모** |
| `avg_views` | BIGINT | **Outlier의 기준선** — 최근 30일 평균 |
| `category` | TEXT NOT NULL, 기본 `'AI'` | 전역 카테고리 필터 |
| `is_active` | BOOLEAN 기본 true | false면 수집 제외 |

> `avg_views`가 이 앱의 심장이다. **"평소"를 모르면 "터졌다"를 못 판단한다.**
> 신규 채널은 처음에 0 → Score가 전부 0으로 보인다(정상, 1~2회 더 수집하면 채워짐).

### `videos` — 개별 영상

| 컬럼 | 타입 | 메모 |
|---|---|---|
| `id` | UUID PK | 뷰에선 **`video_id`**로 노출 |
| `channel_id` | UUID FK → channels | ON DELETE CASCADE |
| `platform_video_id` | TEXT **UNIQUE** | UPSERT 충돌 키 |
| `title`, `description`, `thumbnail_url` | TEXT | Gemini 태깅 입력 |
| `duration` | INTEGER | **초 단위** (ISO8601 `PT#M#S` 파싱 결과) |
| `format` | TEXT | `duration <= 60 ? 'shorts' : 'long'` |
| `published_at` | TIMESTAMPTZ | 30일 평균·주간 집계 기준 |

### `video_snapshots` — 매시간 지표 시계열 ★

| 컬럼 | 타입 | 메모 |
|---|---|---|
| `video_id` | UUID FK → videos | |
| `views`, `likes`, `comments` | BIGINT | 그 시각의 **누적값** |
| `collected_at` | TIMESTAMPTZ | 뷰에선 **`snapshot_at`** |

> **이 테이블이 "원본"과 "단순판"을 가른다.** UPDATE가 아니라 **INSERT만 한다** —
> 매시간 새 행이 쌓여서 시계열이 된다. 덮어썼으면 "지난주 대비"를 영영 못 본다.
> 인덱스 `(video_id, collected_at DESC)`는 뷰의 LATERAL "최신 1건" 조회를 위한 것.

### `content_summaries` — Gemini 태깅 (영상당 1개)

| 컬럼 | 타입 | 메모 |
|---|---|---|
| `video_id` | UUID FK **UNIQUE** | 영상당 1행 (UPSERT) |
| `summary` | TEXT | 한 줄 요약 |
| `keywords` | TEXT[] | 소문자 정규화 — **트렌드·Tier의 재료** |
| `content_tags` | TEXT[] | 7종 고정 (아래) |
| `content_type`, `sentiment` | TEXT | 유형 / `positive·neutral·negative` |

`content_tags` 7종 (제목 후킹 패턴): `Numbers` · `Free` · `Question` · `Time` · `Money` · `Success` · `Change`

> `keywords[0]`가 **Tier 랭킹의 그룹 키**다. 키워드가 안 채워지면 Tier도 트렌드도 빈다.

### `trend_topics` — 키워드 집계 보관

| 컬럼 | 메모 |
|---|---|
| `keyword`, `mention_count`, `video_count` | `video_count` = 고유 video_id 수 |
| `growth_rate` | NUMERIC(10,2) — 직전 기간 대비 % |
| `period_start`, `period_end` | `UNIQUE(keyword, period_start, period_end)` |

> ⚠️ **Trends 화면은 이 테이블을 읽지 않는다.** 화면은 `video_details`에서 **클라이언트 실시간 집계**한다.
> 이 테이블은 `extract-trends`가 기록·보관하는 이력용이다. (헷갈리기 쉬운 지점 — 04 STEP 7 참고)

### `strategy_results` — AI 전략 (P2 선택)

`insights TEXT[]` · `strategies JSONB` · `title_formulas TEXT[]` · `avoid TEXT[]` · `period_days`(기본 14) · `category`(기본 `'전체'`)

### `admin_settings` — 키/값 설정

| key | 기본값 | 용도 |
|---|---|---|
| `admin_password` | `trendboard2026` (bcrypt) | **즉시 변경** |
| `gemini_api_key` | — | Admin에서 저장 |
| `max_videos_per_channel` | `50` | 초기 수집 상한 |
| `ai_model_tags` | `gemini-2.5-flash-lite` | |
| `ai_model_strategy` | `gemini-2.5-flash` | |
| `prompt_tags`·`prompt_summary`·`prompt_strategy` | — | Admin 라이브 편집 |
| `categories` | JSON 배열 | 카테고리 CRUD |

> **키·모델·프롬프트가 코드가 아니라 DB에 있다** = 재배포 없이 Admin에서 바꾼다.

## 2. `video_details` 뷰 — 프론트의 단일 소스 ★

```sql
videos v
  JOIN channels c            ON c.id = v.channel_id
  LEFT JOIN LATERAL (최신 snapshot 1건) s ON true
  LEFT JOIN content_summaries cs ON cs.video_id = v.id
```

**LATERAL이 핵심**: 영상마다 `collected_at DESC LIMIT 1` — "가장 최근에 찍은 지표"만 뽑는다.
스냅샷이 수천 행이어도 영상당 1행으로 접힌다.

### 이름이 바뀌는 컬럼 (⚠️ 바꾸면 프론트가 깨진다)

| 원본 | 뷰에서 | 왜 |
|---|---|---|
| `v.id` | **`video_id`** | `c.id`·`s.id`와 충돌 방지 |
| `s.collected_at` | **`snapshot_at`** | "언제 찍은 값인지" 명시 |
| `c.avg_views` | **`channel_avg_views`** | 영상 지표와 구분 |
| `c.name` | `channel_name` | |
| `c.category` | `channel_category` | |

> 구체화(STEP 2)에서 **"뷰 컬럼명은 바꾸지 마"**를 못박는 이유가 이것이다.
> kkirikkiri가 `id`·`collected_at`으로 만들어버리면 전 화면이 조용히 빈다.

**LEFT JOIN인 이유**: 방금 수집한 영상은 스냅샷·요약이 아직 없다. INNER JOIN이면 신규 영상이
화면에서 통째로 사라진다. → **`views`·`keywords`는 NULL일 수 있다** (프론트가 방어해야 함).

## 3. 지표 공식 — 실제 코드 그대로 ★

전부 **프론트 JS에서 계산**한다. DB에 저장하지 않는다.

```ts
const chAvgMultiplier = avgViews > 0 ? views / avgViews : 0;      // 채널 평균 대비 배율
const vsRatio         = subs > 0 ? (views / subs) * 100 : 0;      // 구독자 대비 %

// ★ Outlier Score = 배율×10 + V/S Ratio
const outlierScore    = avgViews > 0 ? (views / avgViews) * 10 + vsRatio : vsRatio;

const isOutlier       = chAvgMultiplier >= 1.5;   // Mega: >= 5.0
```

> ⚠️ **가장 흔한 오해**: `outlierScore = (views/avg) * 10`. **`+ vsRatio`가 빠졌다.**
> 초안 문서에 그렇게 적혀 있었고 kkirikkiri도 자주 이렇게 만든다 — 구체화·검증에서 반드시 확인.

**두 지표를 왜 더하나:**

| 지표 | 잡는 것 | 혼자 쓰면 |
|---|---|---|
| Ch.Avg 배율 | 그 채널 **평소 대비** 터짐 | 작은 채널이 조회수 1천으로도 10x |
| V/S Ratio | **구독자 밖으로** 퍼진 정도 | 큰 채널이 늘 유리 |

→ 더하면 **"평소보다 터졌고 + 구독자 밖으로도 나갔다"**가 위로 온다. `avg_views = 0`(신규 채널)이면
배율 항이 무의미하므로 `vsRatio`만으로 폴백한다.

### 파생 규칙

```ts
// Tier: keywords[0]로 그룹핑 → 그룹 내 outlierScore 내림차순 → "#1 of N"
// New This Week: published_at >= startOfWeek(월요일 시작)
```

### 백엔드 계산

```ts
// collect-data
uploadsPlaylistId = "UU" + platform_channel_id.slice(2);   // UCxxx → UUxxx
duration(sec)     = PT(H)(M)(S) 정규식 파싱;
format            = duration <= 60 ? "shorts" : "long";
avg_views         = 최근 30일 발행 영상들의 (각 영상 최신 스냅샷 views) 평균;  // Math.round

// extract-trends
growth_rate = prevCount === 0 ? 100 : ((cur - prev) / prev) * 100;   // 소수 2자리
video_count = 고유 video_id 수;
```

> `avg_views`는 **"최근 30일 발행 영상들의, 각각 최신 스냅샷 views"의 평균**이다.
> 스냅샷 전체 평균이 아니고, 전체 영상 평균도 아니다 — 3년 전 히트작이 기준선을 오염시키지 않게.

## 4. RLS — 개인용 MVP

| 테이블 | anon |
|---|---|
| channels · videos · video_snapshots · content_summaries · trend_topics · strategy_results | 읽기·쓰기 **공개** |
| `admin_settings` | **`admin_password`·`gemini_api_key` 제외** 읽기만. 쓰기 정책 없음 |

`admin_settings` 쓰기는 정책이 아예 없다 → anon 불가, **`manage-settings`(service_role) 경유만.**
비번 검증은 `verify_password` / `change_admin_password` RPC (SECURITY DEFINER, bcrypt).

> **검증**: 앱에서 Gemini 키 저장 후 anon으로 `select * from admin_settings` → `gemini_api_key`가
> **안 보여야** 정상. 보이면 RLS 정책이 안 걸린 것.

> ℹ️ 이 RLS는 **개인용 전제**다. 공개 URL이라도 남이 채널을 지울 수 있다는 뜻 —
> 다중 사용자로 갈 거면 `auth.uid()` 기반 정책으로 다시 짜야 한다. (`01-PRD.md` §6 로그인 = Out of Scope)

## 5. 의존 순서 (무엇을 먼저)

```
admin_settings + RPC      ← 키·비번의 기반
channels                  ← fetch-channel
  → videos → video_snapshots      ← collect-data
           → content_summaries    ← generate-tags (Gemini 키 필요)
  → video_details 뷰        ← 위가 다 있어야 의미가 있음
  → trend_topics            ← extract-trends (keywords 필요)
  → strategy_results        ← generate-strategy (전부 전제)
```

## 6. 자주 틀리는 곳 — 구체화 체크리스트

| # | 확인 | 틀리면 |
|:-:|---|---|
| 1 | `outlierScore`에 **`+ vsRatio`** | 정렬이 통째로 잘못됨 |
| 2 | 뷰 컬럼 **`video_id`·`snapshot_at`** | 전 화면 빈 값 |
| 3 | AI 태깅은 **`generate-tags`** (`summarize-videos` 아님) | 체인이 안 돎 (레거시 함수) |
| 4 | Trends는 **뷰에서 실시간 집계** (`trend_topics` 아님) | 화면 빔 |
| 5 | 채널당 **50개**(`max_videos_per_channel`) + 증분/초기 분기 | 데이터 부족 |
| 6 | 스냅샷은 **INSERT** (UPDATE 아님) | 시계열이 안 쌓임 |
| 7 | 지표는 **프론트 계산** (DB 컬럼 아님) | 스키마에 없는 컬럼 찾음 |

> `01-PRD.md` §8은 "채널당 10개"라고 제안한다 — 그건 **PRD 단계의 순진한 기본값**이고,
> 실제 동작 사양은 **50개**다. 구체화(STEP 2)가 바로 이런 차이를 메우는 단계다.
