# Clip 1 — 하네스 엔지니어링 이해하고 내 워크스페이스 설계하기 (이론·설계, ~26분, 실습 32)

> 페르소나 메모. 마케터는 "하네스"라는 말을 부담스러워하니 "이미 만들어왔다"로 바로 풀어준다. PO는 두 층과 4기준을 또렷하게 짚어주면 따라온다. 영업은 "내 업무를 묶는 환경에 사고를 막는 장치까지 있다"는 안심 포인트에 반응한다.

## 이 스킬을 쓰는 법 (먼저 읽기)

이 클립은 슬라이드 6장이 본체다. 강의 내용은 덱이 보여주고, 이 스킬은 두 가지만 돕는다. 하나는 클로드코드를 켜는 두 장면(B1과 B2)이고, 다른 하나는 마지막에 남기는 설계도 한 장이다.

그래서 이 문서를 한꺼번에 다 읽어주면 안 된다. 지금 진행 중인 슬라이드에 해당하는 부분만 안내하고, 뒤 내용은 그 차례가 오기 전에 미리 꺼내지 않는다. 특히 마지막 슬라이드의 "정답 없음" 이야기는 끝에 가서야 나와야 한다. 수강생이 채울 설계도도 답을 먼저 보여주지 말고 빈칸으로 안내한다.


## 클로드코드 화면은 두 번 (B1, B2)

이 클립에서 클로드코드 화면은 딱 두 번 나온다. 그때만 이 스킬이 출력을 만든다.

### B1 — S2 후반, "부품이 흩어져 있다"

학습 워크스페이스(fastcampus-cc)에 이미 들어 있는 하네스 부품을 실제로 보여주는 장면이다. 가짜 목록을 지어내지 말고 `ls`나 glob으로 진짜 파일을 띄운다.

입력 예시:
```
지금 이 워크스페이스에 있는 하네스 부품들을 보여줘. .claude/skills/ 안의 스킬들, CLAUDE.md, .claude/commands/, settings.json 훅을 한눈에 보이게 트리나 표로.
```

띄운 다음 멘트는 이렇게 받는다. "이게 다 하네스 부품인데, 지금은 따로따로 흩어져 있죠."

### B2 — S4 후반, 아키타입 고르기

워크스페이스 아키타입 5종을 표로 띄우고 "딱 하나만 고르세요"로 넘기는 장면이다.

입력 예시:
```
워크스페이스 아키타입 5종을 표로 정리해줘. 아키타입, 어떤 일에 쓰는지, 폴더 예시 한 줄씩.
```

나와야 할 표는 다음과 같다.

| 아키타입 | 어떤 일에 쓰나 | 폴더 예시 |
|---|---|---|
| 글쓰기 | 책이나 문서, 콘텐츠를 쓰는 일 | inbox / drafts / published |
| 리서치 | 조사하고 자료를 모아 보고서를 쓰는 일 | sources / notes / reports |
| 지식베이스 | 자료를 계속 모아 정리하고 찾아보는 일 | raw / wiki / index |
| 개발 | 앱이나 코드를 만드는 일 | src / docs / archive |
| 학습 | 공부하고 지식을 쌓는 일 | material / mywork / review |

여기서 고른 아키타입이 바로 다음 슬라이드(S5 설계도)의 2번 칸에 들어간다. 매주 보고서를 쓰는 사람은 글쓰기에 가깝고, 자료를 계속 모아 정리하는 사람은 지식베이스에 가깝다. 두 개를 고르면 둘 다 어중간해지니 하나만 고르게 한다.

## 슬라이드 진행 순서 (덱이 본체, 이건 위치 잡는 큐 시트)

| 슬라이드 | 시간 | 이 슬라이드에서 다루는 것 |
|---|---|---|
| S1 표지 | 1.5분 | 인사하고 오늘 흐름을 짧게 예고한다 |
| S2 정의 | 6분 | 하네스가 뭔지 정의하고, 왜 필요한지 여행 비유로 풀고, 검증과 갭 메우기까지 짚은 뒤 "이미 만들어왔다"로 닫는다. 끝에 B1을 띄운다. |
| S3 두 층 | 5.5분 | 이름이 왜 하네스인지 강아지로 설명하고, 지시 층과 강제 층으로 나눈 다음 수첩 요령을 덧붙인다 |
| S4 4기준 | 5분 | 설계 4기준을 짚고 어겨보기를 예고한다. 끝에 B2를 띄운다. |
| S5 설계도 | 3.5분 | 다섯 줄 양식을 보여주고 지금 채우게 한 뒤, 다음 클립의 빌더를 살짝 예고한다 |
| S6 마무리 | 5분 | "정답 없음"을 짚고 오늘 내용을 정리한 다음 설계도 작성을 독려한다 |

이 표는 지금 어디쯤인지 위치를 잡으려고 두는 것이다. 각 슬라이드의 실제 멘트는 덱 스피커 노트와 대본에 있으니, 진행 중인 칸만 참고하고 다음 칸을 미리 읽어주지 않는다.

## 오늘 결과물 — 설계도 다섯 줄

수강생이 직접 채운다. 답을 먼저 보여주지 말고 빈칸으로 안내한다.

```
1. 목적:
2. 아키타입:   (글쓰기 / 리서치 / 지식베이스 / 개발 / 학습 중 하나)
3. 폴더 초안:
4. 지시 층:    CLAUDE.md 핵심 규칙 + 배치할 스킬·커맨드
5. 강제 층:    막을 행동 한두 개
```

## 다섯 줄을 채울 때 보는 원리

이 원리들의 원전은 스킬에 내장된 `references/harness-standards/` 12편이다. 설계 단계에서는 아래 정도면 충분하고, 전체 기준은 clip-02에서 하네스 닥터를 만들 때 가져간다.

- **목적**: 워크스페이스 하나에는 에이전트 하나만 둔다. 목적이 섞이면 둘로 쪼갠다. 이 워크스페이스가 절대 안 하는 것까지 한 줄로 적으면 좋다.
- **아키타입**: 다섯 중 하나만 고른다. 두 개 고르면 둘 다 어중간해진다.
- **폴더 초안**: 번호를 붙여서 자료가 들어오는 곳과 결과물이 나가는 곳을 구분한다.
- **지시 층**: 매 세션 꼭 필요한 규칙만 CLAUDE.md에 짧게 적는다. 긴 절차는 스킬로 빼서 부를 때만 읽히게 한다.
- **강제 층**: 이 규칙을 어기면 사고인지 따져본다. 사고가 되는 규칙은 문서가 아니라 훅으로 막는다. 처음엔 한두 개만 건다. 뼈대는 처음부터 갖추되 안전장치는 필요한 만큼만 늘린다.

강의에서는 "두 층"으로 가르치지만, clip-02의 /audit은 이걸 일곱 칸으로 더 잘게 나눠서 점검한다. 지시 층은 CLAUDE.md, 스킬·커맨드, 서브에이전트 셋으로 나뉘고, 강제 층은 훅과 권한 한 칸이다. 거기에 목적과 폴더 같은 뼈대 두 칸, 검증과 변경 이력을 챙기는 운영 한 칸을 더해서 일곱이 된다. 설명할 때는 두 층으로 말하고, 점검할 때는 일곱 칸으로 본다.
