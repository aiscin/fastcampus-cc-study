# LLM Wiki 강의 — 개념 지도 + 설명 방향 (합본)

> 작성: 2026-06-03 · 용도: 최종 프로젝트 LLM Wiki 강의의 개념 선정 + 전달 방향 설정
> 입력: ① 6개 경쟁강의 커리큘럼 분석 / ② LLM Wiki 딥리서치(Karpathy 원문·실제 레포 1차 출처)
> 전제(우리 강의 DNA): 비개발자 대상 · 따라치기 완주형 · 난이도 점진 상승 · "클로드코드랑 대화하면 뭐든 된다"는 자신감 · IDE는 안티그래비티

---

## A. 한눈에 보는 개념 지도 (7개 층)

> 아래로 갈수록 LLM Wiki 고유·심화. 위 3개 층(L0~L2)은 "토대", 가운데(L3~L4)가 "핵심", 아래(L5~L6)가 "차별·확장".

```
L0 마인드셋 ─ 왜 LLM Wiki인가 (문제의식)
L1 토대 도구 ─ 마크다운 · Obsidian · Claude Code · Git
L2 지식 구조 ─ 폴더 분류 · 위키링크 · 메타데이터 · MOC (PKM 차용)
─────────────── 여기까지 "준비" ───────────────
L3 핵심 아키텍처 ─ 3계층(raw/wiki/schema) · 불변식 · 페이지 유형
L4 운영 루프 ─ ingest → compile → query → lint (4동사)
─────────────── 여기까지 "동작하는 위키" ───────────────
L5 효율 원리 ─ tiered read · Work Directory 분할 (왜 안 느려지나)
L6 확장 ─ 검색(qmd) · GraphRAG · 멀티에이전트/봇 · 자기진화
```

## B. 층별 개념 카드 (개념 / 왜 / 어떻게 설명 / 차용처)

### L0. 마인드셋 — 왜 LLM Wiki인가
| 개념 | 왜 필요 | 어떻게 설명(비유/멘트) | 차용 |
|---|---|---|---|
| **맥락 반복설명의 고통** | 학습 동기 부여(훅) | "AI한테 매번 내 상황 처음부터 설명하느라 지치죠?" | 강의①⑥ 훅 |
| **RAG는 축적이 안 된다** | LLM Wiki 차별점의 출발 | "검색은 매번 처음부터. 쌓이는 게 없어요(Karpathy: *Nothing is built up*)" | 딥리서치(Karpathy) |
| **bookkeeping을 LLM이** | 핵심 통찰 | "지식관리의 귀찮은 건 읽기가 아니라 *정리·연결 유지보수*. 그걸 LLM이 대신" | 딥리서치(Karpathy) |
| **Memex 계보(선택)** | 깊이·신뢰 | "1945년 Memex의 꿈을 LLM이 드디어 실현" — 깊이 원하면만 | 딥리서치 |

### L1. 토대 도구 — 무엇을 만지나
| 개념 | 왜 필요 | 어떻게 설명 | 차용 |
|---|---|---|---|
| **마크다운(.md)** | 위키의 물성 | "사람도 읽고 AI도 읽는 글자 파일. 블랙박스 벡터가 아니라 *내가 소유*" | 딥리서치 |
| **위키가 사는 집 = 우리 워크스페이스** | 위키의 위치 | ★"별도 앱 X. `fastcampus-cc` 폴더 안에 그냥 삶. 안티그래비티에서 바로" | 우리 결정 |
| **Claude Code = 파일 읽고 쓰는 엔진** | 위키를 유지하는 손 | "Claude Code가 내 폴더를 read/grep/write. *이게 엔진*. 소비자도 에이전트(사람 아님)" | 강의⑥ |
| **그래프뷰 = Claude Code가 렌더** | 사람 구경용(옵션) | ★"위키는 *Claude Code가 먹는* 거라 그래프 안 봐도 됨. 보고 싶으면 '그래프 그려줘' → Mermaid/HTML 생성 → 브라우저. *환경 이동 0*" | 딥리서치 |
| ~~Obsidian~~ = 각주 | 엔진 아님 | "이미 Obsidian 쓰면 같은 폴더 열면 됨(마이그레이션 0). 안 써도 그래프까지 다 됨" | Karpathy: 도구는 abstract |
| **Git = 자동 버전관리** | 안전망 | "AI가 잘못 고쳐도 되돌릴 수 있는 타임머신. 자동 커밋" | 딥리서치 |
| **터미널/CLI 최소개념** | 비개발자 진입장벽 | "명령어 한 줄 = AI한테 일 시키기. 무서운 거 아님" | 우리 강의 철학 |

### L2. 지식 구조 — 어떻게 정리하나 (PKM 차용)
| 개념 | 왜 필요 | 어떻게 설명 | 차용 |
|---|---|---|---|
| **폴더 분류 사고(PARA)** | raw/wiki 구분의 사전 감각 | ★"강의 내내 쓰던 `fastcampus-cc` 번호 폴더가 *이미 PARA*예요" 콜백 (이론 X) | 우리 워크스페이스 |
| **위키링크 `[[...]]`** | 연결 = 그래프의 엣지 | "단어를 대괄호로 감싸면 페이지끼리 연결. 위키백과처럼" | 강의⑥/딥리서치 |
| **메타데이터(frontmatter/tags)** | AI가 필터·미리보기 | "페이지 맨 위 요약·태그 → AI가 본문 안 열고도 훑음" | 강의①/딥리서치 |
| **MOC(index.md, 진입점)** | 항해의 입구 | "도서관 안내데스크. AI가 *제일 먼저* 보는 목차" | 딥리서치 |
| **그래프뷰** | 연결 시각화·고립 발견 | "내 지식이 어떻게 엮였는지 눈으로. 외톨이 노트 발견" | 강의① |

### L3. 핵심 아키텍처 — LLM Wiki의 정체성 ★
| 개념 | 왜 필요 | 어떻게 설명 | 차용 |
|---|---|---|---|
| **3계층: raw / wiki / schema** | LLM Wiki의 골격 | "원본(안 건드림) → 위키(AI가 씀) → 규칙(CLAUDE.md=AI 두뇌)" | 딥리서치(Karpathy·ekadetov) |
| **불변식: raw는 LLM이 안 만짐** | 신뢰의 기반 | "원본은 박물관 유리장 안. AI는 보고 베끼기만, 훼손 금지" | 딥리서치 |
| **CLAUDE.md = 스키마=두뇌** | AI 작업 일관성 | "AI한테 '이 위키는 이렇게 정리해'라고 적어둔 규칙서" | 딥리서치 |
| **페이지 유형(소스요약/개념/엔티티)** | 위키의 구성단위 | "원본 요약 / 개념 정리 / 인물·사물 페이지 3종" | 딥리서치 |
| **log.md(작업로그)** | 진화 타임라인 | "AI가 한 일을 날짜순 일기로. 무슨 일이 있었는지 추적" | 딥리서치 |

### L4. 운영 루프 — 4개의 동사 ★ (커리큘럼 뼈대 후보)
| 개념 | 왜 필요 | 어떻게 설명 | 차용 |
|---|---|---|---|
| **ingest(수집)** | 원본을 raw/에 넣기 | "자료를 위키 창고에 일단 넣기. 아직 정리 X" | 딥리서치 |
| **compile(정제)** | 원본→위키 페이지 | "AI가 읽고 요약·개념·연결을 *만들어내는* 단계" | 딥리서치(ekadetov) |
| **query(질문)** | 위키에 묻기 | "위키에 질문 → 인용 달린 답 → *그 답도 위키에 저장*(복리)" | 딥리서치 |
| **lint(점검)** | 건강검진 | "죽은 링크·고아 페이지·모순 찾아 고치기" | 딥리서치 |
| (대안) **4-stage** | 한 번에 하는 변형 | "Ingest→Pull(추출)→Merge(통합,중복금지)→Schema(구조정리)" | 딥리서치(Ar9av) |

### L5. 효율 원리 — 왜 커져도 안 느려지나
| 개념 | 왜 필요 | 어떻게 설명 | 차용 |
|---|---|---|---|
| **tiered read(계층적 읽기)** | 전체 로드 회피 | "index 먼저 → 제목·요약만 스캔 → 필요한 본문만 펼침" | 딥리서치 |
| **Work Directory 분할** | 컨텍스트 격리 | "에이전트마다 자기 책상(폴더)만 봄 → 1만 노드 중 수십 개만" | 강의⑥/딥리서치 |
| **토큰=비용 감각** | 실무 현실감 | "AI는 읽은 만큼 돈. 그래서 *덜 읽고 잘 찾기*가 핵심" | 강의①(토큰 최적화) |
| ⚠️ **정량 수치는 단정 금지** | 정직성 | "몇 토큰·recall 몇%는 출처 약함 → *직접 측정 실습*으로" | 딥리서치 caveat |

### L6. 확장 — 차별·심화 (선택)
| 개념 | 왜 필요 | 어떻게 설명 | 차용 |
|---|---|---|---|
| **검색(qmd, BM25+벡터)** | 큰 위키 대응 | "위키 커지면 하이브리드 검색 켜기. 작으면 목차 직접 읽기" | 딥리서치 |
| **GraphRAG 연동** | 위키링크의 자연 확장 | "위키링크가 이미 그래프 → Neo4j로 내보내면 GraphRAG" | 강의③⑥ |
| **멀티에이전트/봇** | 협업·서비스화 | "위키 기반으로 답하는 Discord/Slack 봇·팀" | 강의⑥② |
| **자기진화 루프** | 운영 자동화 | "메모리·로그·스킬로 봇이 스스로 나아짐" | 강의②(Hermes) |

## C. 설명 방향 — 딥리서치 + 강의분석을 어떻게 합쳐 가르칠까

### 원칙 1. "문제 → 도구 → 구조 → 동작 → 효율 → 확장" 한 방향 흐름
난이도가 점진 상승하도록 L0→L6 순서를 깨지 않는다. 비개발자가 **L4까지만 와도 "동작하는 위키"**를 완주하게 설계(L5·L6은 보너스/심화).

### 원칙 2. 개념마다 "비유 → 의미 → 직접 해보기" 3박자
- 추상 개념은 반드시 **현실 비유 먼저**(도서관 사서, 박물관 유리장, 타임머신 등).
- 그다음 의미, 그다음 따라치기. "이게 뭐냐면~" 설명 후 바로 손.

### 원칙 3. 멘트는 "~하려는데 어떻게 해?" 패턴
- ❌ "위키를 컴파일해줘" → ⭕ "이 자료들을 위키로 정리하고 싶은데 어떻게 하지?"
- 학습자가 *의도+방향을 AI와 상의*하는 장면으로. (도구 사용법이 아니라 대화 자신감 체득)

### 원칙 4. 경쟁 강의와의 차별 포지션을 흐름에 녹인다
- 강의①(PKM/비개발자)·⑥(멀티에이전트/Discord)·②(Hermes 자기진화)가 이미 점유.
- 우리 빈자리 = **"비개발자가 따라치며 3계층·4동사로 *끝까지 동작하는* 위키를 완주"**.
- → L0~L4를 두껍게, L6은 "맛보기"로. (남들은 L6에 무게, 우리는 L3·L4 완성도로 승부)

### 원칙 5. 정직성 = 신뢰 (딥리서치 caveat 반영)
- "토큰 N배 절감·recall 100%·벡터DB보다 우월" 같은 **수치 단정 금지**.
- 대신 정성적 가치(축적성·소유성·설명가능성)로 설득 + 효율은 "내 볼트로 측정해보자" 실습.

### 원칙 6. MUST vs OPTION 명확히
- **MUST(완주 필수)**: L0 문제의식 / L1 마크다운·Obsidian·Claude Code·Git / L2 위키링크·메타데이터·MOC / L3 3계층·CLAUDE.md / L4 4동사.
- **OPTION(심화/시간 여유)**: L2 PARA·그래프뷰 깊이 / L5 정량측정 / L6 검색·GraphRAG·봇·자기진화.

## D. 확정된 방향 (2026-06-03)

✅ **1. 끝점 = L4 "동작하는 위키"까지** (확정)
- raw/wiki/schema 3계층 + ingest→compile→query→lint 4동사로 *완전히 동작하는 위키*를 끝까지 완주.
- **L5·L6은 "맛보기"** (마지막 1~2클립 데모/예고 수준). 우리 차별 = 남들이 L6에 쏟을 때 **L3·L4 완성도·완주율**로 승부.

✅ **2. L2 PKM = "워크스페이스 콜백"으로 처리** (확정)
- PARA를 이론으로 가르치지 않음. 강의 내내 쓰던 `workspaces/fastcampus-cc`가 **이미 PARA 기반**임을 한 문단으로 짚는다.
- 매핑: **P**rojects=`50-my-work` / **A**reas=`00-system`,`10-curriculum` / **R**esources=`20-references`,`30-templates`,`40-mock-data` / **A**rchive=`90-archive`.
- 게다가 LLM Wiki 3계층의 씨앗이 워크스페이스에 이미 있음 → **콜백 한 방으로 3계층 도입**:
  - `CLAUDE.md`(이미 있음) = **schema 계층** / `40-mock-data`·참고자료 = **raw 감각** / `50-my-work`(AI가 쓰는 곳) = **wiki 감각**
- 멘트: *"기억나죠? 그 번호 폴더가 PARA예요. LLM Wiki는 그 '폴더마다 역할이 있다'를 지식 정리용으로 특화 → raw/wiki/schema."*

🟡 **3. 구현 기준 레포** (미정 — 다음 결정)
- ekadetov(ingest/compile, Claude Code 플러그인) / Ar9av(4-stage, 멀티에이전트) / 우리 자체 구조.
- 권장: **ekadetov 골격(ingest/compile/query/lint)**을 표준으로, 용어만 우리 식으로 재명명.

🟡 **4. GraphRAG** (미정 — L6 맛보기 후보)
- 끝점이 L4이므로 GraphRAG는 **L6 맛보기에서 "위키링크가 곧 그래프"라는 한 장면**으로만 예고.

## E. 확정 방향 기준 MUST 리스트 (최종 프로젝트 완주 = 이것들)

| 층 | 반드시 가르칠 개념 | 클립 묶음 후보 |
|---|---|---|
| L0 | 맥락 반복의 고통 · RAG는 안 쌓인다 · bookkeeping을 LLM이 | 도입 1클립 |
| L1 | 마크다운 · Obsidian 볼트 · Claude Code · Git 안전망 | 토대 2~3클립 |
| L2 | 위키링크 `[[]]` · 메타데이터 · MOC(index) (PKM은 1~2분) | 구조 1~2클립 |
| L3 | **3계층(raw/wiki/schema) · 불변식 · CLAUDE.md · 페이지 유형 · log** | 핵심 2~3클립 |
| L4 | **ingest → compile → query → lint 루프 (각 1실습)** | 운영 3~4클립 |
| L5 | tiered read·Work Directory (원리만, 정량은 측정 실습) | 맛보기 1클립 |
| L6 | 검색/GraphRAG/봇 = "이렇게 확장됩니다" 예고 | 맛보기 1클립 |

→ 다음 단계: 위 MUST를 **클립 단위 커리큘럼 초안**으로 펼치기.
