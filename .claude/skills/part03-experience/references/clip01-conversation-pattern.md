## ⚠️ 퀴즈 옵션 셔플 (전체 CHECK 공통)

이 파일의 모든 `AskUserQuestion` 호출 시, **`options` 배열을 매번 무작위 순서로 섞어서 제시**할 것. 아래에 적힌 순서는 작성 편의를 위한 것이고 정답이 항상 같은 위치에 오면 안 됨.

운영 규칙:
1. CHECK 블록의 `options` 배열을 그대로 복사하지 말고, **호출 직전에 셔플**한다 (Fisher-Yates 또는 단순 random shuffle).
2. 옵션의 `description` 필드도 그대로 따라가도록 객체 단위로 셔플한다.
3. 정답·피드백은 **옵션 라벨 텍스트로 비교**한다 (인덱스 ②/③ 같은 위치 표현 금지). 정답 표기는 항상 `정답 라벨: "..."` 형식.
4. "강사와 동일", "자유 입력" 같은 **메타 옵션은 항상 마지막**에 고정 (셔플 대상 제외).

## ⚠️ EXPLAIN → CHECK 게이트 (전체 단계 공통)

설명과 퀴즈가 한 화면에 같이 떠서 학생이 산만해지는 걸 막기 위해, 모든 단계에서 EXPLAIN과 CHECK 사이에 사용자 입력 게이트를 둔다.

**진행 패턴**:

```
[단계 N]
   1. EXPLAIN 블록 출력
   2. 끝에 한 줄 안내:
      "충분히 읽으셨으면 `다음` 입력 → 퀴즈로 갑니다."
   3. 사용자가 `다음` 입력 (또는 `진행`, `OK`, `continue`)
   4. AskUserQuestion 호출 → CHECK 표시
   5. 사용자가 옵션 선택 → 정답·피드백 출력
   6. (게이트 없이) 자동으로 다음 단계 EXPLAIN 시작
```

**게이트는 EXPLAIN→CHECK 사이만**. CHECK 답변 → 정답·피드백 → 다음 단계 EXPLAIN은 자동으로 이어진다 (사용자 부담 최소화).

**트리거 단어** (게이트 통과):
- `다음`, `진행`, `OK`, `네`, `continue`, `go`

**OPENING / 시나리오 박스 / CLOSING**도 동일:
- OPENING 출력 → "`다음` 입력하면 1단계 시작" → 게이트 → 1단계 EXPLAIN
- 시나리오 박스(텍스트만, 퀴즈 없음) 출력 → "`다음` 입력하면 마무리로" → 게이트 → CLOSING

**EDGE CASES**:

| 사용자 입력 | 대응 |
|---|---|
| "퀴즈 빼고 설명만" | CHECK 호출 생략, 정답·피드백만 한 줄로 표시 후 다음 단계 EXPLAIN |
| "빨리 가자" / "skip" | 게이트 1회 압축 — 즉시 다음 EXPLAIN으로 |
| 게이트 안 가고 무관한 질문 | 답변 후 "준비되면 `다음` 입력" 재안내 |
| EXPLAIN 다시 보고 싶다 | 같은 EXPLAIN 재출력 |

## ⚠️ AskUserQuestion 한글 안전성 (전체 CHECK 공통)

한글이 깨져 보이거나 잘리는 사고를 막기 위해 호출 직전에 다음을 강제한다.

운영 규칙:
1. **NFC 정규화**: 모든 `question` / `header` / `label` / `description` 문자열을 호출 직전 Unicode NFC로 정규화 (한글 자모 분해형이 섞이면 깨짐).
2. **금지 문자 제거**: 다음을 모두 제거하거나 ASCII 대체:
   - 제로폭 문자 (U+200B, U+200C, U+200D, U+FEFF)
   - smart quotes (` " ' ' `) → ASCII `"` `'` 로 대체
   - 비표준 공백 (U+00A0, U+2009 등) → 일반 공백
   - 이모지 (옵션 라벨/description에서 절대 사용 금지 — 깨짐 사고 잦음)
3. **길이 제한**:
   - `header`: 한글 12자 이내
   - `label`: 한글 40자 이내 (잘림 방지)
   - `description`: 한글 60자 이내, 한 줄로
4. **빈 문자열 금지**: `description` 빈 값이면 키 자체 생략. `""` 보내면 일부 환경에서 깨짐.
5. **줄바꿈·탭 금지**: `\n` `\t` 절대 포함 금지. 모두 한 줄 텍스트.
6. **응답 매칭**: AskUserQuestion 응답으로 받은 라벨은 NFC 정규화 + 양쪽 공백 트림 후 정답 라벨과 비교 (단순 `==` 금지).
7. **사전 점검 함수**: 모든 호출 전에 위 6개 규칙을 통과시키는 sanitize 함수를 거친다. 한 항목이라도 위반하면 호출 중단하고 자동 수정 후 재시도.
8. **fallback**: 그래도 한글 깨짐 발생 시 — 라벨을 영문 약어 + 한글 한 단어로 단축 (예: `"A. 옵션"` `"B. 옵션"`).

## 1단계 — 질문하기

### EXPLAIN

```
'질문하기'는 내 머릿속 바람을 혼잣말로 정리하는 단계가 아니라,
클로드코드에게 "어떻게 접근하면 좋지?"를 실제 질문으로 던지는 단계예요.

흔한 실수는 시작부터 "분석해줘", "앱 만들어줘"라고 명령하는 것.
그런데 초반에는 우리가 아직 모르는 게 더 많습니다.

  • 뭘 어떻게 분석할 수 있는지
  • 어떤 기능 구성이 필요한지
  • 어떤 결과물 형태가 적합한지
  • 먼저 결정해야 할 항목이 뭔지

그래서 1단계는 '정답 지시'가 아니라 '옵션 탐색 질문'입니다.
클로드코드를 제작자보다 먼저, 기획 파트너처럼 쓰는 거예요.

자주 쓰는 질문 형태는 4가지입니다.

  • 가능성 묻기:
    "이걸로 무엇을 할 수 있어?"

  • 방법 묻기:
    "이걸 하려면 어떤 순서로 가야 해?"

  • 표현 방식 묻기:
    "결과를 어떤 형태로 보여주는 게 좋아?"

  • 피저빌리티 묻기 (Claude Code가 진짜 할 수 있나):
    "이거 지금 환경에서 진짜 구현 가능해? 막히는 부분 있어?
     일단 핵심 한 조각만 작은 시제품으로 만들어서 동작하는지 보자."

마지막 4번째가 빠지면 — 멋진 기획 다 짜놓고 막상 만들 때 'Claude Code로
이건 안 됩니다' 통보 받게 돼요. 그래서 1단계 끝부분에는 항상 '단계별
시도(spike)'를 한 번 끼워 넣습니다. 작은 조각으로 진짜 동작하는지 확인.

클로드코드가 선택지·시제품을 펼쳐주면, 그다음부터 비로소 본인이 고르고 좁혀
갑니다. 그 재료가 2단계 '기획하기'로 넘어갑니다.

예 — 경쟁사 분석 임원 보고:
   "경쟁사 분석 임원 보고를 만들 거야. 정해야 할 항목 체크리스트와
    슬라이드 구성 옵션 2-3개 알려줘."
   → 클로드코드가 청중·목적·구성 옵션을 펼침 → 본인이 'C 의사결정용 8장' 선택
```

> 5단계 풀이는 아래 "한 시나리오로 5단계 풀어보기" 섹션에서 표로 한 번에 확인.

## 2단계 — 기획하기

### EXPLAIN

```
'기획하기'는 두 가지 행위가 한 단계 안에 같이 일어나는 단계예요.
BUILD에서 U(Understand)에 해당합니다.

  ① 정리: 1단계에서 받은 옵션 중 고른 방향을 본인이 한 문단으로 묶음.
     - 무엇을 (대상·범위)
     - 어떤 근거로 (추측 금지·숫자 인용·출처 등 검증 형식)
     - 어떤 형식으로 (3섹션 리포트 / HTML / 30분 영상 등)

  ② 의도 정렬: 정리한 걸 클로드코드에게 보낸 뒤, "방금 내가 부탁한 거 다시
     설명해봐" 시켜서 클로드코드가 받아들인 내용을 본인이 다시 들음.
     - 본인 머릿속과 어긋난 부분이 있으면 그 자리에서 수정
     - "응 그 말이 맞아" 가 나올 때까지 한두 번 핑퐁

①만 하고 ②를 건너뛰면 3단계 만들기에서 엉뚱한 결과가 나옵니다. AI가
받아들인 그림과 본인 의도가 다르다는 걸 결과물 보고서야 알게 되거든요.
그래서 기획하기 = 정리 + 의도 정렬 두 동작 모두입니다.

예 — 경쟁사 분석 임원 보고:
   "8장: ①표지 ②포지셔닝맵 ③④⑤A·B·C 비교 ⑥위협기회 ⑦Q3권고 ⑧Q&A.
    IR PDF 사용. 톤은 데이터 기반 임원용 격식.
    너가 받은 거 다시 설명해봐. 어긋난 거 짚어줘."
   → AI: "⑦은 단순 권고보다 매트릭스가 임원에게 통할 거예요"
   → 본인: "맞아, ⑦은 매트릭스로. 진행."
```

### CHECK

{
  "questions": [{
    "question": "경쟁사 분석 슬라이드 — 기획하기가 잘 된 프롬프트는?",

    "options": [
      {"label": "임원이 좋아할 슬라이드로", "description": "추상"},

## 3단계 — 만들기

### EXPLAIN

```
'만들기'는 기획 기반으로 클로드코드가 실제 결과물을 생성하는 단계.
여기서 우리가 할 일은 의외로 적어요 — 잘 기획해서 보내고, 진행 상황을
지켜보는 것.

단, 한 번에 모든 걸 만들라고 시키면 결과가 어긋나요. 그래서 핵심 원칙:

  • 뼈대 먼저: 목차·구조·최소 동작본
  • 살은 나중에: 본문 채우기·디자인·확장 기능

뼈대 단계에서 '아 이거 빠졌네' 발견하기가 훨씬 쉽습니다.

예 — 경쟁사 분석 임원 보고:
   "먼저 ①~⑧ 목차와 각 장 핵심 메시지 1줄만 제안해줘"
   → 클로드코드가 8장 목차 + 메시지 전달 → 본인 검토 → "OK 본문 작성해줘"
   → 클로드코드가 IR 자료·시장 조사 PDF 읽고 8장 자동 생성 (차트·매트릭스 포함)
```
